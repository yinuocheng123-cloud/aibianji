# 版本记录：v1.8-login-page-cache-refresh

## 1. 本轮目标

本轮目标是修复“点击登录进入中台，进不去”的问题，并把当前版本恢复到用户可以直接测试的状态。

为什么优先处理：

- 这个问题直接阻塞了整套中台的演示与测试。
- 用户反馈发生在登录入口，属于当前版本最靠前、最影响体验的问题。

本轮属于：

- Bug 修复
- 可测试性恢复

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.7-retention-reason-and-smoke-cleanup.md`
- `custom/notes/v1.6-manual-retain-and-task-restore.md`

与本轮最相关的历史结论：

- `v1.6` 与 `v1.7` 主要在收口工作流和回归脚本，没有专门处理浏览器端缓存问题。
- 后端登录接口在最近几轮一直可用，因此“点登录进不去”更值得先排查前端静态资源、浏览器缓存和页面加载状态。

本轮开始前执行了：

- `git pull --ff-only`

结果：

- `Already up to date.`

## 3. 问题分析

### 3.1 表面现象

用户反馈：

- 点击“登录进入中台”，无法进入中台主界面。

### 3.2 排查结果

本轮先验证了三件事：

1. 后端登录接口
- `POST /api/auth/login`
- 使用 `admin / admin123`
- 返回 `200`

2. 本地服务状态
- `http://127.0.0.1:3210`
- 返回 `200`

3. 登录页首屏渲染
- 用 Edge headless 截图
- 登录弹层、输入框、按钮均正常显示

### 3.3 核心矛盾

本轮没有复现出“后端拒绝登录”或“页面模板损坏到无法渲染”的问题。

因此更可信的判断是：

- 用户浏览器端当时命中了旧静态资源，导致当前页面与最新 `app.js` / `styles.css` 没有对齐。

换句话说，这次更像是静态资源刷新问题，而不是接口逻辑问题。

## 4. 候选方案比较

### 方案 A：直接重改登录逻辑或后端鉴权

优点：

- 如果真是后端问题，可以一次修正。

缺点：

- 当前证据不支持这个方向。
- 容易在没有根因证据的情况下误改业务逻辑。

放弃原因：

- 登录接口已经明确返回 `200`，没有理由先动后端。

### 方案 B：重新整理前端静态资源版本并重启服务

核心思路：

- 不动业务登录逻辑。
- 更新 `index.html` 中 `styles.css` / `app.js` 的版本参数。
- 重启 `npm run dev`，确保浏览器拿到最新资源。

优点：

- 改动最小。
- 与当前现象最匹配。
- 风险最低。

缺点：

- 不能直接证明用户之前一定是缓存问题，但能最大概率修正这种问题。

最终采用原因：

- 当前最符合证据，也最稳。

## 5. 最终决策

本轮最终采用：

- 仅调整前端静态资源版本号
- 重启本地 `npm run dev`
- 用接口和无头浏览器做最小验证

没有做的事：

- 没有修改后端登录逻辑
- 没有修改会话逻辑
- 没有扩展新功能

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/index.html`

只做了两项有效改动：

- 将样式版本号从 `v11-mvp-testable` 更新为 `v12-login-fix`
- 将脚本版本号从 `v11-mvp-testable` 更新为 `v12-login-fix`

这样做的目的，是强制浏览器重新拉取最新静态资源，避免继续命中旧缓存。

### 6.2 运行态处理

- 停掉旧的 `3210` 端口进程
- 重新启动 `npm run dev`

## 7. 本轮优点

当前方案的优点：

1. 改动极小，不会误伤现有业务逻辑。
2. 直接针对最可能的缓存问题处理。
3. 通过服务重启和资源版本提升，能让用户最快拿到最新页面。

## 8. 本轮缺点与代价

本轮也有边界：

1. 这次修复更偏“静态资源刷新收口”，不是登录逻辑层面的功能修复。
2. 目前还不能 100% 证明用户之前一定是缓存导致，只能说这是当前证据下最合理、最小风险的处理。
3. `app.js` 与其他前端文件里仍有历史乱码文案残留，本轮没有顺手全清。

### 9.5 回归脚本

- 执行：`npm run smoke:e2e`

失败原因：

- 当前本地 SQLite 中已有历史测试数据，导致这次随机抓取命中旧重复文章，被直接判重。

这条失败与本轮登录缓存修复无直接关系，但说明 `smoke:e2e` 仍存在“受历史测试数据影响”的已知残留。

本轮最重要的结论：

- 当前登录接口和页面首屏都是正常的。
- 这次“登录进不去”更像是浏览器命中了旧静态资源，而不是代码里的登录逻辑失效。
- 通过更新资源版本号和重启服务，当前版本已经重新回到可测试状态。
