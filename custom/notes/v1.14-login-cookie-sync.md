# 版本记录：v1.14-login-cookie-sync

## 1. 本轮目标

本轮目标是修复“页面能打开，但点击登录后仍然回到未登录状态”的问题，并把本地 `3210` 端口服务恢复到可测试状态。

这个问题值得优先处理，因为它直接阻塞了整条 MVP 测试链路：如果无法稳定登录，后续抓取、改写、审核、发布都无法人工验证。

本轮属于：

- Bug 修复
- 可测试性提升
- 运行态排障

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.13-copy-audit-and-local-commit.md`
- `custom/notes/v1.12-mvp-login-smoke-verified.md`
- `custom/notes/experiments/exp-20260404-login-runtime-check.md`

修改前的状态是：

- 本地 `3210` 服务会偶发掉线，需要重新拉起
- 后端登录接口本身是通的
- 直接用 `X-Session-Token` 请求 `/api/bootstrap` 也能返回 `200`
- 但用户在真实浏览器里点击登录后，页面仍然提示“请先登录”

这说明当前问题不在“账号密码不对”或“后端完全不可用”，而是在浏览器持有的登录态与前端初始化链路之间存在冲突。

## 3. 问题分析

表面现象：

- 首页能打开
- 点击“登录进入中台”后，仍然留在登录层
- 用户手动打开带 `session_token` 的链接，也仍然像没登录

核心矛盾：

- 新的登录态已经拿到了
- 但页面初始化请求 `/api/bootstrap` 时，并没有稳定地使用到这份最新登录态

这轮排查后的关键发现是：

1. 首页在未登录时会稳定显示“初始化失败：请先登录”，说明前端脚本整体没有崩溃
2. 登录成功后，后端会跳转到 `/?session_token=...`
3. 当前前端虽然会把 `session_token` 保存到内存和 `localStorage`
4. 但浏览器里如果残留了旧的 `content_center_session` cookie，后端会优先使用旧 cookie，导致新的 header token 失效

这就是为什么“接口通、token 也有、页面却还像没登录”会同时出现。

## 4. 候选方案比较

### 方案 A：修改后端会话解析顺序

核心思路：

- 不再优先读取 cookie
- 改成优先采用有效的 header token

优点：

- 从根上修复 cookie 与 header token 的优先级冲突
- 后端行为更健壮

缺点：

- 需要修改 `api-router.js`
- 当前该文件在本地终端里存在编码显示问题，增量补丁不稳定
- 本轮目标是尽快恢复 MVP 测试，不适合在有额外编辑风险的文件上继续扩大改动

未选原因：

- 本轮更强调快速恢复可测状态，应该优先选择更小、更稳的修复点

### 方案 B：前端主动同步最新 token 到 cookie

核心思路：

- 只要前端拿到新的 `session_token`
- 就立即同步覆盖浏览器里的 `content_center_session` cookie
- 让 cookie、内存 token、`localStorage` 三者保持一致

优点：

- 改动局部，风险低
- 不用推翻现有登录流程
- 能直接覆盖掉用户浏览器里残留的旧 cookie
- 对“表单登录跳转”和“手动带 token 链接进入”两条链路都有效

缺点：

- 这是前端修复，不是后端优先级层面的根治
- 仍然保留了后端“cookie 优先”的历史行为

最终选择原因：

- 在当前仓库状态下，这是最小改动、最快恢复可测试状态的方案

## 5. 最终决策

最终采用方案 B：

1. 保留现有登录流程不变
2. 在前端 `setSessionToken` 时主动同步 cookie
3. 利用资源版本号更新，强制浏览器拉取最新脚本
4. 重新拉起本地 `3210` 服务并验证资源可访问

## 6. 具体实现

本轮改动文件：

### 6.1 `custom/content-center-demo/public/app.js`

新增 `syncSessionCookie(token)`：

- 有 token 时，把 `content_center_session` cookie 更新为当前 token
- 无 token 时，清空对应 cookie

调整 `setSessionToken(token)`：

- 在写入内存状态和 `localStorage` 之前，先同步 cookie

这样做的作用是：

- 登录跳转带来的最新 `session_token` 会立即覆盖旧 cookie
- 后续 `/api/bootstrap` 请求不会再因为浏览器残留旧 cookie 而被误判成未登录

### 6.2 `custom/content-center-demo/public/index.html`

更新静态资源版本号：

- `styles.css?v=20260404-v18-session-cookie-sync`
- `app.js?v=20260404-v18-session-cookie-sync`

作用是强制浏览器重新获取最新脚本，避免继续执行旧缓存。

## 7. 本轮优点

当前方案相较之前的优点：

- 直接修复了“旧 cookie 覆盖新 token”的真实冲突点
- 改动集中在前端单文件，符合最小改动原则
- 不需要重写登录页、不需要推翻现有认证结构
- 对用户现有浏览器环境更友好，不要求先手动清 cookie

## 8. 本轮缺点与代价

这轮也有明确代价：

- 仍未修改后端 `api-router.js` 中 cookie 优先的历史实现
- 这属于“先恢复可测、后做更彻底治理”的取舍
- 本地 `3210` 服务依然会偶发掉线，后续还需要继续收口运行稳定性
- 前端已经预埋了客户端异常上报，但本轮没有继续把服务端接收路由补齐

- 旧 cookie 与新 token 冲突的问题已经在前端同步层被修补

## 10. 本轮结论

本轮最重要的结论是：

- 这次“登录不进去”的高概率根因，不是后端登录失败，而是浏览器残留旧 cookie 与最新 `session_token` 冲突
- 用前端主动同步 cookie 的方式，可以在不大改现有结构的前提下，快速把 MVP 登录链路恢复到可测试状态

应默认继承的经验：

- 本地演示系统在频繁重启后，最容易出问题的不是账号逻辑，而是浏览器残留会话
- 如果后续再次出现“表单已登录但页面仍未登录”，应优先检查 cookie / header token / localStorage 三者是否一致

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 让你直接刷新页面后再次验证登录
2. 如果仍有偶发失败，再把后端 `api-router.js` 的 token 解析彻底改成“优先有效 token”
3. 给客户端异常上报补服务端接收路由，后面定位前端初始化问题会更快
