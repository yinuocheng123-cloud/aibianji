# 版本记录：v1.19-simulated-portal-columns

## 1. 本轮目标

本轮目标是先把真实整木网接口之外的前置链路跑通，并把“转发到整木网”收口成可测试的模拟闭环。

这轮优先处理这个问题，是因为用户已经明确要求：

- 先把前面的抓取、内容池、AI 编辑和转发链路跑通
- 最后一步再接整木网真实接口
- 整木网资讯栏目当前先模拟为 3 个栏目：行业资讯、企业资讯、市场动态

本轮属于：

- 功能补充
- 工作流收口
- MVP 可测试性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.18-front-workflow-and-backstage-hide.md`
- `custom/notes/v1.17-login-bootstrap-sync.md`

修改前的系统状态是：

- 登录链路已经跑通
- 前台已经开始按“抓取 -> 内容池 -> AI 编辑 / 直接转发”的方向收口
- AI 配置和失败日志已经通过前端逻辑做了隐藏
- 但默认栏目数据仍是旧演示口径：整木快讯、整木智造、整木材料、品牌观察
- “转发到整木网”仍只是通用占位回写，没有体现目标栏目

这意味着当前前台虽然写着“转发到整木网对应栏目”，但底层仍是旧栏目名和泛化回写，不利于做下一步真实接口对接前的联调验证。

## 3. 问题分析

本轮的表面现象不是页面报错，而是产品闭环还不够实：

1. 前台工作流已经改成整木网口径，但栏目数据还是旧 demo 口径
2. 发布回写仍然是通用 `portal.example.com`，没有体现整木网资讯栏目
3. 当前 3210 服务在运行态上还保留着旧进程时，用户会看到“前台新、后台旧”的不一致

核心矛盾是：

- 前台文案已经表达成“整木网资讯栏目”
- 但状态层和发布层没有同步变成“整木网模拟栏目”

真正需要收口的点不是再堆更多页面，而是把“栏目定义、推荐映射、发布回写、运行态服务”统一到同一口径。

## 4. 候选方案比较

### 方案 A：只改前台文案，不动默认栏目和发布回写

核心思路：

- 继续保留现有类别与演示回写
- 仅把前台提示改成行业资讯、企业资讯、市场动态

优点：

- 改动最小

缺点：

- 前台与后台语义仍不一致
- 内容池“直接转发”仍然不知道自己转到哪个栏目
- 后续接真实接口前还要再拆一次

放弃原因：

- 这会继续制造“页面看起来对了，但实际数据还是旧的”问题

### 方案 B：同步收口默认栏目、转发回写和前台提示

核心思路：

- 把默认栏目统一成 3 个模拟整木网资讯栏目
- 把旧 demo 栏目在状态归并时迁移成新栏目
- 发布动作按所选栏目生成模拟主站 ID 和 URL
- 前台按钮和结果提示明确写出目标栏目

优点：

- 从抓取任务、关键词映射、内容池到转发结果，整条链路语义一致
- 不需要引入真实接口，也能先做完整联调
- 后续接真实整木网接口时，只需要替换发布适配层

缺点：

- 需要同时改状态层、路由层和前台提示
- 现有数据库中的旧栏目数据需要兼容迁移

最终选择原因：

- 这是当前最符合“先把前面跑通，最后一步再打通整木网”的方案

## 5. 最终决策

本轮最终采用方案 B，核心决策如下：

1. 默认整木网资讯栏目固定模拟为：
   - 行业资讯
   - 企业资讯
   - 市场动态
2. 若历史状态里仍是旧 demo 栏目，则在状态归并时自动替换成这 3 个模拟栏目
3. 关键词和文章中的栏目引用若指向无效栏目，则自动回退到默认栏目
4. 发布时不接真实整木网接口，而是按栏目生成模拟主站 ID 与 URL
5. 前台继续隐藏 AI 配置和失败日志，但把栏目和转发结果的提示改成整木网口径

## 6. 具体实现

### 6.1 `custom/content-center-demo/src/storage/state-store.js`

本文件是本轮最关键的状态层调整点。

具体改动：

- 新增 `SIMULATED_PORTAL_CATEGORIES`
  - 统一定义 3 个模拟整木网资讯栏目
- 新增 `LEGACY_PORTAL_CATEGORY_NAMES`
  - 用于识别旧 demo 栏目数据
- 新增 `shouldUseSimulatedPortalCategories(categories)`
  - 判断当前状态里的栏目是否仍属于旧 demo 口径
- 新增 `normalizePortalCategories(categories)`
  - 把旧栏目迁移为新模拟栏目，并补齐 `portalCategoryId`
- 把默认状态中的 `categories` 从旧 4 栏目改成 3 个整木网资讯栏目
- 在 `mergeState(rawState)` 中：
  - 统一归并栏目
  - 修正关键词的 `categoryId`
  - 修正文章的 `recommendedCategoryId`

作用：

- 无论是新初始化数据还是历史 SQLite 数据，只要还是旧 demo 栏目，都能自动切到新口径

### 6.2 `custom/content-center-demo/src/routes/api-router.js`

本文件负责把“转发到整木网”从泛化占位改成模拟栏目回写。

具体改动：

- 新增 `simulatedPortalRoutes`
  - 把栏目名映射到模拟 URL 路径
- 新增 `getPortalCategoryTarget(categoryId)`
  - 根据文章推荐栏目找到目标栏目名、栏目 ID 和路径段
- 把仪表盘里“已发布”的提示改为“已模拟回写整木网资讯栏目”
- 在 `/api/articles/:id/review` 的 `publish` 分支中：
  - 按栏目生成 `portalArticleId`
  - 按栏目生成 `portalUrl`
  - 发布日志明确写出转发到了哪个整木网资讯栏目

效果：

- 现在发布不再是泛化的 `portal.example.com`
- 而是会明确回写到模拟的整木网资讯栏目路径

### 6.3 `custom/content-center-demo/public/app.js`

本轮主要补了前台对栏目语义的表达。

具体改动：

- 新增 `getCategoryById()`、`getCategoryName()`
  - 统一前台取栏目名称逻辑
- `ensureDirectForwardButton()` 默认文案改为“直接转发到整木网资讯栏目”
- `configureFrontstageLayout()` 中：
  - 关键词说明改成“整木网资讯栏目推荐映射”
  - 内容池提示改成“当前模拟 3 个资讯栏目”
  - 编辑区栏目标签改成“整木网资讯栏目”
  - 默认按钮文案改成“审核后转发到整木网资讯栏目”
- `renderDetail()` 中：
  - 根据当前文章的目标栏目动态更新：
    - 直接转发按钮文案
    - 审核转发按钮文案
    - 转发结果文案
- `directForwardArticle()` 中：
  - 默认审核备注会明确写出转发到哪个模拟栏目

作用：

- 用户在内容池和工作台里能直接看到当前转发目标，而不是只看到抽象的“发布”

### 6.4 `custom/content-center-demo/public/index.html`

本文件做了前台静态兜底。

具体改动：

- 前台导航中移除 `AI 配置` 和 `失败日志`
- 对应卡片默认改成 `hidden`
- 栏目标签静态文案改成“整木网资讯栏目”
- 静态资源版本号升级到 `v22-simulated-portal-columns`

作用：

- 即使前端脚本未执行，后台模块也不会再在前台直接显示

## 7. 本轮优点

相比上一轮，这轮带来的提升是：

- 前台、默认数据和发布回写终于统一到了整木网资讯栏目口径
- 内容池“直接转发”不再是空心按钮，而是会明确落到模拟栏目
- 旧 demo 栏目数据被兼容迁移，不需要手工清库
- 后续接真实整木网接口时，主要只需替换发布回写实现
- 运行态验证也形成了更明确的方法：既要看页面版本，也要看后端 bootstrap 数据

## 8. 本轮缺点与代价

这轮仍有明确代价和妥协：

- 目前只是模拟整木网资讯栏目，不是真实接口发布
- AI 配置和失败日志是“前台隐藏”，还不是独立后台页面
- 栏目迁移逻辑优先兼容旧 demo 数据，但如果未来出现新的自定义栏目，还需要再明确迁移策略
- 3210 服务在当前环境下仍需要手动确认是否重启到了最新代码

## 10. 本轮结论

本轮最重要的结论是：

- 在不接真实整木网接口的前提下，前置链路已经可以先按“整木网资讯栏目”口径完整跑通

被证实的判断：

- 只改前台文案不够，必须同步改默认栏目与发布回写
- 历史旧栏目数据可以在状态归并层平滑迁移
- 运行态验证必须同时关注“静态资源版本”和“后端 bootstrap 数据”

被修正的旧判断：

- 之前默认栏目还停留在旧 demo 口径，这已经不适合继续做前台流程确认
- 之前“发布回写主站信息”过于泛化，不利于下一步真实接口接入

本轮之后应默认继承的经验：

- 在真实接口未接前，所有前台转发流程都应以模拟整木网资讯栏目为准
- 若用户看到页面是新版本，但栏目仍是旧的，优先检查 3210 后端是否还是旧进程

## 11. 下一轮建议

下一轮最值得继续做的 3 件事：

1. 实测前台“直接转发到行业资讯 / 企业资讯 / 市场动态”三种路径是否都按预期回写
2. 把工作台剩余“发布”字样继续统一成“转发到整木网”
3. 在真实整木网接口准备好后，把当前模拟发布适配层替换成真实接口调用

当前收益最高的是第 1 件事，因为它能把这轮模拟栏目是否真正够用直接验证出来。
