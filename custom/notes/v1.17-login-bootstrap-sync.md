# 版本记录：v1.17-login-bootstrap-sync

## 1. 本轮目标

本轮目标是继续修复“登录请求已经成功，但页面仍停留在登录层”的问题。

这个问题值得优先处理，因为用户最新截图已经明确给出新的现象：`/api/auth/login` 返回 `200`，`/api/client-events` 也返回 `200`，说明账号校验和前端提交链路都已经通了，当前真正阻塞的是“登录成功后界面没有切换到中台工作台”。

本轮属于：

- Bug 修复
- 交互链路加固
- 本地 MVP 可测试性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.16-login-fetch-and-favicon.md`
- `custom/notes/v1.15-login-token-priority-and-client-events.md`
- `custom/notes/experiments/exp-20260404-login-runtime-check.md`

修改前，系统已经具备这些条件：

- 登录按钮走前端 `fetch('/api/auth/login')`
- 登录接口返回 `200`，并返回完整 bootstrap 风格数据
- `client-events` 不再 404
- 后端会优先选择有效 token，而不是被旧 cookie 抢走登录态

但用户最新截图表明：点击登录后网络里只有 `login 200` 和 `client-events 200`，没有看到后续 `bootstrap` 请求，页面视觉上仍停留在登录层。

## 3. 问题分析

表面现象是：

1. 登录请求成功
2. 页面依旧停留在登录卡片
3. 浏览器没有继续发起 `bootstrap` 请求

核心矛盾不是“登录有没有成功”，而是“登录成功后页面如何稳定切换到已登录视图”。

当前旧实现的问题点有两个：

1. 登录成功后直接 `hideLogin(); refreshData(result);`
   - 理论上可行
   - 但它完全依赖当前页面这一次渲染立即成功
   - 一旦浏览器在显示层、DOM 状态或缓存脚本上出现轻微异常，用户看到的仍可能是登录浮层

2. 登录浮层只依赖 `.hidden` 类名隐藏
   - 正常情况下足够
   - 但在真实浏览器里，如果某一轮样式状态没有按预期同步，视觉层仍可能停留

## 4. 候选方案比较

### 方案 A：继续沿用登录返回包直接渲染

核心思路：

- 保持 `login -> refreshData(result)` 不变
- 仅继续加日志

优点：

- 改动最小

缺点：

- 仍然把成功后的界面切换完全寄托在当前这一次前端渲染上
- 无法主动复用已经稳定存在的 `bootstrap` 初始化链路

未选原因：

- 用户已经连续多次遇到“登录成功但没进中台”，继续赌当前页渲染稳定性收益太低

### 方案 B：登录成功后强制再走一次 bootstrap，并把登录浮层做 DOM 级强隐藏

核心思路：

- 登录成功后先保存 `sessionToken`
- 再显式调用 `loadBootstrap()`
- 由现有 bootstrap 链路统一负责 `hideLogin()`、刷新页面状态和记录 `boot_success`
- 同时把登录浮层的显示/隐藏从“只改 class”升级为：
  - class
  - `hidden` 属性
  - 内联 `display`

优点：

- 登录后的页面进入路径和页面首次初始化路径统一，后续更容易验证
- 能直接复用 `/api/bootstrap` 这条已稳定验证过的链路
- 浮层隐藏更强，不再只依赖某一个 CSS 状态

缺点：

- 登录成功后会多一次 `bootstrap` 请求
- 前端代码路径比之前多了一层跳转

最终选择原因：

- 这是当前最贴近用户现象、同时最稳妥的修法

## 5. 最终决策

本轮最终采用方案 B：

1. 登录成功后，如果返回里有 `sessionToken`，则立即调用 `loadBootstrap()`
2. `loadBootstrap()` 成功后再统一完成中台视图切换
3. `showLogin()` / `hideLogin()` 同时控制 class、`hidden` 属性和内联 `display`
4. 升级前端资源版本，强制浏览器拿到新脚本

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

改动一：强化登录浮层显示/隐藏

- `showLogin()` 现在会：
  - 移除 `hidden` 类
  - 设置 `hidden = false`
  - 清理内联 `display`

- `hideLogin()` 现在会：
  - 添加 `hidden` 类
  - 设置 `hidden = true`
  - 直接写入 `display: none`

这样做的原因是避免“类名已经改了，但视觉层仍停留”的浏览器状态差异。

改动二：登录成功后改走 `loadBootstrap()`

- 旧逻辑：
  - `setSessionToken(result.sessionToken)`
  - `hideLogin()`
  - `refreshData(result)`

- 新逻辑：
  - `setSessionToken(result.sessionToken)`
  - `await loadBootstrap()`

保留兜底：

- 如果登录返回中没有 `sessionToken`，仍退回原来的 `hideLogin() + refreshData(result)` 逻辑

### 6.2 `custom/content-center-demo/public/index.html`

- 将 `styles.css` 与 `app.js` 的版本号升级到 `v20-login-bootstrap-sync`

作用：

- 强制浏览器拉取最新前端资源，避免继续执行旧版登录脚本

## 7. 本轮优点

相比上一轮，这一版的优点是：

- 登录后的进入路径更加统一
- 登录成功不再只依赖当前页一次性渲染
- 浮层隐藏更强，浏览器视觉残留概率更低
- 如果问题仍存在，后续排查可以直接围绕 `loadBootstrap()` 成败继续收口

## 8. 本轮缺点与代价

当前方案也有明确代价：

- 登录成功后会额外请求一次 `/api/bootstrap`
- 这仍然是本地 MVP 导向修补，不是正式生产级登录框架
- 仓库中历史乱码数据和旧测试数据依然存在，本轮没有同时清理

## 9. 验证与结果

本轮计划验证这些内容：

1. `git pull --ff-only`
2. `node --check public/app.js`
3. 首页返回最新资源版本
4. 本地 `3210` 服务重启后可访问
5. 登录接口成功后，前端会进入 `loadBootstrap()` 路径

本轮确认结论：

- 代码层已经完成登录后强制 bootstrap 的切换
- 浮层显示/隐藏逻辑已经做成更强的 DOM 级控制
- 新版本前端资源会通过版本号强制更新

## 10. 本轮结论

本轮最重要的结论是：

- 用户当前碰到的根因已经不再是“登录鉴权失败”
- 更可能是“登录成功后视图切换不稳定”
- 让登录成功后统一回到 `bootstrap` 主链路，是当前最稳的收口方式

后续默认应继承的经验：

- 本地 MVP 中，登录成功后的界面切换应尽量复用统一初始化链路
- 视觉浮层不要只靠 class 切换，必要时同步控制 DOM 属性和内联样式

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 重新启动 `3210` 服务并验证用户浏览器是否能真正进入中台
2. 如果仍异常，补充 `loadBootstrap()` 前后的客户端事件埋点
3. 等登录链路彻底稳定后，再继续清理前端残留乱码文案
