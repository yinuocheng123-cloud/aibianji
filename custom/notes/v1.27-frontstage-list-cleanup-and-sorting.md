# 版本记录：v1.27-frontstage-list-cleanup-and-sorting

## 1. 本轮目标

本轮要解决的问题，是在 `v1.26` 已经把“重点源 / 开放发现”放回内容池之后，继续把前台列表做得更清楚、更轻。

这一轮优先处理它的原因是：

- 当前前台结构已经基本正确，但列表表达仍然偏重
- 信息源和内容池还残留部分解释性文案与旧渲染逻辑
- 如果不先收口展示层，后面再接真实整木网接口时会把问题继续带下去

本轮属于：

- 前台结构收口
- 列表表达优化
- 可维护性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.26-content-pool-split-by-priority-and-open.md`
- `custom/notes/v1.25-source-split-priority-and-discovery.md`
- `custom/notes/v1.24-keyword-stats-and-toggle.md`

修改前的状态是：

- 内容池已经按“重点源 / 开放发现”分组
- 抓取源已经回归为抓取对象列表
- 关键词列表已经改成“抓取量 + 启用勾选”

但还有几个明显问题：

1. 内容池文章行的信息仍然偏多，像技术卡片，不像文章清单
2. 信息源顶部和表单区仍残留解释性提示，和“清清爽爽”的目标不一致
3. `app.js` 里还保留着旧的关键词渲染函数块，继续迭代时阅读成本偏高
4. 编审台和内容池列表默认顺序不够稳定，最近文章不一定排在前面

## 3. 问题分析

表面现象是“页面还有点乱”，但真正的核心矛盾有三层：

1. 结构正确，不代表信息密度正确  
   `v1.26` 解决了概念位置问题，但列表项本身仍然承载了过多辅助信息，用户第一眼还是不够轻松。

2. 清爽感主要取决于列表，而不是卡片数量  
   当前前台已经压到 5 个主块，不是模块太多，而是每个块里的列表还不够像“工作清单”。

3. 演化期遗留逻辑开始影响后续维护  
   比如 `renderKeywords` 的旧实现虽然不再生效，但继续留在文件里，会增加后续判断成本。

当前瓶颈不是功能不够，而是：

- 列表信息需要继续减负
- 旧实现需要开始做最小清理
- 默认排序需要更符合使用直觉

## 4. 候选方案比较

### 方案 A：只调 CSS，不动渲染逻辑

核心思路：

- 不改 `app.js`
- 只通过样式把列表看起来“压轻”

优点：

- 改动最小
- 风险最低

缺点：

- 列表里的信息量并没有真正减少
- 信息源顶部说明和内容池摘要仍然偏重
- 无法顺手清掉旧渲染逻辑

放弃原因：

- 这更像“化妆”，不是“收口”

### 方案 B：保持现有模块结构，精简列表渲染 + 补排序 + 清少量旧逻辑

核心思路：

- 不改前台五大分区
- 只调整信息源、内容池、编审台里最直接影响阅读的列表渲染
- 让最近文章优先显示
- 顺手把明显失效的旧函数块改名或隔离

优点：

- 改动边界仍然小
- 对用户感知最直接
- 能同时改善页面体验和后续维护性

缺点：

- `app.js` 仍然没有彻底拆分
- 一些更深层的历史函数没有在这轮全部清掉

最终选择原因：

- 这是当前“收益最高 / 风险最低”的收口方式

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 保持五大主块结构不变  
   不再重新调整首页骨架，避免反复改信息架构。

2. 把优化集中在最常用的三个列表  
   - 抓取源列表
   - 内容池文章列表
   - 编审台列表排序

3. 内容池继续保留“重点源 / 开放发现”双分组  
   但文章行只保留更关键的信息：
   - 标题链接
   - 摘要
   - 来源
   - 时间
   - 栏目

4. 对明显失效的旧实现做最小清理  
   把旧关键词渲染函数改成 `renderKeywordsLegacy`，避免后续继续误读。

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

本轮主要改了 4 类内容：

1. 新增文章新鲜度排序辅助
   - `getArticleFreshness`
   - 用于内容池和编审台按最近时间倒序展示

2. 通过后置覆盖方式，重写了前台关键渲染函数
   - `getFilteredArticles`
   - `renderSources`
   - `renderSelectionSummary`
   - `renderArticles`

   作用是：
   - 抓取源列表更像“对象清单”
   - 内容池更像“文章清单”
   - 选中状态摘要从长句改成了指标条

3. 收紧前台说明文案
   - 通过 `configureFrontstageLayout` 动态重写区块说明
   - 去掉关键词和抓取源表单区的解释性提示

4. 清理一处明显失效的旧逻辑
   - 原先重复的 `renderKeywords`
   - 将旧版本改名为 `renderKeywordsLegacy`

### 6.2 `custom/content-center-demo/public/styles.css`

补了前台列表收口所需的样式：

- `source-main`
- `source-url`
- `tag-row`
- `mini-badge`
- `summary-metrics`
- `summary-chip`
- `article-main`
- `article-summary`

作用是：

- 抓取源列表从“说明卡片”变成“对象条目”
- 内容池摘要区更轻
- 选中摘要更像操作状态条

### 6.3 `custom/content-center-demo/public/index.html`

本轮只做了一个必要调整：

- 把前端静态资源版本升到 `v30-frontstage-cleanup`

作用是：

- 强制浏览器加载这一轮新的 JS 和 CSS

## 7. 本轮优点

相比 `v1.26`，本轮带来的正向效果主要有：

- 内容池文章行更像真实文章列表，而不是技术信息卡
- 关键词和抓取源表单区更干净
- 抓取源列表更符合“网址 + 一句话简介”的使用心智
- 编审台和内容池默认更偏向“最近内容优先”
- 旧关键词渲染函数不再与当前实现同名，后续误判风险降低

## 8. 本轮缺点与代价

这轮也有明确代价：

1. 采用了后置覆盖方式重写若干渲染函数  
   这是为了尽量少碰前面已有实现，但也意味着文件里仍存在一部分旧逻辑。

2. 还没有彻底清理所有历史辅助函数  
   比如上一轮开放发现聚合用到的一些遗留函数，这轮没有继续深挖清掉。

3. 当前仍是单文件前端脚本  
   列表体验已经更清楚，但 `app.js` 的组织仍然偏大，后面继续迭代会越来越需要拆分。

## 10. 本轮结论

本轮最重要的结论是：

当前 demo 继续优化时，收益最高的方向不是再加功能，而是把前台列表再做轻一点、顺一点。

被证实的假设：

- 内容池真正需要的是“清晰的文章行”，不是更多说明
- 选中摘要改成指标条后，可读性明显更好
- 关键词和抓取源表单里的解释性提示可以继续减少

被修正的旧判断：

- 之前默认“结构已经对了，主要靠样式再压一压就行”，这次证明还需要直接动渲染逻辑，才能真正清爽

以后应默认继承的经验：

- 前台列表优先展示“可操作信息”，不要优先展示“解释信息”
- 内容池和编审台默认都应按最近时间排序

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 继续清理 `app.js` 中剩余的历史辅助逻辑  
   收益最高，能显著降低后续维护成本。

2. 把编审台右侧表单继续压轻一轮  
   风险低，且用户感知会比较直接。

3. 等前台结构稳定后，再接真实整木网发布接口  
   这样能避免接口适配过程中又回头改页面。
