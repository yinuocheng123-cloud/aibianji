本轮属于：

## 3. 问题分析
表面现象：

- `smoke:e2e` 有时会因为“抓取任务未生成可处理文章”失败
- 编审台右侧虽然已经比之前轻，但按钮、说明和结果提示仍有局部漂移

核心矛盾：

- 回归脚本依赖模拟抓取内容，而模拟内容跨轮结构过于接近，容易与旧 smoke 数据在正文相似度上撞线
- 前台 copy 已经开始集中，但还没有覆盖到次级按钮、错误提示和工作区内的说明层

主要原因：

1. 清理接口此前只能按单个 `matchText` 匹配，无法对 smoke 产生的多种固定前缀做一次兜底清理
2. 模拟抓取文章虽然标题和来源不同，但正文模板过于稳定，历史 smoke 文章积累后容易被判为重复
3. 编审台右侧的 AI 区和发布结果区缺少一层“轻提示”，因此视觉上仍显得块状偏重

## 4. 候选方案比较
### 方案 A：只改 smoke 脚本，在每轮前手工删除旧数据
核心思路：

- 不动服务端和模拟内容
- 仅在脚本里尽量清除当前轮次相关的旧数据

优点：

- 改动面小

缺点：

- 只能解决一部分历史污染
- 如果正文模板仍然高度相似，仍可能撞上未被清掉的旧文章

风险：

- 回归稳定性仍然依赖偶然性

### 方案 B：只改模拟文章正文，让每轮内容更唯一
核心思路：

- 不改清理接口
- 只增强模拟内容的唯一指纹

优点：

- 能直接降低误判概率

缺点：

- 旧 smoke 数据会继续长期积累
- 清理链路问题没有解决

风险：

- 后续回归库继续膨胀，日志和发布榜会越来越脏

### 方案 C：清理接口支持多匹配文本 + smoke 预清理 + 模拟内容唯一指纹
核心思路：

- 清理接口支持 `matchTexts`
- smoke 运行前先按固定前缀清掉旧 smoke 数据
- 同时增强模拟正文的唯一指纹，双保险降低重复误判

优点：

- 能同时解决“旧数据污染”和“模拟内容过像”两个问题
- 不需要触碰正式去重阈值
- 后续每轮验证都更稳定

缺点：

- 需要跨脚本、服务层和路由层同时改动

风险：

- 改动面比单点修补略大，但仍局限在测试与模拟链路

最终选择：

- 采用方案 C

原因：

- 这是当前收益最高、对主业务逻辑扰动最小的一种收口方式

## 5. 最终决策
最终采用三段式收口：

1. `/api/testing/cleanup` 支持 `matchTexts`
2. `smoke:e2e` 在每轮开始前先按 smoke 固定前缀做预清理
3. 模拟文章写入明确唯一指纹，进一步降低历史相似度误判

在前台层面，同时继续收两类问题：

- 把剩余次级按钮、状态和错误文案继续收回统一 `FRONTSTAGE_COPY`
- 让编审台 AI 区和发布结果区增加更轻的说明层，并压缩间距

## 7. 本轮优点
当前方案相比之前的好处：

- `smoke:e2e` 更不容易再被旧 smoke 数据干扰
- 不需要动正式去重阈值，风险更低
- 次级按钮和错误提示开始统一进入 `copy/helper`
- 编审台右侧更轻，信息层级更稳定
- 后续继续收 UI 时，有更清楚的 copy 收口基础

## 8. 本轮缺点与代价
已知代价和妥协：

- 这轮仍然基于模拟抓取文章，不是真实外部发现链路
- smoke 清理接口虽然更强了，但仍属于测试辅助能力，不是正式数据治理能力
- 前台 copy 体系依然没有覆盖所有动态文本，只是又推进了一层
- 编审台的 AI 区和发布结果区虽然更轻了，但右侧整体仍然是原生脚本拼装，不是组件级 UI

## 10. 本轮结论
本轮最重要的结论是：

- smoke 的历史数据干扰问题，不能只从脚本一侧修，而要同时解决“旧数据污染”和“模拟内容过像”两件事
- 前台 copy 的统一工作仍然值得继续做，而且越往次级按钮和提示文案收，页面越稳定
- 编审台右侧是否显得清爽，关键不是继续删字段，而是把说明层和结果层做轻

本轮继承了旧思路：

- 不扩新功能，继续收基础问题
- 保持前台清爽，不把后台配置重新暴露出来

本轮修正了旧判断：

- 不再认为 smoke 偶发失败只需要单侧脚本兜底
- 不再认为次级按钮和提示文案可以继续散写
