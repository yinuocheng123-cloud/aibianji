# 版本记录：v1.11-session-token-fallback

## 1. 本轮目标

本轮目标是继续修复“点击登录后仍然进不去”的问题，并在现有 cookie 登录链路之外，加一层更稳的会话兜底。

这个问题值得继续优先处理，因为：

- `v1.10` 已经修复了表单字段缺少 `name` 的问题，但用户实际反馈仍然是“进不去”。
- 这说明仅靠浏览器 cookie 持有登录态，在当前本地访问环境下仍然不够稳。

本轮属于：

- Bug 修复
- 登录链路加固
- 可测试性补强

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.10-login-form-field-names.md`
- `custom/notes/v1.9-login-form-fallback.md`

修改前的状态是：

- 登录表单已经可以原生 `POST`
- 用户名和密码字段已经具备 `name`
- 后端已支持表单登录与 `302` 回跳

但新的现象是：

- 用户仍反馈点击登录后无法进入中台

这让本轮判断从“表单字段问题”继续推进到“浏览器环境是否稳定保留 cookie / 登录态”。

## 3. 问题分析

表面现象：

- 登录表单结构已经修正，但用户仍无法进入系统。

本轮继续分析后，核心矛盾被重新定义为：

- 即使登录请求成功，浏览器环境也可能没有稳定沿用 cookie，从而导致后续 `/api/bootstrap` 仍返回未登录。

换句话说，之前被默认接受的旧前提是：

- “只要登录接口返回成功并写 cookie，后续页面就一定能拿到登录态。”

本轮修正后的新判断是：

- 当前本地预览环境下，登录态不能只押注 cookie，至少要有一条不依赖 cookie 的兜底通路。

## 4. 候选方案比较

### 方案 A：继续只排查 cookie

核心思路：

- 继续围绕 `Set-Cookie`、浏览器缓存和重定向行为做修补。

优点：

- 改动范围更小。

缺点：

- 仍然把登录成功后的可用性完全绑定在 cookie 上。
- 一旦当前访问环境对 cookie 支持不稳定，问题仍会反复出现。

放弃原因：

- 用户已经连续反馈“点了还是进不去”，继续只在 cookie 路线上打补丁，收益开始变低。

### 方案 B：增加 session token 兜底

核心思路：

- 登录成功后，除了写 cookie，还把 session token 临时回带到首页。
- 前端优先从 URL / 本地存储读取 token，并通过请求头发给后端。
- 后端鉴权同时接受 cookie 和 `X-Session-Token`。

优点：

- 即使 cookie 在当前环境下不稳定，前端仍能通过 header 携带登录态。
- 不推翻现有 session 结构，只是在现有方案上增加一条兜底通路。

缺点：

- 会话 token 在登录成功跳转阶段短暂出现在 URL 中。
- 前后端都要改，复杂度高于单纯修表单。

最终选择原因：

- 这是在不大改架构前提下，最直接提升“点登录后能不能真的进系统”稳定性的方案。

## 5. 最终决策

本轮最终采用方案 B：

1. 前端支持读取、保存和清理 session token
2. 所有 API 请求优先追加 `X-Session-Token`
3. 后端鉴权允许从 cookie 或 header 获取 token
4. 表单登录成功后，跳转到带 `session_token` 的首页地址

关键决策原因：

- 保留 cookie 方案，不推翻原有设计。
- 对本地预览环境增加更强兜底。

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

新增或调整：

- `appState.sessionToken`
- `readStoredSessionToken`
- `setSessionToken`
- `clearSessionToken`
- `getSessionToken`
- `request` 中自动追加 `X-Session-Token`
- `applyLoginQueryState` 同时处理 `session_token` 和 `login_error`
- `logout` 时清理本地 token
- `loadBootstrap` 时如果返回 token，则再次同步到前端

作用：

- 让前端在 cookie 不稳定时，仍有能力用 header 带登录态。

### 6.2 `custom/content-center-demo/src/routes/api-router.js`

新增或调整：

- `getCurrentUser` 优先从 cookie 或 `X-Session-Token` 读取 token
- `buildBootstrapPayload(user, sessionToken = "")`
- 表单登录成功时改为跳转到：
  - `/?session_token=...`
- JSON 登录成功时回包同时带上 `sessionToken`

作用：

- 让后端会话识别不再依赖单一路径。

### 6.3 `custom/content-center-demo/public/index.html`

调整：

- 静态资源版本号更新为 `v16-session-token-fallback`

作用：

- 强制浏览器重新请求最新前端资源。

## 7. 本轮优点

当前方案的主要优点：

- 登录态从“只靠 cookie”变成“cookie + header token”双通道。
- 即使本地预览环境对 cookie 不稳定，也保留进入系统的机会。
- 改动仍然围绕现有 session 结构展开，没有重做鉴权体系。

## 8. 本轮缺点与代价

本轮也有明确代价：

- 登录成功时 token 会短暂出现在 URL 中，这只适合本地 MVP 环境，不适合正式生产。
- 本轮仍未彻底清理前端残留乱码文案。
- 在本地服务重启验证过程中，发现后台启动方式不稳定，影响了完整回归验证。

- 服务重启后，后台常驻方式出现不稳定，导致登录重定向与 token 兜底链路的完整回归没有彻底收完。

## 10. 本轮结论

本轮最重要的结论是：

- 如果用户环境对本地 cookie 支持不稳定，仅修表单和重定向还不够，登录态本身也要有第二条兜底通路。

以后应默认继承的经验：

- 对本地 MVP 预览环境，关键登录动作最好准备一条不依赖单一浏览器行为的兜底路径。

## 11. 下一轮建议

下一轮最值得继续做的 3 件事：

1. 先把本地服务启动方式稳定下来。
2. 重新完整验证 `session_token -> X-Session-Token -> /api/bootstrap` 这条链路。
3. 登录收口后，再继续清理前端乱码文案。
