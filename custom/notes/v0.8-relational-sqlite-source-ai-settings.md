# 版本记录：v0.8-relational-sqlite-source-ai-settings

## 1. 本轮目标

本轮目标是承接 `v0.7-sqlite-real-adapters`，把上一轮提出但尚未真正落地的 3 件事补完整：

1. 把 SQLite 从“整包快照存储”继续拆到关系型表结构
2. 给抓取源补正文提取规则配置与失败日志
3. 给 AI 增加后台配置界面，不只依赖环境变量

这轮属于：

- 功能补充
- 可维护性提升
- Bug 修复

## 2. 背景与上下文

上一轮已经把本地持久化从 JSON 切到 SQLite，并接了真实抓取优先、DeepSeek 优先的适配路径，但仍有三个明显短板：

- SQLite 仍偏“状态快照型”，后续很难继续扩业务表
- 抓取源虽然能跑任务，但缺少正文提取规则和失败记录闭环
- AI 只能靠环境变量配置，管理员无法在后台直接调参数

本轮开始前按要求重新尝试了 `git pull --ff-only`，结果依然失败，原因是 `D:\ceshi\aibianji` 当前不是 Git 仓库。

本轮继续沿用 `v0.7` 的方向，不推翻“真实能力优先、失败时本地回退”的主思路，只把它补到更适合 MVP 测试和后续接手的程度。

## 3. 问题分析

这轮真正的问题不是“功能完全没有”，而是“功能已经有雏形，但还不够落地”。

主要矛盾有三层：

1. 数据结构问题  
   之前虽然已经引入 SQLite，但主数据仍然更像一份快照对象，难以按任务、文章、失败、AI 配置做细粒度追溯。

2. 抓取调试问题  
   站点提取失败时，页面上缺少明确的失败中心，管理员看不到到底是规则问题、关键词未命中，还是网页抓取本身失败。

3. AI 运维问题  
   仅靠环境变量配置模型参数，意味着测试人员无法在页面直接调整调用参数，也无法确认当前到底跑的是环境变量还是后台配置。

此外，这轮还有一个额外问题：`server.js` 在先前迭代中混入了大量乱码字符串，其中一部分把模板字符串语法也带坏了，导致文件虽然“看起来有代码”，但实际上已经不能正常通过 `node --check`。

## 4. 候选方案比较

### 方案 A：只补前端界面，不动后端结构

核心思路：
只在页面上增加 AI 配置卡片和失败日志区域，后端继续沿用旧状态结构。

优点：

- 改动快
- 页面上能先看见新增功能

缺点：

- 数据无法真正持久化
- 抓取失败与 AI 配置仍不可追溯
- 很快会再次返工

结论：

- 放弃。因为用户要求的是“补全并开始落代码”，不是只做假界面。

### 方案 B：重写服务端，彻底清理所有旧逻辑

核心思路：
把 `server.js` 整体推翻重写，连旧数据兼容一起重做。

优点：

- 长期看最干净
- 可顺手清理乱码和历史包袱

缺点：

- 改动面太大
- 风险高
- 不符合当前仓库强调的最小改动原则

结论：

- 放弃。本轮不做整体重写，只对关键路径定点修复和增量扩展。

### 方案 C：增量扩关系表 + 增量补 API + 顺手修复关键语法坏点

核心思路：

- 保留现有 demo 架构
- 只在现有 `server.js` / `index.html` / `app.js` / `README.md` 上做增量补充
- 把影响运行的乱码语法点修到可用
- 让新增功能真实落到后端和持久化层

优点：

- 改动面可控
- 符合当前工程形态
- 可以直接把 demo 提升到可测 MVP

缺点：

- 仍背着单文件服务端的历史包袱
- 文件里仍有部分旧乱码文案未完全清除

结论：

- 本轮采用方案 C。

## 5. 最终决策

本轮最终决策如下：

1. 在现有 SQLite 初始化逻辑中，继续以关系表作为主数据结构  
   `users / source_sites / categories / keywords / crawl_tasks / task_keywords / articles / article_keywords / article_tags / article_ai_history / logs / crawl_failures / ai_settings`

2. 保留旧 `app_state` 作为迁移兜底，不再作为主写入路径

3. 抓取源新增 `parseRule` 与 `excludeRule` 配置，抓取失败统一写入 `crawl_failures`

4. 后台新增 AI 配置界面，支持保存并回显有效配置，且保留环境变量回退能力

5. 不在这轮顺手重构大文件结构，只修复影响运行、验证和交接的关键问题

## 7. 本轮优点

本轮相比 `v0.7` 的提升主要有：

- SQLite 结构从“能存”进一步提升到“能查、能分层追溯”
- 抓取源终于具备了规则配置与失败记录这两个调试入口
- AI 配置不再只能改环境变量，管理员在页面就能直接调
- `server.js` 从“语法已损坏”恢复到可正常校验和启动
- README 和记录恢复成可交接状态，不再是乱码占主导

## 8. 本轮缺点与代价

这部分需要明确保留，不做粉饰：

- `server.js` 仍然是大单文件，后续继续堆功能会越来越重
- 文件内仍残留部分旧乱码文案，虽然不影响本轮核心功能，但后续最好继续清理
- 真正的正文抽取仍是轻量规则，不具备复杂站点自适应能力
- SQLite 本轮虽然已经关系化，但仍未拆出真正的模块边界和仓储层
- 主站发布仍是演示回写，不是正式发布链路

本轮最重要的结论有三个：

## 11. 下一轮建议

下一轮最值得继续推进的 3 件事：

1. 把 `server.js` 按“存储层 / 抓取层 / AI 层 / 路由层”做有限拆分  
   这是收益最高的一项，能明显降低后续继续堆功能时的风险。

2. 继续清理残留乱码文案并统一状态枚举  
   这是风险最低、但对可维护性帮助很大的收尾工作。

3. 给抓取源规则增加“最近一次提取预览”  
   这样管理员在正式执行任务前就能先验证规则是否命中正文，能显著降低调试成本。
