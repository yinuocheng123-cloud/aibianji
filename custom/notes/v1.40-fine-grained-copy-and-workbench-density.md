# 版本记录：v1.40-fine-grained-copy-and-workbench-density

## 1. 本轮目标
本轮继续收前台基础问题，不扩新功能，重点解决两类细碎但高频的问题：

1. 把前台剩余更细粒度的错误提示、空态说明和次级状态文本继续并入统一 `copy/helper`
2. 再压一轮编审台右侧结果区、AI 历史块和只读区的行内密度，让右侧工作区更像轻量编辑台

这一轮值得优先做，原因是：

- `v1.39` 已经把登录、初始化、交互异常的一批主路径文案收进了统一入口，继续往下收的边际成本已经很低
- `v1.38` 和 `v1.39` 连续两轮都证明，当前收益最高的工作已经不是补功能，而是继续清理表达层噪音
- 右侧工作区虽然已经可用，但只读块、AI 历史块和结果区仍然偏厚，离“正式编辑工作台”的感觉还差一层

本轮属于：

- 可维护性提升
- 前台体验收口
- 验证基线巩固

## 3. 问题分析
表面现象是“页面已经比较清爽了，但局部还不够稳”，核心矛盾主要有两个：

1. `copy/helper` 体系已经建立，如果还允许细粒度文本继续散落在渲染层和入口层，后续每一轮都会继续把字符串写回业务代码
2. 编审台右侧是否显得轻，关键不只是分区是否合理，还取决于只读块、历史块和结果块的行内密度是否继续压得住

主要原因：

- 前几轮优先收的是主按钮、主说明和主状态，剩下的“细碎文案”自然分散在多处 helper 和 render 分支里
- 右侧工作区在视觉上仍保留了较明显的块感：
  - `detail-meta` 间距偏松
  - `read-box` padding 和 line-height 偏大
  - `history-item` 与 `publish-result` 仍然有较明显的卡片层级

需要区分主次：

- 主因：剩余细粒度 copy 仍然有散写
- 次因：右侧工作区的密度还可以继续压轻
- 表象：页面“还能更清爽”，但真正要解决的是表达层一致性和密度层级

## 4. 候选方案比较
### 方案 A：只继续收 copy，不动右侧密度
核心思路：

- 继续把剩余次级文案并入统一 `copy/helper`
- 暂时不再碰编审台右侧样式

优点：

- 改动最小
- 维护性继续提升

缺点：

- 用户侧感知有限
- 右侧工作区仍会保留偏厚的视觉感受

放弃原因：

- 当前阶段用户侧最容易感知的问题，已经是“页面还可以再轻一点”

### 方案 B：只继续压右侧样式，不继续收 copy
核心思路：

- 只做样式层再压轻
- 暂时保留剩余散写细粒度文案

优点：

- 页面感知变化明显

缺点：

- copy 入口继续分散
- 后续再收文案时还要继续回到多文件找字符串

放弃原因：

- 会让维护成本继续滞留在渲染层

### 方案 C：继续收细粒度 copy，同时再压一轮右侧密度
核心思路：

- 把剩余细粒度状态文本、空态说明、fallback 模板继续并入统一 `copy/helper`
- 同步对右侧只读块、AI 历史块和发布结果区再做一轮低风险密度收口

优点：

- 可维护性和用户感知同时提升
- 与 `v1.38`、`v1.39` 的迭代方向完全一致
- 不影响业务主链路

缺点：

- 需要同时改 `app-state / app-utils / app-render / app.js / styles / index`

最终选择：

- 采用方案 C

## 5. 最终决策
本轮最终采用以下策略：

1. 继续扩展 `FRONTSTAGE_COPY`
2. 在 `app-utils.js` 中补更细的 helper：
   - 预览正文 fallback 组装
   - 关键词抓取量文本
   - 来源备注模板
3. 在 `app-render.js` 中继续删除散写文本：
   - 原始响应摘要前缀
   - 关键词统计标签
   - 筛选默认项
   - AI 历史无输出说明
   - 任务完成态判断
4. 在 `app-actions.js` 和 `app.js` 中补齐默认值和异常 fallback 的统一来源
5. 在 `styles.css` 中继续压轻编审台右侧只读块、历史块和结果块
6. 在 `index.html` 中同步升级前端资源版本，确保浏览器拿到新资源

## 10. 本轮结论
本轮最重要的结论是：

- 前台 copy 要真正收稳，就必须继续覆盖到 fallback、默认值、筛选默认项和历史块提示这种“看起来很小”的文本
- 编审台右侧是否像正式工作台，关键仍然是密度，而不是继续删字段
- 在 smoke 基线已经稳定的前提下，继续做前台表达层收口，仍然是当前收益最高、风险最低的方向

本轮继承了哪些旧思路：

- 延续 `v1.39` 的方向，继续把入口层和次级动态文案收回统一 `copy/helper`
- 延续 `v1.38` 的方向，继续压内容呈现层的密度和层级

本轮修正了哪些旧判断：

- 不再把“主按钮和主说明已统一”视为 copy 收口已基本完成
- 不再把“右侧已经压过一轮”视为编审台视觉密度已足够

形成了哪些新结论：

- 关键词统计、筛选默认项、来源备注、预览摘要前缀这类小文本，也应该视为 copy 体系的一部分
- 右侧工作区真正显得轻，要同时压 `detail-meta / read-box / history-item / publish-result`

## 11. 下一轮建议
