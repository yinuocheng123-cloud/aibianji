# 版本记录：v1.1-add-dev-script-and-run

## 1. 本轮目标

本轮目标是让 `npm run dev` 在当前项目中真正可用，并把本地服务按 `dev` 方式跑起来。

为什么这个问题值得优先处理：

- 用户直接要求执行 `npm run dev`
- 当前项目只有 `start` 脚本，没有 `dev`，命令本身无法成立

本轮属于：

- 功能补充
- 可维护性提升

## 2. 背景与上下文

本轮承接 [v1.0-local-git-repo-bootstrap.md](D:\ceshi\aibianji\custom\notes\v1.0-local-git-repo-bootstrap.md)。

`v1.0` 已经把根目录初始化成了本地 Git 仓库，但这轮开始时出现了新的 Git 状态：

- `git pull --ff-only` 不再报“不是 Git 仓库”
- 现在改成报 `dubious ownership`

这说明仓库基线已经建立，但当前 Windows 账户与 `.git` 所属用户不一致，后续如果要继续执行 Git 拉取，需要先处理 `safe.directory`。

## 3. 问题分析

这轮的表面问题是 `npm run dev` 不能执行，但核心原因其实很简单：

1. [package.json](D:\ceshi\aibianji\custom\content-center-demo\package.json) 只有 `start`
2. 项目本身是一个 Node HTTP 服务，不依赖额外 dev server
3. 所以缺的是脚本入口，不是业务实现

换句话说，本轮不需要引入新依赖，也不需要重构启动逻辑，只要补一个最小 `dev` 脚本即可。

## 4. 候选方案比较

### 方案 A：继续只保留 `start`

优点：

- 不改任何文件

缺点：

- 用户输入的 `npm run dev` 仍然失败
- 使用体验不一致

结论：

- 放弃。

### 方案 B：新增 `dev` 脚本并复用现有 `server.js`

核心思路：

- 在 [package.json](D:\ceshi\aibianji\custom\content-center-demo\package.json) 中新增 `"dev": "node server.js"`
- 保留现有 `start`
- 不增加第三方依赖

优点：

- 改动最小
- 不引入新风险
- 立即满足用户命令

缺点：

- `dev` 与 `start` 当前行为相同，不具备热更新能力

结论：

- 本轮采用方案 B。

## 5. 最终决策

最终采用：

- 重写 [package.json](D:\ceshi\aibianji\custom\content-center-demo\package.json)
- 新增 `dev` 脚本
- 保持 `start` 不变
- 直接按 `npm run dev` 启动服务

这是当前最符合“最小改动原则”的方案。

## 6. 具体实现

本轮改动文件：

- [package.json](D:\ceshi\aibianji\custom\content-center-demo\package.json)
  - 修正项目描述文案
  - 新增 `"dev": "node server.js"`
  - 保留 `"start": "node server.js"`

本轮执行动作：

- 执行 `git pull --ff-only`
- 执行 `npm run dev`
- 用后台方式拉起服务
- 校验 `http://127.0.0.1:3210`

## 7. 本轮优点

- `npm run dev` 现在已经真正可用
- 没有新增任何依赖
- 启动方式和现有项目结构保持一致
- 用户后续本地使用时不会再因为缺脚本而卡住

## 8. 本轮缺点与代价

- `dev` 目前只是 `node server.js`，不带热更新
- Git 的 `safe.directory` 问题这轮没有顺手处理
- 当前仓库仍然没有首个提交，也没有远端

## 9. 验证与结果

本轮实际验证：

## 10. 本轮结论

本轮最重要的结论是：

- [D:\ceshi\aibianji\custom\content-center-demo](D:\ceshi\aibianji\custom\content-center-demo) 现在支持 `npm run dev`
- 服务已经按 `dev` 模式跑起来
- Git 当前的新阻塞点已经从“没有仓库”变成“safe.directory 未配置”
