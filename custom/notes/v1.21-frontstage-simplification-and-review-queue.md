# 版本记录：v1.21-frontstage-simplification-and-review-queue

## 1. 本轮目标

本轮目标是继续收口整木网编辑中台 demo 的前台结构，把页面从“功能都在，但显得杂乱”调整为更接近真实工作台的形态。

这轮优先处理这个问题，是因为用户已经明确指出：

- demo 当前太杂乱
- 前台主结构要收成固定的 5 个主块
- 内容池需要支持多选
- 多选后的内容要么直接转发到对应栏目，要么进入编审台做 AI 编辑后再转发

本轮属于：

- 交互结构重整
- 可维护性提升
- MVP 可测试性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.20-portal-regression-and-wording.md`
- `custom/notes/v1.19-simulated-portal-columns.md`

修改前已经成立的前提有：

- 仪表盘统计结构已经确认，不需要改
- 整木网资讯栏目已经收口为 3 个模拟栏目：行业资讯、企业资讯、市场动态
- AI 配置和失败日志已经在前台隐藏
- 登录、抓取、内容池、AI 编辑和模拟转发链路已经基本跑通

但前台仍存在明显问题：

- 顶部导航还是旧的多模块分散结构
- 抓取任务、抓取源、关键词是分散卡片，信息源口径不统一
- 内容池只能单选，不符合当前测试目标
- 编审台缺少“多篇内容进入后如何切换”的队列视图
- 发布榜还只是日志思维，不是面向“整木网资讯栏目”的结果视图

## 3. 问题分析

这轮表面上是“页面不好看”，但真正的问题不是纯视觉，而是信息架构没有收口：

1. 前台模块边界还是按技术实现拆分，不是按编辑工作流拆分
2. 内容池缺少多选和批量动作，导致“先筛一批，再处理一批”的流程断开
3. 编审台没有待处理队列，多选后也无法形成连贯的编辑链路
4. 发布榜没有明确表达“哪篇内容转发到了哪个模拟栏目”

核心矛盾是：

- 当前系统已经具备一定链路能力
- 但前台结构仍像“演示多个模块”，不像“围绕编辑工作流组织的中台”

本轮最值得优先收口的不是再扩功能，而是把既有能力重新装配成更清晰的工作台。

## 4. 候选方案比较

### 方案 A：仅修改静态文案和导航名称

核心思路：

- 不动主体 DOM 结构
- 只把导航和标题改成“仪表盘、信息源、内容池、编审台、发布榜”

优点：

- 改动小
- 风险低

缺点：

- 页面骨架依旧分散
- 内容池仍然单选
- 编审台没有队列
- 用户看到的仍然是“名字变了，工作流没变”

放弃原因：

- 这只能解决“看起来”的问题，不能解决操作路径杂乱的问题

### 方案 B：保留现有接口，重组前台结构并补足多选与编审队列

核心思路：

- 不新增后端接口
- 通过前端重组，把抓取任务、抓取源、关键词合并到“信息源”
- 给内容池增加多选、批量直接转发、送入编审台
- 给编审台增加待编审队列
- 给发布榜增加整木网 3 个模拟栏目维度的汇总

优点：

- 不破坏既有后端闭环
- 页面结构与工作流一致
- 直接满足本轮用户口径
- 后续接真实整木网接口时，前台结构不需要再重做

缺点：

- 前端交互逻辑会变复杂
- 批量直发目前要复用现有单篇转发接口，前端需要顺序串行处理
- 这轮主要是前台收口，不会顺带把后台独立出来

最终选择原因：

- 这是当前“最小改动但效果最大”的方案

## 5. 最终决策

本轮最终采用方案 B，核心决策如下：

1. 前台主导航固定为 5 个主块：
   - 仪表盘
   - 信息源
   - 内容池
   - 编审台
   - 发布榜
2. 抓取任务、抓取源、关键词合并到“信息源”卡片内部
3. 内容池新增多选与批量动作，但不新增后端批量接口，继续复用现有单篇保存/审核/转发链路
4. 编审台新增“待编审队列”，用于承接内容池多选后送入的文章
5. 发布榜不再展示泛化日志，而是优先展示整木网 3 个模拟栏目的转发结果与栏目累计数

## 7. 本轮优点

这轮相较之前的改进主要有：

- 前台主结构终于和编辑工作流对齐了
- 抓取任务、抓取源、关键词不再散落成多个一级区域
- 内容池具备了“多选 -> 直发 / 送编审”的基础能力
- 编审台具备了承接多篇文章的队列视图
- 发布榜开始真正体现整木网 3 个模拟资讯栏目的结果
- 没有新增后端接口，风险控制在前端层

## 8. 本轮缺点与代价

这轮仍然有明确代价和妥协：

- 批量直发目前是前端顺序循环调用单篇接口，不是服务端原生批量接口
- 编审台队列目前只做前端队列，不是独立的持久化“待办池”
- AI 配置和失败日志这轮仍然只是前台隐藏，不是独立后台页面
- 历史数据库里仍有旧测试数据和旧乱码数据，发布榜可能混入旧记录
- 这轮主要完成的是前台工作流收口，后台权限页没有同步重构

## 10. 本轮结论

本轮最重要的结论是：

- 这套 demo 的主要问题不是功能不够，而是前台工作流没有收口
- 只要不动后端接口，单靠前端重组就能把可测试性明显拉高
- “信息源 / 内容池 / 编审台 / 发布榜”这套结构，比之前分散的模块式导航更适合作为后续真实整木网接口接入前的基线

本轮被证实的假设：

- 批量直发可以先复用单篇接口实现，不必立刻新增后端批量接口
- 编审台队列只要先做前端态，就已经能明显改善多篇内容处理体验

本轮保留的经验：

- 仪表盘结构继续保持不动
- 栏目继续维持 3 个模拟栏目，不再扩
- AI 配置和失败日志仍不放到前台主链路中

## 11. 下一轮建议

下一轮最值得继续做的 3 件事：

1. 实测一轮“内容池多选 -> 送入编审台 -> 切换队列 -> 编辑 -> 转发”
   - 收益最高
2. 给发布榜补“按栏目最近转发 Top 列表”的排序细化
   - 风险低，能继续提升可读性
3. 等整木网真实接口准备好后，再把当前模拟转发适配层替换成真实接口
   - 需要更多外部信息后再做
