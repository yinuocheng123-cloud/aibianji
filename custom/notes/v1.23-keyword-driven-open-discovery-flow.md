# 版本记录：v1.23-keyword-driven-open-discovery-flow

## 1. 本轮目标

本轮要解决的问题，是把前台 demo 的采集模式从“看起来像手动跑任务”收口成更符合业务口径的模式：

- 关键词驱动
- 重点抓取源优先
- 开放发现补充
- 开放发现的新优质来源可转为重点抓取源持续跟踪

之所以优先处理这个问题，是因为用户已经进一步明确了系统定位：

- 重点抓取源是优先范围，不是唯一范围
- 用户先配关键词，再配重点抓取源
- 所有结果统一进入内容池
- 内容池中的开放发现来源，后续要能沉淀回重点抓取源

本轮属于：

- 功能补充
- 工作流修订
- 前台信息架构与采集链路统一

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.22-clean-list-layout-and-source-delete.md`
- `custom/notes/v1.21-frontstage-simplification-and-review-queue.md`

修改前的状态是：

- 前台已经被收成“仪表盘 / 信息源 / 内容池 / 编审台 / 发布榜”五块
- 信息源已经只保留关键词和抓取源
- 内容池、编审台和发布榜已经是清爽列表形态
- 但采集逻辑仍然偏向旧的“单源手动抓取任务”
- 开放发现并没有在前台和任务逻辑里形成明确闭环

也就是说，上一轮解决的是“展示结构太杂乱”，这一轮要继续解决“采集模式与业务口径不一致”。

## 3. 问题分析

表面现象是用户给出了一段新的业务描述，但真正的核心矛盾有三层：

1. 采集入口和业务口径不一致  
   前台已经是面向编辑工作流，但采集动作底层仍然要求先选某个源执行单次任务，更像技术调试，不像“关键词驱动采集”。

2. 抓取源和开放发现之间没有桥梁  
   当前抓取源列表默认都是“已配置来源”，但系统并没有真正表达“重点源优先 + 开放发现补充”的关系。

3. 开放发现没有资产沉淀能力  
   即使开放发现补到内容池，也没有一条直接的动作把它转成重点抓取源，导致“发现 -> 沉淀 -> 持续跟踪”无法闭环。

真正的瓶颈不是“采集不到内容”，而是：

- 用户看不懂系统到底是怎么采集的
- 开放发现只是一次性内容，没有回流到信息源
- 前台缺少一个能直接点的“开始采集”入口

## 4. 候选方案比较

### 方案 A：只改文案，不改采集逻辑

核心思路：

- 保留现有隐藏任务逻辑
- 只在信息源区加说明文字
- 不新增采集入口，不新增开放发现回流动作

优点：

- 改动最小
- 风险最低

缺点：

- 只是“说自己支持”，不是真的“能操作”
- 开放发现仍然无法转成重点抓取源
- 采集模式和页面口径仍然不一致

放弃原因：

- 这轮用户说的是系统工作方式，不只是页面文案

### 方案 B：保留现有底层结构，新增“关键词驱动采集 + 开放来源转重点源”能力

核心思路：

- 继续保留原有单源任务接口，避免破坏旧链路
- 新增一个更符合业务口径的“关键词驱动采集”接口
- 让该接口自动：
  - 使用所有已启用关键词
  - 优先扫描已配置抓取源
  - 再补充开放发现结果
- 内容池中对开放发现文章增加“转为重点源”动作

优点：

- 改动范围集中
- 不用推翻现有持久化结构
- 能真正形成“重点源优先 + 开放发现补充 + 回流沉淀”的闭环
- 前台更符合用户理解

缺点：

- 同时保留了旧任务接口和新采集接口，系统内部会短期并存两套入口
- 开放发现目前还是模拟补充，不是真实全网发现

最终选择原因：

- 这是当前收益最高、同时对既有代码破坏最小的方案

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 已配置抓取源统一视为“重点抓取源”  
   不额外引入复杂的优先级字段，避免修改数据库结构。对当前 MVP 来说，用户在信息源里录入的抓取源，就是重点抓取源。

2. 新增“按关键词开始采集”作为前台显性入口  
   用户不再需要理解隐藏任务区；只需要理解“系统会拿已启用关键词去重点源抓取，并补充开放发现”。

3. 开放发现文章通过 `sourceId = 0` 与重点源文章区分  
   这样不需要扩表，也能在内容池中明确标记“重点抓取源 / 开放发现”。

4. 内容池中为开放发现文章补一个“转为重点源”动作  
   该动作会把文章来源沉淀到抓取源列表里，形成持续跟踪能力。

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/index.html`

本轮对前台骨架做了这些收口：

- 信息源区顶部新增了显性的采集入口：
  - `#discover-button`
  - `#discover-status`
- 信息源说明文案改成新的业务口径：
  - 关键词驱动
  - 重点源优先
  - 开放发现补充
- 抓取源卡片的角标改成“重点抓取源”
- 内容池说明文案改成“所有抓取结果统一进入内容池”
- 内容池默认提示里加入“开放发现来源可转为重点抓取源”
- 详情区“来源”标签改成“来源 / 采集方式”
- 静态资源版本号升级到：
  - `v26-keyword-discovery`

### 6.2 `custom/content-center-demo/public/styles.css`

本轮只补了与新采集入口直接相关的样式：

- 新增 `source-action-bar`
- 新增 `collection-note`

作用是：

- 让信息源区顶部有一个更清晰的采集动作区
- 不破坏上一轮已经稳定下来的极简列表布局

### 6.3 `custom/content-center-demo/public/app.js`

本轮前端逻辑新增和调整如下：

1. 新增采集入口元素引用
   - `discoverButton`
   - `discoverStatus`

2. 新增来源识别辅助函数
   - `getTrackedSourceForArticle(article)`
   - `getArticleCollectionMode(article)`
   - `canPromoteSource(article)`

3. 新增前台采集动作
   - `runDiscovery()`
   - 直接调用新的 `/api/tasks/discover`
   - 自动收集已启用关键词

4. 新增开放来源转重点源动作
   - `promoteArticleSource(articleId)`
   - 内容池中对开放发现文章渲染“转为重点源”

5. 重写了部分展示逻辑
   - `renderCollectionStatus()`：展示当前启用的关键词数、重点源数和最近一次采集结果
   - `renderSources()`：明确把抓取源解释成“重点抓取源”
   - `renderKeywords()`：关键词说明改成采集入口词
   - `renderArticles()`：给文章打上“重点抓取源 / 开放发现”标签，并给开放发现文章补转源动作
   - `renderDetail()`：来源区域显示来源名和采集方式
   - `renderSelectionSummary()`：提示统一成“直接发布 / 送编审 / 开放发现可沉淀”

### 6.4 `custom/content-center-demo/src/routes/api-router.js`

本轮新增了两个关键接口：

- `POST /api/tasks/discover`
  - 使用当前已启用关键词启动“关键词驱动采集”
  - 由后端统一组织重点源优先和开放发现补充

- `POST /api/articles/:id/promote-source`
  - 把内容池中的开放发现来源转成重点抓取源
  - 如果域名已存在，则更新已有源并把文章绑定回该源
  - 如果域名不存在，则新建抓取源

同时补了两个辅助函数：

- `guessSourceTypeFromUrl()`
- `buildPromotedSourceFromArticle()`

### 6.5 `custom/content-center-demo/src/services/crawl-service.js`

这是本轮最核心的服务层改动。

1. 把真实抓取文案从“白名单”统一调整为“重点抓取源”

2. 扩展 `createSimulatedArticle()`
   - 支持 `mode = "priority"` 和 `mode = "open"`
   - 同一套模拟文章生成逻辑，按采集模式输出不同摘要、SEO 和来源说明

3. 新增 `createOpenDiscoveryArticle()`
   - 用固定开放来源样例生成开放发现文章
   - 当前样例里包含：
     - 图森官网
     - 木作品牌号
     - 家居产业观察

4. 新增 `processCandidateArticle()`
   - 统一封装去重判断、疑似重复保留与入池逻辑

5. 新增 `runDiscoveryTask()`
   - 输入：关键词列表
   - 执行方式：
     - 先从所有已启用重点抓取源中按关键词采集
     - 再按同一个关键词补充一条开放发现文章
   - 输出：
     - 统一写入内容池
     - 写入任务记录
     - 写入抓取日志

## 7. 本轮优点

相比上一轮，本轮最大的提升有：

- 采集模式终于和业务描述一致了，不再只是隐藏技术任务
- 前台出现了真正可点的“按关键词开始采集”
- 信息源和内容池之间形成了清晰的关系：
  - 关键词驱动
  - 重点源优先
  - 开放发现补充
- 开放发现不再是一次性结果，已经能沉淀回重点抓取源
- 这条闭环不需要改数据库表结构，延续了“最小改动”原则

## 8. 本轮缺点与代价

这一轮仍然有明确代价和未完成点：

- 开放发现目前还是模拟发现，不是真实全网搜索
- 旧的 `/api/tasks/run` 仍然保留着，内部短期存在两种采集入口
- “转为重点源”现在是依据文章原始链接域名推断来源，规则还比较朴素
- 抓取源依然没有单独的“重点 / 非重点”字段，只是默认把已配置源视为重点源
- 前台虽然已经更符合业务语义，但后台的 AI 配置和失败日志仍然只是隐藏，不是独立后台页

## 10. 本轮结论

本轮最重要的结论是：

- 对当前 MVP 来说，“已配置抓取源即重点抓取源”是一个足够稳妥的中间方案
- 不改数据库结构，也能先把“关键词驱动 + 重点源优先 + 开放发现补充 + 来源回流沉淀”跑通
- 前台用户现在更容易理解系统的采集逻辑，而不是把它当成一堆散乱功能块

被证实的假设：

- 开放发现文章用 `sourceId = 0` 与重点源文章区分，是可行的
- 内容池里加“转为重点源”按钮，可以自然地承接来源沉淀动作
- 用新的关键词驱动任务接口，不需要推翻原有单源任务结构

被修正的旧判断：

- 之前默认认为“信息源只要收清爽就够了”，这轮证明还不够，必须把采集动作本身也说清楚、做出来

## 11. 下一轮建议
