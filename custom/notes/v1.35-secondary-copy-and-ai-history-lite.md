# 版本记录：v1.35-secondary-copy-and-ai-history-lite

## 1. 本轮目标

本轮继续承接前台基础收口工作，不扩新功能，重点解决两件事：

1. 继续把前台剩余次级动态文案并入统一 copy/helper 体系
2. 再压一轮编审台里的 AI 历史记录和辅助提示块，让右侧工作区更轻

本轮属于：

- 可维护性提升
- 前台体验收口

之所以优先处理，是因为 `v1.34` 已经把编审台主路径按钮、placeholder 和发布结果提示收了一轮，当前最明显的不统一点主要剩在：

- 信息源区的状态提示和次级按钮
- 编审台 AI 历史块的空状态、辅助说明和行级展示

## 2. 背景与上下文

本轮承接：

- [v1.34-workbench-copy-lite-and-button-unification.md](D:\ceshi\aibianji\custom\notes\v1.34-workbench-copy-lite-and-button-unification.md)
- [v1.33-dynamic-copy-and-meta-tightening.md](D:\ceshi\aibianji\custom\notes\v1.33-dynamic-copy-and-meta-tightening.md)

修改前的状态：

- `v1.34` 已经统一了编审台右侧一批高频按钮与结果提示
- 但信息源区仍有一部分运行时动态提示直接写在渲染函数里
- 次级按钮如“保存关键词 / 保存抓取源 / 提取预览 / 全选 / 清空”虽然页面能用，但还没正式纳入 copy/helper 统一管理
- AI 历史块仍是较粗的 `mini-item` 列表，没有独立空状态，也没有轻量辅助说明

已知约束：

- 不改整木网模拟发布逻辑

## 3. 问题分析

表面现象是“前台只剩一些小文案和一个 AI 历史块还不够顺”，但核心矛盾其实是：

- 前台已经开始做统一 copy 体系，如果次级按钮和状态提示还散落在渲染函数里，后续每一轮小改都会继续回到“到处找字符串”的状态
- 编审台右侧主表单已经压轻，但 AI 历史块仍然显得像技术日志列表，不像编辑工作台里的轻记录区

主要原因有三类：

1. 次级按钮和状态提示还没有完全进入统一 copy/helper
2. 信息源采集状态仍由渲染函数直接拼接文本，维护入口不统一
3. AI 历史块只有最小可用渲染，没有“空状态 / 轻说明 / 结果摘录”这三个能明显降低噪音的层次

## 4. 候选方案比较

### 方案 A：只改静态 HTML 文案和样式

核心思路：

- 只改 `index.html` 里的按钮和说明文案
- 只在 `styles.css` 里压轻 AI 历史区域

优点：

- 改动快
- 初始页面会更轻

缺点：

- 运行时动态文案仍然分散
- 登录后和交互后的界面仍可能漂移
- 不能真正解决 copy 入口不统一的问题

放弃原因：

- 这会重复 `v1.32` 之前“静态一套、运行时一套”的老问题

### 方案 B：继续扩统一 copy/helper，并顺手重做 AI 历史块渲染

核心思路：

- 扩展 `FRONTSTAGE_COPY`
- 用 helper 收掉信息源状态提示和剩余次级按钮文案
- 在不改接口的前提下，把 AI 历史块改成更轻的行级记录

优点：

- 与 `v1.32 ~ v1.34` 的方向一致
- 改动面可控
- 能同时改善维护性和页面观感

缺点：

- 仍然不是完整的前台 copy system 终态
- 还有一部分更次级的动态文案后续要继续收

最终选择原因：

- 最符合当前迭代阶段
- 收益高，风险低

## 5. 最终决策

本轮采用方案 B。

关键决策如下：

1. 继续把次级按钮和动态状态提示收进 `FRONTSTAGE_COPY`
2. 新增最小 helper，避免信息源状态提示和 AI 历史摘要继续散写
3. AI 历史块不新增接口字段，只基于现有 `type / model / createdAt / outputText` 做轻量展示
4. 保持结构不变，只压轻右侧历史与辅助提示区域

## 8. 本轮缺点与代价

本轮仍有明确边界：

- 还没把所有次级动态文案全部收完
- AI 历史块虽然更轻了，但还没有做更细的筛选或折叠
- 当前仍然依赖现有接口返回的最小字段，无法展示更丰富的 AI 处理上下文
- 前台脚本虽然已拆层，但仍是原生全局脚本方案，不是更彻底的组件化实现

也就是说，这轮解决的是“剩余次级 copy 和历史块视觉噪音”的问题，不是彻底完成前台文案系统或工作台组件体系。

本轮完成了以下验证：

- `node --check custom/content-center-demo/public/scripts/app-utils.js`

结果：全部通过。

2. 页面访问与资源版本检查

- 访问 `http://127.0.0.1:3210`
- 已确认页面返回 `200`
- 已确认首页内容包含：
  - `v38-secondary-copy-ai-history-lite`
  - `ai-history-note`
  - 新版信息源状态文案和 AI 历史区说明

结果：通过。

3. 回归脚本验证

- 执行 `npm run smoke:e2e`
- 再执行清理模式 `$env:SMOKE_CLEANUP='1'; npm run smoke:e2e`

结果：两次都失败。

失败原因：

- 当前本地 SQLite 中存在历史测试文章
- 新生成的测试关键词在回退模拟抓取后，与旧文章命中相似度 `80%`
- 因此脚本在“抓取任务未生成可处理文章”这一断言处中断

结论：

- 这轮改动本身没有引入语法错误或页面不可访问问题
- 但固定 smoke 回归当前受历史测试数据干扰，后续需要单独收这个验证基线问题

## 10. 本轮结论

本轮最重要的结论是：

- 当前前台继续收口时，最值得优先处理的已经不是大结构，而是“剩余次级文案入口”和“局部高频块的噪音压缩”

被证实的判断：

- 次级按钮和状态条如果不继续进统一 helper，后续维护成本会明显上升
- AI 历史块用轻量行级记录替代简单 `mini-item`，更符合当前工作台风格

被修正的旧判断：

- `v1.34` 里主要收的是编审台主动作；这轮证明信息源状态条和 AI 历史块仍然是明显的后续收口点

后续应默认继承的经验：

- 新出现的动态文案，优先放进 `FRONTSTAGE_COPY` 与 helper，而不是直接拼在 render 函数里
- 工作台里的“辅助区块”如果没有空状态和轻说明，很容易重新显得杂乱

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 继续把前台剩余更次级的动态文案也收进统一 copy/helper  
   收益最高，能继续降低后续散写字符串的概率。

2. 单独修正 `smoke:e2e` 被历史测试数据干扰的问题  
   风险最低，但对后续每一轮回归验证的价值很高。

3. 再压一轮编审台中 AI 处理区和发布结果区的间距与层级  
   在不改流程的前提下，还能继续让右侧工作区更稳。
