# 版本记录：v1.29-workbench-lite-and-legacy-reduction

## 1. 本轮目标

本轮要解决两个连续遗留问题：

1. 继续清理 `app.js` 中剩余的历史辅助逻辑和同名旧实现
2. 把编审台右侧表单再压轻一轮，让它更像清爽的工作区，而不是一整段连续表单

之所以优先做这两件事，是因为 `v1.28` 已经把登录与 bootstrap 基础问题收住了，当前影响体验判断的主要矛盾，已经回到：

- 前端单文件里仍有一部分演化遗留函数
- 编审台右侧信息层次还不够轻，用户进入后第一眼仍然偏“长表单感”

本轮属于：

- 可维护性提升
- 前台结构收口
- 交互表达优化

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.28-bootstrap-login-and-legacy-cleanup.md`
- `custom/notes/v1.27-frontstage-list-cleanup-and-sorting.md`

修改前的状态是：

- `v1.28` 已经修好了未登录被误判为初始化失败的问题
- `app.js` 中一部分最明显的旧函数已经被改成 `Legacy`
- 但仍然存在剩余的同名重复函数：
  - `getFilteredArticles`
  - `renderSelectionSummary`
- 编审台右侧虽然功能已经齐全，但区域组织还是连续堆叠：
  - 原文参考
  - AI 动作
  - 修订内容
  - 发布设置
  - 编审动作
  这些内容混在一整段里，浏览成本偏高

已知约束：

1. 这轮不新增功能，不改接口
2. 只做最小结构调整，不推翻现有编辑链路
3. 继续沿用“先局部收口，再考虑拆文件”的策略

## 3. 问题分析

当前问题分两层：

### 3.1 代码层问题

`app.js` 仍保留部分“旧实现 + 当前实现”并存的情况。  
虽然浏览器最终以文件后部定义为准，但这种状态会带来两个问题：

- 后续代理继续读文件时，仍然要先判断哪段才是当前生效逻辑
- 每继续迭代一次，误改旧实现的概率都会上升

### 3.2 界面层问题

编审台右侧目前的核心问题，不是字段太多，而是层次太平：

- 原文参考和修订内容没有明显分区
- AI 动作夹在中间，像工具条而不像单独能力区
- 发布设置和编审动作之间缺少结构分隔

这会让用户进入编审台后，更容易感觉是在填写一张大表，而不是处理一篇稿件。

## 4. 候选方案比较

### 方案 A：只改样式，不改 HTML 结构

核心思路：

- 保持右侧 DOM 结构不变
- 仅通过边距、边框、背景色做视觉区分

优点：

- 改动最小
- 几乎没有行为风险

缺点：

- 结构层次仍然靠视觉猜
- 后续继续压轻时空间很有限

放弃原因：

- 这只能“修饰”，不能真正解决右侧分区不明确的问题

### 方案 B：保留全部字段与交互，只重排成 5 个轻分区

核心思路：

- 不改字段 id、不改按钮逻辑
- 只把右侧组织成：
  - 原文参考
  - AI 快捷处理
  - 修订内容
  - 发布设置
  - 发布动作
- 同时继续把剩余同名旧函数改成 `Legacy`

优点：

- 不影响现有逻辑绑定
- 用户视角更清楚
- 后续继续压轻或拆分更容易

缺点：

- HTML 和 CSS 需要一起改
- 仍然没有真正拆分 `app.js`

最终选择原因：

- 这是当前最符合“最小改动、收益明显”的方案

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 继续清理剩余同名旧函数，避免文件里继续出现双定义
2. 编审台右侧不删字段、不改接口，只改成分区式布局
3. AI 动作保留，但从“夹在表单中间”改成独立轻区块
4. 发布设置和编审动作分离，降低大表单感

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

本轮继续清理了一批历史遗留实现：

- `getFilteredArticles` -> `getFilteredArticlesLegacy`
- `renderSelectionSummary` -> `renderSelectionSummaryLegacy`

同时把原先已经只剩历史意义的提示 helper 继续收口：

- `upsertHint` -> `upsertHintLegacy`
- `removeHint` -> `removeHintLegacy`

并移除了 `configureFrontstageLayout()` 中“先加 hint 再立刻删掉”的旧调用。

这样做的作用是：

- 当前生效逻辑边界更清楚
- 后续继续读 `app.js` 时，不会再把这些旧函数误当成主路径

### 6.2 `custom/content-center-demo/public/index.html`

本轮把编审台右侧改成了 5 个分区：

1. 原文参考
2. AI 快捷处理
3. 修订内容
4. 发布设置
5. 发布动作

保留了所有原有字段、按钮和 id：

- `new-title`
- `summary`
- `rewritten-content`
- `category`
- `tags`
- `seo-title`
- `seo-description`
- `source-note`
- `review-comment`

同时把资源版本升级到：

- `v32-workbench-lite`

### 6.3 `custom/content-center-demo/public/styles.css`

新增了编审台右侧轻分区样式：

- `.detail-panel`
- `.editor-section`
- `.section-mini-head`
- `.compact-editor-grid`
- `.compact-actions`
- `.editor-actions-section`
- `.compact-review-box`
- `.compact-history-box`

作用是：

- 把右侧从“大段连续表单”变成可快速扫视的工作区
- 保持原有苹果极简风，不重新做重装饰

## 7. 本轮优点

相比 `v1.28`，本轮的正向效果主要有：

- `app.js` 中剩余最明显的同名旧函数继续被清出主路径
- 编审台右侧结构更清晰，进入后更容易判断先看什么、先改什么
- AI、修订、发布设置、发布动作之间的边界明确了
- 不改字段 id，因此行为风险低

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. `app.js` 仍然是单文件
2. 旧逻辑并没有一次性彻底删光，只是继续降低干扰
3. 编审台右侧虽然更轻了，但字段数量本身仍然不少，后续若继续加功能仍需要克制

## 10. 本轮结论

本轮最重要的结论是：

- 编审台是否“清爽”，关键不在少字段，而在明确分区

被证实的假设：

- 保留全部现有字段，只通过分区重排，就能显著降低大表单感
- 在不拆文件的前提下，继续把旧实现改成 `Legacy`，仍然是当前性价比最高的维护方式

被修正的旧判断：

- 之前默认“编审台已经能用，主要靠样式再压一压就行”，这次证明还是需要轻量结构重排

以后应默认继承的经验：

- 前台工作区优先按“任务步骤”分区，而不是按字段堆叠
