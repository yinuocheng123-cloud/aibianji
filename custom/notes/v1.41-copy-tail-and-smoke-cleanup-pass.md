# 版本记录：v1.41-copy-tail-and-smoke-cleanup-pass

## 1. 本轮目标

本轮目标有两个：

1. 继续把前台剩余更细碎的动态提示、空态说明和次级状态文本并入统一 `copy/helper` 体系。
2. 修复 `smoke:e2e` 在连续执行时仍会被历史测试数据干扰的问题，同时再压一轮编审台右侧与发布榜的行级密度。

这轮属于：

- 可维护性提升
- Bug 修复
- 可测试性提升

之所以优先处理这两个问题，是因为前台已经进入“继续收口而不是继续扩功能”的阶段。此时如果 copy 继续散落、smoke 继续偶发假红，后续任何真实接口适配都会被基础噪音拖慢。

## 3. 问题分析

表面现象有两类：

1. 前台仍有少量动态文本在 helper 或 render 层直接拼接，导致 copy 统一还不彻底。
2. `smoke:e2e` 的 cleanup 模式偶发失败，表现为新生成文章被上一轮测试文章判成高相似重复。

核心矛盾分别是：

- 表达层方面：主按钮和主说明已经统一，但更细的 meta 文本、会话摘要、状态串仍有局部散写。
- 验证层方面：cleanup 接口之前主要按来源、关键词和部分标题字段清理，遗漏了文章的修订标题、摘要、正文、SEO 等字段，导致上一轮文章即使来源被删，仍可能残留足够多的可比对文本。

主要原因：

- copy 收口推进到后期后，剩下的问题都不是“缺模块”，而是“细碎文本还分散”。
- smoke 的问题不是脚本单侧 bug，而是清理匹配范围仍偏窄，无法完全覆盖上一轮测试稿件留下的可比对文本。

## 4. 候选方案比较

### 方案 A：继续只改前端，不动 smoke 清理接口

核心思路：

- 只继续统一前台文案和压样式。
- 对 smoke 偶发失败先接受，继续靠重复跑脚本解决。

优点：

- 改动面最小。
- 不需要动后端。

缺点：

- 无法解决 cleanup 模式的假红。
- 后续每轮验证仍可能被历史数据干扰。

放弃原因：

- 收口阶段最怕“验证不稳”，这个问题继续拖着会放大后续成本。

### 方案 B：同时收 copy 尾巴，并扩展 cleanup 匹配范围

核心思路：

- 继续把 helper / render 中剩余的模板串并入统一 `copy/helper`。
- 扩展 `/api/testing/cleanup` 对文章字段的匹配范围，让 smoke 能真正清干净上一轮测试稿。

优点：

- 同时解决表达层和验证层的尾部问题。
- 改动局部，风险可控。
- 对后续每轮迭代收益最高。

缺点：

- 需要同时改前端和后端。
- cleanup 匹配范围变宽，需要确保只作用于 smoke 标记文本。

最终选择：

- 采用方案 B。

## 5. 最终决策

最终采用的方案是：

1. 在前台继续补一批模板和 helper，但不再重构架构。
2. 在 cleanup 接口中增加文章更多字段的匹配清理，确保连续 smoke 能稳定通过。
3. 在样式层继续压轻编审台右侧和发布榜的行级密度，而不是新增块或删字段。

关键设计决策：

- 不强行替换所有旧 helper，而是增量补一批更明确的新 helper，并在 render 层切过去。
- cleanup 不走“按数据库全量重置”，仍坚持“按 smoke 标记文本精准清理”。
- UI 密度继续做轻，不改主结构，避免用户重新适应。

这轮仍有明确代价和未完成项：

- `app-utils.js` 中旧 helper 仍然存在部分重叠能力，本轮没有再做更激进的清理。

## 10. 本轮结论

本轮最重要的结论有三点：

1. copy 收口进入后期后，真正值得继续做的是“细碎状态文本”和“helper 内模板串”，而不是再去改结构。
2. smoke 稳定性不能只靠脚本前置清理，还要让 cleanup 真正覆盖测试稿可能残留的正文与修订字段。
3. 编审台是否像正式工作台，关键仍然是密度和层级，而不是继续删模块。

这些结论后续应默认继承。

## 11. 下一轮建议

   - 这一步应放在基础体验完全稳定之后。
