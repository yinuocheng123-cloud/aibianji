# 版本记录：v1.9-login-form-fallback

## 1. 本轮目标

本轮目标是继续修复“点击登录进入中台，进不去”的问题，并把登录入口改成在浏览器环境下更稳的双通道模式。

为什么这个问题仍值得继续优先处理：

- `v1.8` 已经处理了静态资源刷新，但用户反馈仍然“进不去”。
- 这说明仅靠缓存刷新不够，需要让登录入口在“有前端脚本”和“没有前端脚本”两种情况下都能工作。

本轮属于：

- Bug 修复
- 可测试性恢复
- 交互链路加固

## 3. 问题分析

### 3.1 表面现象

用户仍然反馈：

- 点击“登录进入中台”后，不能真正进入中台主界面。

### 3.2 进一步排查结果

本轮继续确认了 3 件事：

1. 后端 JSON 登录接口仍然正常。
2. 页面能正常打开，服务也正常响应。
3. 当前登录表单仍主要依赖前端 `submit -> fetch -> refreshData` 这条链。

### 3.3 核心矛盾

真正的矛盾是：

- 只要登录动作完全依赖前端脚本绑定，浏览器缓存、旧脚本、脚本异常或事件没绑上时，点击按钮就可能什么都不发生。

也就是说，之前虽然“接口能登”，但“按钮点击一定能进”这件事并没有被保证。

## 4. 候选方案比较

### 方案 A：继续只修前端 `fetch` 登录逻辑

优点：

- 变动范围小。

缺点：

- 仍然依赖脚本必须正常加载和正常绑定事件。
- 无法彻底兜住“用户页面还是旧 JS”这类问题。

放弃原因：

- 这正是当前问题最可能卡住的地方。

### 方案 B：增加浏览器原生表单登录兜底

核心思路：

- 登录表单本身设置 `method="post"` 与 `action="/api/auth/login"`。
- 后端登录接口同时支持：
  - JSON 请求：继续返回 JSON，供当前前端逻辑使用
  - 表单请求：成功时 `302` 回首页，失败时 `302` 回带错误参数的地址

优点：

- 即使前端脚本没接上，点登录也能成功提交。
- 不推翻现有 AJAX 登录逻辑。
- 是对当前问题最稳的兜底。

缺点：

- 后端登录接口要同时兼容 JSON 与表单两种请求形态。

最终采用原因：

- 这是“最小新增复杂度，换取最大登录稳定性”的方案。

## 5. 最终决策

本轮最终采用：

1. 登录表单补原生提交能力
2. 登录接口兼容表单请求并返回 302
3. 前端支持通过查询参数回显登录失败文案

关键设计决策：

- 不废弃现有 AJAX 登录逻辑，只做兜底增强
- 表单提交成功后依然回到 `/`，让现有前端初始化逻辑继续接管
- 错误密码走 `/?login_error=invalid_credentials`，避免静默失败

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/index.html`

- 给登录表单增加：
  - `method="post"`
  - `action="/api/auth/login"`
- 将静态资源版本继续更新为 `v13-login-fallback`

作用：

- 即使浏览器没有正确执行前端登录脚本，按钮也会走原生表单提交。

### 6.2 `custom/content-center-demo/src/shared/utils.js`

- 扩展 `readRequestBody`
- 新增对 `application/x-www-form-urlencoded` 的解析支持

作用：

- 让后端登录接口能同时读 JSON 请求和浏览器原生表单请求。

### 6.3 `custom/content-center-demo/src/routes/api-router.js`

- 新增 `isHtmlFormRequest`
- 登录路由 `POST /api/auth/login` 现在支持两种响应模式：
  - JSON 登录：继续返回 `200 + JSON + Set-Cookie`
  - 表单登录：
    - 成功：`302 -> /`，并写入会话 cookie
    - 失败：`302 -> /?login_error=invalid_credentials`

作用：

- 把“按钮能点”和“真正能进中台”之间的链路从单一脚本依赖，改成脚本和原生表单双保险。

### 6.4 `custom/content-center-demo/public/app.js`

- 新增 `applyLoginQueryState`
- 若 URL 带有 `login_error`，则直接在登录浮层显示中文错误文案
- 显示后再清掉地址栏查询参数

作用：

- 表单兜底链路在输错账号密码时，也能给出可见反馈。

## 7. 本轮优点

当前方案的主要优点：

1. 登录入口从“只有脚本路径”变成了“双通道路径”。
2. 即使用户浏览器还残留旧 JS，点击登录也能通过原生表单进入系统。
3. 不推翻现有前端登录逻辑，兼容性更稳。
4. 登录失败不再只靠控制台或静默返回，而是能直接显示错误。

## 8. 本轮缺点与代价

这轮也有明确代价：

1. 登录接口现在同时承担 API 和浏览器表单两种职责，复杂度略有增加。
2. 路由层里仍有一些历史乱码文案，本轮没有顺手全清。
3. 当前验证重点是“登录可用性”，没有顺手收口 `smoke:e2e` 的历史重复问题。

## 10. 本轮结论

本轮最重要的结论是：

- 这次“点击登录进不去”的关键修复，不是继续改账号逻辑，而是把登录入口做成浏览器原生表单也能成功。
- 现在登录按钮即使不依赖前端脚本，也能建立会话并回到首页。

以后应默认继承的经验：

- 对登录、支付、确认提交这类关键入口，不能只留一条纯前端脚本路径，最好保留一条浏览器原生兜底链路。
