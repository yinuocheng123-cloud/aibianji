# 版本记录：v1.34-workbench-copy-lite-and-button-unification

## 1. 本轮目标

本轮继续承接前台基础收口工作，不扩新功能，重点解决两件事：

1. 把前台剩余的编审台主路径动态按钮文案继续并入统一 copy/helper 体系。
2. 再压一轮编审台右侧的提示文案和结果提示，让工作区更轻，不再有“局部已经统一、局部还在散写”的割裂感。

本轮属于：

- 可维护性提升
- 前台体验收口

之所以优先处理，是因为 `v1.33` 已经把内容池和发布榜收得更像文章清单，当前最显眼的“不统一感”主要集中在编审台右侧。

## 2. 背景与上下文

本轮承接：

- [v1.33-dynamic-copy-and-meta-tightening.md](D:\ceshi\aibianji\custom\notes\v1.33-dynamic-copy-and-meta-tightening.md)
- [v1.32-front-copy-unification.md](D:\ceshi\aibianji\custom\notes\v1.32-front-copy-unification.md)

修改前的状态：

- `v1.33` 已经统一了动态栏目按钮标题、直接发布备注、发布结果主提示
- 但编审台右侧仍存在几类残留：
  - `保留疑似稿 / 审核通过 / 驳回修改 / 归档` 这些按钮仍然主要依赖静态 HTML
  - `AI 快捷处理` 按钮文案没有纳入统一 copy
  - 编审备注输入框 placeholder 和发布结果空态提示还比较重
  - 编审台说明文案也偏长

已知约束：

- 不改后端接口
- 不改内容池与发布榜结构
- 不推翻现有前台脚本拆层边界
- 保持整木网仍然是模拟栏目，不提前接真实接口

## 3. 问题分析

表面问题是“右侧还有几句文案没统一”，但核心矛盾其实是：

- 前台现在已经进入稳定打磨阶段，如果编审台这条主路径继续混合使用静态 HTML 文案和运行时散写文案，后续每次小改都会继续回到“到处找字符串”的状态

主要原因有三类：

1. 剩余按钮文案还没有统一配置来源  
   尤其是编审动作和 AI 快捷处理这组按钮，属于用户高频点击区域。

2. 右侧提示文字仍然偏解释型  
   `detail-empty`、`review-comment` placeholder、发布结果提示文字都偏长，和当前“轻工作台”方向不一致。

3. 发布结果虽已统一 helper，但表现仍偏重  
   `v1.33` 前是长结果字符串；本轮前虽然已经开始缩短，但还没有把“未发布时的空提示”也统一成同一套状态表达。

## 4. 候选方案比较

### 方案 A：只改静态 HTML 提示

核心思路：

- 只把 `index.html` 里的提示改短
- 不动脚本层 copy 和 helper

优点：

- 改动最快
- 页面初始观感会变轻

缺点：

- 运行时文案仍然分散
- 登录后与交互后状态仍可能重新漂移
- 不能真正解决“维护入口不统一”的问题

放弃原因：

- 这会重复 `v1.32` 之前“静态一套、运行时一套”的老问题

### 方案 B：继续补统一 copy，并把编审台提示链路一并压轻

核心思路：

- 扩展 `FRONTSTAGE_COPY`
- 让按钮、placeholder、空结果提示统一走同一来源
- 编审台右侧只做轻量表达压缩，不改流程

优点：

- 与 `v1.32 / v1.33` 的方向一致
- 改动面小
- 用户体感改善明显

缺点：

- 仍然不是完整的前台 copy system 终态
- 一些次级动态文案后续还要继续收

最终选择原因：

- 最符合当前工程阶段
- 能继续压低维护成本，同时保持低风险

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 在 `FRONTSTAGE_COPY` 中继续补齐编审台高频按钮、placeholder、状态提示和结果模板。
2. 在 `app-utils.js` 中继续沿用 helper 方式，而不是重新引入新层或新组件。
3. 在 `app-render.js` 中把“已发布结果”和“未发布提示”统一成同一组件状态切换。
4. 静态 HTML 同步改成更短的编审台说明和结果空态提示，避免首次进入与运行时出现不一致。

## 7. 本轮优点

本轮的主要优点：

- 编审台右侧高频按钮文案终于不再依赖静态 HTML 单独维护
- 输入提示、空结果提示、发布结果提示形成了一套更一致的表达链
- 右侧工作区比上一轮更轻，不再有明显的“重提示块”
- 继续沿用现有 copy/helper 方向，没有引入重构风险

改善最明显的质量属性：

- 可维护性
- 前台一致性
- 页面清爽度

## 8. 本轮缺点与代价

这一轮仍然有明确边界：

- 还没有把所有动态文案全部统一完，当前优先覆盖的是编审台主路径
- AI 快捷处理按钮虽然进入统一 copy 了，但还没有把更细的反馈提示也一起统一
- 发布结果这轮为了更轻，弱化了 URL 直接展示，详细链接更多依赖发布榜查看
- 前台仍是原生全局脚本分层，不是更细颗粒度组件方案

也就是说，这轮解决的是“主路径表达一致性”，不是“前台文案系统彻底完成”。

## 10. 本轮结论

本轮最重要的结论是：

- 对当前这套前台 MVP 来说，真正让工作区“轻下来”的，不是删字段，而是把提示链路统一并减短

被证实的假设：

- 编审台右侧最适合继续收的是按钮文案、placeholder 和结果提示
- 发布结果的“空态提示”和“成功提示”统一成同一组件状态后，页面观感更稳定

被修正的旧判断：

- `v1.33` 里更多聚焦内容池和发布榜，这轮证明编审台右侧仍然是下一块最值得继续压轻的区域

建议后续默认继承的经验：

- 高点击区域的按钮与提示，应该优先纳入统一 copy/helper
- 结果区如果允许空白，很容易在页面里显得“像没做完”；轻提示态比空白块更稳

## 11. 下一轮建议

下一轮最值得继续做的 1~3 件事：

1. 继续把前台剩余次级动态文案并入统一 copy/helper  
   收益最高，因为这会继续降低后续维护时的查找成本。

2. 再压一轮编审台的 AI 历史记录与操作提示块  
   风险最低，因为不需要改接口，只是继续前台收口。

3. 等前台主链路稳定后，再考虑真实整木网接口适配层  
   这件事仍然需要更多接口资料，不适合提前展开。
