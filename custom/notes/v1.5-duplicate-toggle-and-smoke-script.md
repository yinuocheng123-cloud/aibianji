# 版本记录：v1.5-duplicate-toggle-and-smoke-script

## 1. 本轮目标

本轮目标是在 `v1.4` 的基础上继续把 MVP 从“能手工测”推进到“能稳定回归测”，重点完成两件事：

- 给抓取任务补“保留疑似重复”显式开关
- 补一条可重复执行的本地端到端回归脚本

为什么这轮值得优先处理：

- `v1.4` 已经把主流程跑通，但回归仍依赖手工点击
- 疑似重复现在默认保留，虽然适合编辑验证，但缺少显式开关，无法模拟更严格的入池策略

本轮属于：

- 功能补充
- 可测试性提升
- 可维护性提升

## 3. 问题分析

本轮要解决的核心问题有两个：

1. 疑似重复当前只有一种默认策略

表面现象：

- 当前相似度 60%-79% 的内容会直接入池，并打上“疑似重复”

核心矛盾：

- 这种策略适合编辑验证，但不适合做“严格去重”场景演示
- 没有显式开关，就无法快速切换“保留”与“跳过”的不同业务口径

2. 端到端回归还没有固定脚本

表面现象：

- 当前要验证整条链路，需要手工登录、建源、建词、跑任务、AI、保存、提审、发布

核心问题：

- 手工回归成本高
- 每轮验证不完全一致
- 后续再修代码时，容易重新踩到已经解决过的问题

## 4. 候选方案比较

### 方案 A：只在 README 写手工回归步骤

优点：

- 改动最小
- 没有新增脚本维护成本

缺点：

- 仍然依赖人工操作
- 无法形成稳定回归基线

结论：

- 放弃。它只能补说明，不能真正提高可回归性。

### 方案 B：补任务开关 + 固化脚本

核心思路：

- 任务面板新增“保留疑似重复”复选框
- 后端 `runTask` 接收该选项，并在日志中记录实际策略
- 新增 `smoke:e2e` 脚本，把整条工作流固化为单命令回归

优点：

- 用户界面和后端策略一致
- 可快速切换不同去重口径
- 回归验证从“靠记忆操作”变成“跑脚本”

缺点：

- 需要同时改前端、后端和说明文档
- 回归脚本会向本地数据库写入测试数据

结论：

- 本轮采用方案 B。

## 5. 最终决策

最终采用：

- 在抓取任务区显式加入“保留疑似重复”开关，默认开启，保持与当前行为兼容
- 后端在任务执行时识别该开关：
  - 开启时：疑似重复文章继续入池
  - 关闭时：疑似重复文章直接跳过，并写入任务日志
- 新增本地回归脚本 `npm run smoke:e2e`，固化整条主流程

这个方案延续了 `v1.4` 的增量收口策略，没有扩大到数据库 schema 调整，也没有把回归逻辑塞回主服务文件。

## 8. 本轮缺点与代价

这轮依然有几个清晰的代价：

- 回归脚本每次都会写入新的测试抓取源、关键词和文章
- “疑似重复跳过”当前只影响任务执行策略，没有扩展到后续人工复核界面
- 任务的“疑似重复开关”没有落到数据库独立字段，而是通过日志追踪当前策略

也就是说，这轮优先解决的是“能跑、能测、能追”，不是把这一块做成完整配置中心。

## 10. 本轮结论

本轮最重要的结论是：

- 这套 MVP 现在不仅可以手工测，而且已经具备了固定回归能力

被证实的判断：

- 固定脚本能明显提升后续迭代的信心和效率

以后应默认继承的经验：

- 主流程一旦能跑，就要尽快沉淀成脚本
- 所有会改变业务口径的任务开关，都要保证在日志里能看出来
