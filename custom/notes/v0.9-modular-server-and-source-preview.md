# 版本记录：v0.9-modular-server-and-source-preview

## 1. 本轮目标

本轮目标是承接 `v0.8-relational-sqlite-source-ai-settings`，继续推进 3 件事：

1. 拆分 `server.js` 的存储层、抓取层、AI 层和路由层
2. 继续清理残留乱码文案，并开始统一状态枚举
3. 给抓取源增加“最近一次提取预览”，让规则能先验再跑任务

本轮属于：

- 结构重构
- 功能补充
- 可维护性提升

## 2. 背景与上下文

`v0.8` 已经把关系型 SQLite、抓取失败日志和 AI 后台配置补到了代码层，但服务端仍有两个明显问题：

- `server.js` 仍然过大，存储、抓取、AI、路由耦合在一起
- 乱码文案和状态硬编码仍然比较多，后续继续迭代会越来越难控

本轮开始前按要求重新执行了 `git pull --ff-only`，结果依然失败，因为 `D:\ceshi\aibianji` 当前不是 Git 仓库。

## 3. 问题分析

本轮真正的核心矛盾不是“有没有功能”，而是“能不能继续迭代”。

主要问题有三层：

1. 单文件过重  
   `server.js` 既负责 SQLite、又负责正文提取、又负责 AI、又负责全部路由，修改任何一块都容易牵动全局。

2. 状态文案分散  
   文章状态、发布状态、去重状态和日志类型散落在多处，既不利于统一，也容易被乱码污染。

3. 规则调试入口不够靠前  
   抓取源虽然能保存规则，但管理员仍需要真正跑任务后才能看到提取效果，调试成本偏高。

## 4. 候选方案比较

### 方案 A：继续在现有 `server.js` 上堆功能

优点：

- 改动最少
- 短期开发快

缺点：

- 文件会继续膨胀
- 维护成本持续上升
- 后续再拆会更痛

结论：

- 放弃。本轮不再继续堆。

### 方案 B：一次性重构成完整工程结构

优点：

- 长期结构最干净

缺点：

- 对当前 demo 来说改动过重
- 风险高
- 不符合本仓库的最小改动原则

结论：

- 放弃。本轮只做最小模块拆分，不做完整工程化。

### 方案 C：按职责做最小四层拆分

核心思路：

- 保留现有 demo 目录
- 新建少量模块文件
- 让 `server.js` 只保留入口职责
- 旧功能原样迁移，不借拆分机会顺手改太多业务

优点：

- 结构明显变清楚
- 风险可控
- 方便下一轮继续拆细

缺点：

- 前端和部分旧文案仍未完全清理干净
- 仍然属于 demo 级结构，不是完整工程体系

结论：

- 本轮采用方案 C。

## 5. 最终决策

本轮最终采用的结构是：

- `src/storage/state-store.js`
- `src/services/crawl-service.js`
- `src/services/ai-service.js`
