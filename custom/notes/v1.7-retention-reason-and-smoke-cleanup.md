# 版本记录：v1.7-retention-reason-and-smoke-cleanup

## 1. 本轮目标

本轮继续承接 v1.6 的“疑似重复稿处理链路”收口，重点解决两个问题：

1. “保留疑似稿”动作虽然已经可用，但缺少明确理由，审核留痕不完整。
2. `smoke:e2e` 虽然能跑通主流程，但每次都会在本地 SQLite 里留下测试数据，不利于持续回归。

这轮目标属于：

- 功能补充
- 可维护性提升
- 测试闭环增强

## 2. 背景与上下文

本轮直接承接：

- `custom/notes/v1.6-manual-retain-and-task-restore.md`

v1.6 已经完成了两件关键事情：

1. 内容池支持“保留疑似稿”人工动作。
2. 任务的“保留疑似重复”策略在服务重启后可以恢复展示。

但 v1.6 留下了两个明显缺口：

1. 人工保留动作没有理由字段，后续只能看到状态变化，看不到判断依据。
2. `smoke:e2e` 只验证“能跑通”，没有验证“跑完后能回收测试数据”。

另外，本轮开始前先执行了 `git pull --ff-only`，结果为 `Already up to date.`，说明本地工作基线与 GitHub 远端一致。

## 3. 问题分析

### 3.1 表面现象

- 编辑或审核可以点“保留疑似稿”，但系统只知道“保留了”，不知道“为什么保留”。
- 回归脚本每跑一次，抓取源、关键词、任务、文章、失败日志都会继续累积。

### 3.2 核心矛盾

真正的核心矛盾不是“按钮少一个输入框”，而是这两条链路都缺了“可追溯”和“可重复”的最后一步：

- 疑似稿保留没有理由，就不算完整的审核留痕。
- 回归脚本没有清理模式，就不算完整的测试闭环。

### 3.3 主要原因

主要原因有三个：

1. v1.6 先优先打通了动作链路，故意没有同时引入额外字段输入，以避免前端交互一次改太多。
2. 当前 SQLite 仍是轻量单机结构，测试数据默认保留虽然方便排错，但会逐渐污染演示环境。
3. 抓取预览失败场景会生成失败记录，单纯删除文章和任务并不能彻底清干净。

## 4. 候选方案比较

### 方案 A：新增独立“保留理由”输入框和专门字段

核心思路：

- 在编辑/审核区额外新增一个专门的理由输入框。
- 后端单独保存一份“疑似稿保留理由”字段。

优点：

- 语义最清晰，理由字段和审核意见字段天然隔离。

缺点：

- 前端界面要再加一个输入区，当前演示界面会更拥挤。
- SQLite 和状态模型都要再补一个字段，属于小题大做。

放弃原因：

- 当前 MVP 阶段更重视“链路完整”而不是“字段拆得绝对精细”。

### 方案 B：复用现有审核意见输入区，保留时强制填写理由

核心思路：

- 复用当前已存在的审核意见输入框。
- 触发“保留疑似稿”时，要求该输入框非空，并把内容写回 `reviewComment` 与审核日志。

优点：

- 改动小，前端结构几乎不变。
- 保留动作立即具备理由留痕。
- 不需要额外改 SQLite 结构。

缺点：

- “审核意见”和“保留理由”共用同一个输入区，语义上不是最理想。

最终选择原因：

- 更符合当前“最小改动 + 先跑通测试闭环”的目标。

### 方案 C：不增加清理接口，只在 smoke 脚本里记录唯一前缀

核心思路：

- 继续让测试数据保留在本地。
- 仅通过唯一标记区分“哪批数据是 smoke 产生的”。

优点：

- 后端不需要增加清理接口。

缺点：

- 数据会持续堆积。
- 每次人工排查时仍然要手动分辨测试数据。

放弃原因：

- 这会让回归变成“越跑越脏”，不适合作为持续测试基线。

### 方案 D：增加仅本地测试使用的清理接口，并让 smoke 脚本可选调用

核心思路：

- 新增 `/api/testing/cleanup`。
- 由 smoke 脚本在 `SMOKE_CLEANUP=1` 时自动调用。
- 按本轮唯一标记批量清理抓取源、关键词、任务、文章、失败日志与相关日志。

优点：

- 主流程和清理流程都能自动验证。
- 不影响默认“保留数据便于排查”的工作方式。

缺点：

- 需要额外维护一个测试专用接口。
- 清理逻辑依赖唯一标记匹配，不是通用数据治理方案。

最终选择原因：

- 最符合“本地 MVP 可持续回归”的要求，且风险可控。

## 5. 最终决策

本轮最终采用两条策略：

1. “保留疑似稿”继续复用现有审核意见输入区，但变为保留动作的必填理由。
2. 增加本地测试清理接口，由 `smoke:e2e` 通过环境变量显式开启清理模式。

关键设计决策有三点：

1. 不新增数据库字段，先把理由写入现有 `reviewComment`，同时写入审核日志。
2. 清理接口只接受管理员调用，并通过唯一 `stamp` 做局部清理，避免误删普通数据。
3. 清理时不仅清理文章和任务，还额外清理抓取预览失败记录，补上之前最容易残留的一环。

## 7. 本轮优点

本轮相比 v1.6 的正向提升主要有：

1. 疑似稿保留从“只有动作”变成了“动作 + 理由 + 日志”完整留痕。
2. `smoke:e2e` 从“只会制造测试数据”变成了“可选自动回收测试数据”。
3. 清理范围覆盖到了预览失败残留，测试闭环更加完整。
4. README 回到可读中文状态，降低了后续接手成本。

从质量属性看，本轮主要提升了：

- 可维护性
- 调试便利性
- 实验可复现性

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. “保留理由”仍然复用 `reviewComment`，并不是一个独立字段。
2. 清理接口是本地测试导向设计，不是通用后台数据清理能力。
3. 清理逻辑基于唯一标记匹配，适合 smoke 回归，但不适合复杂生产数据治理。
4. `api-router.js` 当前仍然是较大的路由文件，未来继续堆功能会再次变重。

也就是说，本轮仍然属于“先把测试闭环补完整”，不是最终形态。

## 10. 本轮结论

本轮最重要的结论有三点：

1. “保留疑似稿”必须带理由，这样才算真正可追溯。
2. `smoke:e2e` 必须具备可选清理模式，才能长期作为稳定回归基线。
3. 抓取预览失败残留不能忽略，它正是最容易让演示环境越跑越脏的一类数据。

被证实的假设：

- 复用现有审核意见输入区，也能以很小代价完成理由留痕。
- 基于唯一标记的局部清理，足以支撑本地 MVP 的持续回归。
