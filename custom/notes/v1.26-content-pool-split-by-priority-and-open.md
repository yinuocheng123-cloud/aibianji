# 版本记录：v1.26-content-pool-split-by-priority-and-open

## 1. 本轮目标

本轮要解决的问题，是撤回上一轮“在抓取源区内拆分重点源 / 开放发现”的方案，改成更符合用户最新口径的结构：

- 抓取源只表示抓取对象
- 内容池再分为：
  - 重点源
  - 开放发现

之所以优先处理这个问题，是因为用户已经明确重新定义了两个概念：

1. 抓取源是抓取对象，比如网站、公众号、自媒体号
2. 重点源 / 开放发现的区分，应该发生在内容池的文章层，而不是抓取源层

本轮属于：

- 前台结构修正
- 概念边界澄清
- 已有方案回退与重定向

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.25-source-split-priority-and-discovery.md`
- `custom/notes/v1.24-keyword-stats-and-toggle.md`

修改前的状态是：

- 信息源区中的“抓取源”已被错误地拆成了两个并列子栏目：
  - 重点源
  - 开放发现
- 这个方案虽然试图强化来源分层，但它和用户刚重新明确的概念不一致

本轮的核心背景不是“功能没做出来”，而是上一轮对概念落点的理解偏了：

- 上一轮把“重点源 / 开放发现”理解成来源分层
- 用户本轮明确说明，这其实是文章来源方式分层，应该展示在内容池

也就是说，本轮不是普通增量迭代，而是对上一轮局部方案的纠偏。

## 3. 问题分析

表面现象是“上一个方案要撤回”，但真正的核心矛盾有三层：

1. 抓取源和内容来源方式不是同一层概念  
   抓取源描述的是“抓什么对象”，比如网站、公众号、自媒体号；而重点源 / 开放发现描述的是“文章怎么被抓到”。

2. 上一轮把来源对象和采集方式混在了一层  
   把抓取源区拆成“重点源 / 开放发现”，会让人误以为开放发现也是一种抓取源配置项，这和用户定义不一致。

3. 内容池才是最适合承载这层区分的地方  
   因为重点源文章和开放发现文章最终都统一进入内容池，且后续操作都是“直接发布”或“送入编审台”，所以真正需要并列展示的是文章列表，而不是源列表。

真正的瓶颈不是样式问题，而是信息分层位置错了。

## 4. 候选方案比较

### 方案 A：保留上一轮抓取源双列结构，只改文案

核心思路：

- 不撤结构
- 只把“重点源 / 开放发现”改成更中性的提示语

优点：

- 改动较小
- 不需要再动内容池结构

缺点：

- 概念层次仍然是错的
- 用户仍会把“开放发现”误解成一种抓取源对象
- 后续越改越乱

放弃原因：

- 这不是文案问题，而是信息架构问题

### 方案 B：抓取源回归单列表，内容池改成重点源 / 开放发现双分区

核心思路：

- 抓取源区恢复成“抓取对象列表”
- 内容池内部改成两个并列子栏目：
  - 重点源：来自抓取源对象的文章
  - 开放发现：通过关键词补充抓取到的全网公开文章

优点：

- 和用户重新定义的口径完全一致
- 不需要改数据库结构
- 不需要新增后端接口
- 内容池中的后续操作链路不变

缺点：

- 需要撤回上一轮刚做好的抓取源双列结构
- 内容池渲染逻辑需要重新分组

最终选择原因：

- 这是唯一真正符合概念边界的方案

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 抓取源恢复成单列表  
   只表示抓取对象，不再承担“重点源 / 开放发现”的分类职责。

2. 内容池承担重点源 / 开放发现分栏展示  
   因为这两个概念本质上是在区分文章获取方式，而不是区分抓取对象。

3. 重点源文章的判定继续复用现有逻辑  
   只要文章能匹配到 `sourceSites`，就视为重点源文章。

4. 开放发现文章继续复用现有文章级动作  
   在内容池里仍可以把开放发现文章“转为重点源”，保持沉淀闭环。

## 7. 本轮优点

当前方案相比上一轮的优点有：

- 概念边界更正确
- 抓取源回到“对象层”
- 内容池回到“文章层”
- 用户更容易理解重点源和开放发现的差异
- 不需要动后端存储结构
- 不需要新增后端接口

## 8. 本轮缺点与代价

这一轮的代价也很明确：

1. 上一轮刚做的抓取源双栏结构被撤回  
   这属于必要回退，但说明上一轮对概念边界判断有偏差。

2. 内容池现在承担更多结构表达职责  
   虽然更正确，但内容池区域会比上一轮更复杂一些，需要继续注意视觉收敛。

3. 开放发现当前仍然依赖文章的 `sourceId = 0` 口径  
   对 MVP 足够，但不是最终生产口径。

## 10. 本轮结论

本轮最重要的结论是：

“重点源 / 开放发现”必须归到内容池，而不是抓取源。

被证实的假设：

- 抓取源更适合作为抓取对象清单
- 内容池更适合作为文章获取方式的分栏展示位置

被否定的旧判断：

- 上一轮“在抓取源区里拆重点源 / 开放发现”的方案，不符合用户真实定义，应该撤回

以后默认应继承的经验：

- 一旦出现“对象层”和“结果层”两个概念，必须分开，不要混在同一区块里展示

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 继续压缩内容池里的辅助说明文案  
   让“重点源 / 开放发现”更多依赖结构本身表达。

2. 给内容池分栏补更轻的视觉区分  
   比如只通过标题、计数、淡边框来区分，避免重新变杂乱。

3. 如果你确认这一版口径没问题，再把本轮和前面未提交改动整理成一次提交  
   这样后续继续推真实整木网接口时更稳定。
