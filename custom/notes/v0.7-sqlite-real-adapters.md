# 版本记录：v0.7-sqlite-real-adapters

## 1. 本轮目标

本轮目标是承接 `v0.6-mvp-testable-local` 的下一步建议，继续把本地 MVP 从“可测”推进到“可接真实能力”：

- 用 SQLite 替换 JSON 作为主持久化后端
- 接入 DeepSeek 真实调用适配层
- 把抓取任务改成白名单真实抓取优先，失败时回退模拟

本轮目标类型：

- 功能补充
- 可维护性提升
- 实验探索

## 2. 背景与上下文

上一轮已经把系统提升到可本地做 MVP 测试的程度，但当时仍有三个清晰的后续方向：

1. 接真实 DeepSeek API
2. 把模拟抓取替换为白名单真实抓取最小链路
3. 把 JSON 持久化替换为正式数据库

用户本轮明确表示“按照建议处理”，因此这轮不再只停留在本地假数据闭环，而是开始把真实能力接入点补上。

## 3. 问题分析

`v0.6` 版本虽然已经能测，但还有三个明显短板：

1. 持久化仍然是 JSON 文件。
   - 适合演示，但不适合继续往真实系统靠拢。

2. AI 仍然只有本地占位结果。
   - 无法验证真实模型效果。

3. 抓取任务仍然只是模拟。
   - 无法验证真实白名单网页抓取的最小链路。

本轮最重要的判断是：

- 不能为了保持“始终可测”而永远停在纯模拟阶段
- 但也不能直接切到只依赖真实外部服务，否则一旦网络或配置缺失，整个本地 MVP 又会不可用

因此需要一种“真实优先 + 本地回退”的中间方案。

## 4. 候选方案比较

### 方案 A：一次性切到只依赖真实服务

优点：

- 更接近正式生产系统

缺点：

- 受网络、配置、外部接口稳定性影响太大
- 本地测试环境很容易直接失效

结论：

- 不采用。

### 方案 B：维持纯模拟，不接真实能力

优点：

- 稳定
- 本地测试简单

缺点：

- 永远无法验证真实能力接入路径

结论：

- 不采用。

### 方案 C：SQLite 正式化持久化 + AI/抓取“真实优先，失败回退”

优点：

- 本地依然能测
- 外部能力的真实接入路径也已经具备
- 后续切生产时改动成本更低

缺点：

- 代码复杂度会上升
- 需要处理迁移与回退逻辑

结论：

- 本轮采用方案 C。

## 5. 最终决策

本轮最终决策如下：

1. 使用 `sql.js` 作为本地 SQLite 后端。
2. 保留旧 `state.json` 作为迁移入口，但主持久化改为 `content-center.sqlite`。
3. AI 调用时：
   - 如果提供 `DEEPSEEK_API_KEY`，则调用真实 DeepSeek 接口
   - 如果缺少配置或调用失败，则回退到本地占位结果
4. 抓取任务执行时：
   - 先尝试抓取白名单入口页和同域链接
   - 如果无法抓到合适内容，则回退到模拟抓取

## 6. 具体实现

本轮改动文件：

- `custom/content-center-demo/server.js`
  - 引入 `sql.js`
  - 加入 SQLite 初始化与状态迁移
  - 把保存逻辑切到数据库导出文件
  - 新增 DeepSeek 调用函数
  - 新增真实网页抓取、文本清洗、链接提取和标题提取逻辑
  - 把抓取任务改为真实优先、失败回退
- `custom/content-center-demo/package.json`
  - 新增 `sql.js` 依赖
- `custom/content-center-demo/README.md`
  - 更新为 SQLite 与真实能力适配说明
- `custom/notes/v0.7-sqlite-real-adapters.md`
  - 记录本轮取舍、结构和验证

## 7. 本轮优点

- 本地持久化已经从 JSON 升级到 SQLite
- 真实 DeepSeek 接入路径已经具备
- 真实白名单抓取最小链路已经具备
- 即使真实能力失败，MVP 仍然可继续测试

## 8. 本轮缺点与代价

- 真实抓取能力目前仍是最小链路，正文提取规则还比较朴素
- DeepSeek 接入需要手动提供环境变量
- SQLite 目前仍承载的是集合快照型存储，不是完整关系型业务建模
- 当前目录仍然不是 Git 仓库，所以依旧无法执行真正的 `git pull`

## 9. 验证与结果

本轮验证方式：

1. `node --check` 检查服务端与前端语法
2. 启动临时端口服务
3. 登录管理员账号
4. 执行一次抓取任务
5. 调用一次 AI 摘要接口
6. 检查 SQLite 文件是否生成

本轮结果：

- 脚本语法通过
- 登录成功
- 抓取任务成功执行
- AI 接口成功返回结果（当前环境无真实 Key 时会回退占位）
- SQLite 文件已生成

## 10. 本轮结论

本轮最重要的结论是：

- 对当前阶段来说，最佳策略不是“纯模拟”或“纯真实”，而是把系统做成“真实优先、失败回退”的混合形态

下一轮最值得做的事：

1. 把 SQLite 的集合快照进一步拆成关系型表结构
2. 补充真实抓取的正文抽取规则配置
