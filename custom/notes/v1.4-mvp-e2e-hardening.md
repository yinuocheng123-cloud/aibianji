# 版本记录：v1.4-mvp-e2e-hardening

## 1. 本轮目标

本轮目标是把“整木网编辑中台”继续收口到可直接测试的程度，重点解决 3 件事：

- 修复保存草稿接口的稳定性问题
- 把前端状态展示切到后端返回的统一状态枚举
- 把抓取源提取预览失败时的兜底展示补完整，并跑通一条可复现的端到端链路

为什么这轮值得优先处理：

- 当前页面和基本流程已经有了，但“保存草稿 500”会直接阻断主链路
- 前端仍保留本地状态常量，和后端状态源有分叉风险
- 抓取预览和抓取入池是用户验证 MVP 的第一入口，必须在失败场景下也能给出可解释反馈

本轮属于：

- Bug 修复
- 可维护性提升
- 可测试性提升

## 3. 问题分析

本轮实际遇到的核心问题有 3 个：

1. 保存草稿接口偶发 `500`

表面现象：

- 在“AI 摘要 -> 保存草稿”的联调里，后端返回 `500`

核心原因：

- 草稿保存接口没有对入参做强规范化
- 前端通常会传数组 `tags`，但旧前端缓存或脚本联调可能传字符串
- 存储层在落库时按数组遍历 `article.tags`，一旦拿到字符串就会在持久化阶段报错

2. 前端状态枚举仍是本地副本

表面现象：

- 前端自己维护了一套 `ARTICLE_STATUS / PUBLISH_STATUS / DUPLICATE_STATUS`

核心矛盾：

- 后端已经在 bootstrap 返回 `statusEnums`
- 如果两边继续各自维护，后续一旦改文案或状态值，就会出现筛选、按钮判断和展示不一致

3. 模拟抓取的文案模板过于接近

表面现象：

- 新建测试关键词后，抓取任务有时会因为高相似度被判为重复，导致“新线索入池”不稳定

核心原因：

- 模拟抓取文案虽然带了不同关键词，但大部分骨架文本相同
- 当前去重是标题 + 正文相似度双算，模板过于统一会放大误判

## 4. 候选方案比较

### 方案 A：只修保存接口，不碰前端与模拟抓取

优点：

- 改动最小
- 可以快速消掉 `500`

缺点：

- 前后端状态源仍然分叉
- 抓取任务新增文章仍可能不稳定
- 用户拿到的并不是“可稳定测试”的版本

结论：

- 放弃。只修一个点不足以解决本轮目标。

### 方案 B：后端入参规范化 + 前端状态统一 + 模拟抓取去重收口

核心思路：

- 在 API 路由层把草稿字段统一规范化
- 在前端改为消费 bootstrap 返回的 `statusEnums`
- 提取预览失败时补全原始响应摘要展示
- 增强模拟抓取文案的区分度，降低测试场景下的误判概率

优点：

- 能同时解决运行稳定性、状态一致性和联调可复现性
- 改动仍保持在现有架构内，不需要推翻模块拆分

缺点：

- 会同时动到前端、路由层和抓取层
- 模拟抓取仍然只是测试用，不代表正式生产抓取质量

结论：

- 本轮采用方案 B。

## 5. 最终决策

最终采用的策略是：

- 后端把保存草稿入口改成“强规范化写入”
- 前端把状态筛选和状态样式判断切到后端枚举
- 提取预览失败时在前端显式展示“失败兜底 + 原始响应摘要”
- 模拟抓取文本加入更强的来源、编号和关键词特征，让新增测试文章更容易稳定入池

这个方案延续了 `v0.9` 之后“分层增量演进”的思路，没有回到单文件重写，也没有为了联调而绕过现有去重和状态流。

## 7. 本轮优点

相比之前，这轮的直接收益有：

- 草稿保存不再依赖前端“必须传对格式”
- 前后端状态源开始统一，后续改状态文案更安全
- 预览失败不再只有一句报错，而是能保留可追溯的原始响应摘要
- 模拟抓取更适合做联调，不会因为模板太像而频繁把新线索挡在内容池外
- 服务端出错时终于会有明确日志，便于继续迭代

质量属性上的改善：

- 稳定性提升
- 可测试性提升
- 可维护性提升
- 调试便利性提升

## 8. 本轮缺点与代价

这轮仍然保留了几个明显妥协：

- 模拟抓取虽然更适合测试，但依然不代表正式采集质量
- 当前新增测试文章仍可能被判成“疑似重复”，只是不会再被直接挡掉
- 前端已经切到后端状态枚举，但还没有把所有业务常量完全抽成单独共享文件
- 这轮没有继续做真实主站 API，也没有接入正式数据库 schema 演进

换句话说，这轮是“把 MVP 跑稳”，不是“把系统做完”。

## 10. 本轮结论

本轮最重要的结论是：

- 当前这套“整木网编辑中台”已经可以用来做真实的 MVP 测试，不再只是看界面

被证实的判断：

- 保存接口的问题，核心就是入参未规范化
- 前端切到后端状态枚举是值得做的，不会增加复杂度，反而让联调更稳
- 模拟抓取文本必须为测试留出足够差异度，否则三层去重会误伤测试链路

以后应默认继承的经验：

- 涉及落库的字段，后端必须自己做最终规范化，不能把正确格式完全交给前端保证
- 对“失败预览”这类场景，要保留可解释的上下文，而不是只给一句失败

## 11. 下一轮建议

- 风险最低
- 能继续减少前后端文案漂移

3. 给主流程补一个固定的本地联调脚本

- 能把这轮手工验证沉淀成下一轮的回归基线
