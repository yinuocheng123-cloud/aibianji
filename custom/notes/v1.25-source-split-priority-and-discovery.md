# 版本记录：v1.25-source-split-priority-and-discovery

## 1. 本轮目标

本轮要解决的问题，是把“信息源 -> 抓取源”进一步收口成更清楚的结构，让：

- 重点源
- 开放发现

不再混在同一个列表里，而是分成两个并列子栏目展示。

之所以优先处理这个问题，是因为用户已经明确指出上一版虽然比更早版本清爽，但来源层级还不够一眼看懂。当前最需要继续优化的不是多加功能，而是让“重点源优先、开放发现补充”的业务关系在信息源区直接可见。

本轮属于：

- 前台结构收口
- 信息层级澄清
- 可测试性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.24-keyword-stats-and-toggle.md`
- `custom/notes/v1.23-keyword-driven-open-discovery-flow.md`

修改前的状态是：

- 信息源区已经拆成“关键词 + 抓取源”
- 关键词已经收成轻量列表
- 抓取源仍只有一个单列列表
- 开放发现虽然已经在内容池里可见，也能“转为重点源”，但它还没有在信息源区形成并列视图

也就是说，上一轮已经解决了“关键词列表太重”的问题，但这一轮还要继续解决“来源层级关系不够直观”的问题。

## 3. 问题分析

表面现象是抓取源列表只有一列，但真正的核心矛盾有三层：

1. 重点源和开放发现属于不同角色  
   重点源是人工维护的优先范围，开放发现是系统补充发现的新来源。它们的业务意义不同，放在同一列里会混淆。

2. 开放发现虽然存在，但沉淀路径不够直观  
   当前用户要先在内容池看到开放发现文章，再理解“这个来源可以转为重点源”。这个链条太绕。

3. 信息源区还没有体现来源资产沉淀逻辑  
   系统定位里已经强调“开放发现的新优质来源，可转化为重点抓取源持续跟踪”，但页面结构还没有把这条路径明确呈现出来。

真正的瓶颈不是功能缺失，而是信息架构还没有把“来源沉淀闭环”表达清楚。

## 4. 候选方案比较

### 方案 A：继续保留单列抓取源列表，只在文案里说明“包含开放发现”

核心思路：

- 不改列表结构
- 只在说明文案里补充“开放发现”

优点：

- 改动最小
- 风险最低

缺点：

- 仍然需要用户自己理解来源分层
- 开放发现不会形成明确的并列视图
- 页面结构与业务口径仍然不完全一致

放弃原因：

- 只能“解释”，不能真正“看见”

### 方案 B：在抓取源区内改成两个并列子栏目

核心思路：

- 保留当前“抓取源”面板不变
- 面板内部把来源列表改为两列：
  - 左边：重点源
  - 右边：开放发现
- 重点源继续支持编辑、删除
- 开放发现继续复用现有“转为重点源”动作

优点：

- 和业务口径最一致
- 不需要新增后端接口
- 不需要新建独立模块
- 用户在信息源区就能直接理解“优先采集 + 补充发现 + 回流沉淀”

缺点：

- `source-management` 作为右侧面板，本身宽度有限，双列后会更紧凑
- 开放发现来源当前仍是基于内容池文章聚合而来，不是独立持久化表

最终选择原因：

- 在现有 MVP 结构里，这是最小改动、最高表达力的一种做法

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. “抓取源”面板内部改成两个并列子栏目  
   继续沿用“信息源区只放来源相关内容”的方向，不把开放发现再塞进内容池主列表里承担说明责任。

2. 重点源继续沿用现有 `sourceSites`  
   不新增新的持久化结构，避免扩散到后端存储层。

3. 开放发现来源通过当前内容池文章聚合得到  
   只要文章尚未匹配到重点源，就把它作为开放发现来源候选。

4. “转为重点源”继续复用现有文章级动作  
   不新开接口，仍由现有 `promoteArticleSource` 完成来源沉淀。

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

本轮主要补了两部分逻辑：

1. 来源聚合辅助函数
   - `inferSourceTypeFromUrl`
   - `getOpenDiscoverySources`

作用：

- 根据文章原始链接推断来源类型
- 把当前内容池中未匹配重点源的文章聚合成“开放发现来源”
- 同源多篇文章合并成一个来源项，并统计线索数、保留最近时间和摘要

2. 抓取源渲染逻辑重写
   - 更新 `renderSources`

作用：

- 不再直接渲染单列 `sourceList`
- 改成：
  - 重点源列表
  - 开放发现列表
- 重点源项支持编辑、删除
- 开放发现项支持“转为重点源”

### 6.2 `custom/content-center-demo/public/styles.css`

新增了并列子栏目的样式：

- `source-split-grid`
- `source-subpanel`
- `subpanel-head`

作用：

- 让“重点源 / 开放发现”在抓取源区内部形成两个并列子栏目
- 在窄屏下自动退回单列，避免移动端过挤

### 6.3 `custom/content-center-demo/public/index.html`

本轮只做了两项必要调整：

- 抓取源面板角标改成：
  - `重点源 / 开放发现`
- 静态资源版本升级到：
  - `v28-source-split`

作用：

- 页面口径与结构一致
- 防止浏览器继续使用旧缓存

## 7. 本轮优点

当前方案相比之前的优点有：

- 信息源区的来源层级更清楚
- 开放发现不再只是内容池里的“隐含概念”
- 来源沉淀路径更直观：发现 -> 观察 -> 转为重点源
- 没有新增后端接口
- 没有改动数据库结构
- 继续保持前台结构简洁

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. 开放发现来源仍是前端聚合结果  
   目前不是独立持久化数据，只适合 MVP 阶段。

2. 右侧抓取源面板内部再拆双列后，桌面端会更紧凑  
   当前通过简化字段和移动端回退单列来控制，但后续如果再往里加信息，会重新变拥挤。

3. 仍然依赖文章级“转为重点源”动作  
   当前是够用的，但后续若要批量沉淀来源，可能还要补专门的来源级动作。

## 10. 本轮结论

本轮最重要的结论是：

“重点源”和“开放发现”不适合继续混在一个单列来源列表里，它们需要在信息源区被明确并列出来，用户才能更自然地理解这套采集模式。

被证实的假设：

- 这两个来源角色分栏后，更符合用户的业务理解
- 不改后端接口，也能把来源沉淀路径表达清楚

被修正的旧判断：

- 之前默认“抓取源一个列表就够了”，现在证明这会把来源角色混在一起

以后应默认继承的经验：

- 只要业务里存在“人工维护范围”和“系统补充发现”这两类来源，就不要再混成一个来源列表

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 把信息源区剩余细碎文案再压一轮  
   进一步减少解释性文字，让结构自己说话。

2. 对开放发现来源增加“最近来源文章数”或“最近一次进入内容池时间”的更轻量展示  
   收益高，但要控制不把卡片重新做重。

3. 如果你认可这版结构，再把本轮和前几轮未提交改动统一整理成一次提交  
   风险最低，也方便继续往真实整木网接口过渡。
