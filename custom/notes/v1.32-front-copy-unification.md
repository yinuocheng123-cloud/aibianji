# 版本记录：v1.32-front-copy-unification

## 1. 本轮目标

本轮继续承接前台收口工作，但不扩真实整木网接口，只解决一个很基础、却会持续影响后续迭代的问题：

1. 把内容池、编审台、发布榜相关的空状态、说明文案和按钮口径统一起来。
2. 避免静态 `index.html` 和运行时 JS 渲染各说各话。

这轮属于：

- 可维护性提升
- 前台文案收口
- 体验一致性修复

之所以优先处理，是因为上一轮已经把脚本拆层完成，如果文案来源继续分散，后面每次改一处前台提示都要同时检查 HTML 和多个脚本文件，维护成本会继续上升。

## 2. 背景与上下文

本轮直接承接：

- [v1.31-front-modularization-and-publish-board-lightening.md](D:\ceshi\aibianji\custom\notes\v1.31-front-modularization-and-publish-board-lightening.md)
- [v1.30-content-list-tightening-and-legacy-removal.md](D:\ceshi\aibianji\custom\notes\v1.30-content-list-tightening-and-legacy-removal.md)

当前模块在修改前的状态是：

- 前台脚本已经拆成状态层、会话层、工具层、渲染层、动作层和入口层。
- 发布榜已经改成按日期分组的轻清单。
- 内容池和编审台的结构已经基本稳定。

但修改前仍然存在这些问题：

- `index.html` 里的初始文案和运行时 `configureFrontstageLayout()` 覆写后的文案不完全一致。
- `renderArticles()`、`renderReviewQueue()`、`renderPublishBoard()` 等函数各自维护空状态文本。
- 批量发布、送入编审台、保存修订稿等按钮文案虽然已接近统一，但仍然分散在静态 HTML 与 JS 中。

## 3. 问题分析

表面现象是“有些文字不一致”，但核心矛盾其实是：

- 前台结构已经进入可持续迭代阶段，文案如果继续散落在多个文件和多个函数里，后面每次小调整都容易漏改。

主要问题分为两类：

### 3.1 静态与运行时文案分离

打开页面时，用户先看到的是 `index.html` 里的静态文本；登录后再由 JS 改写部分区域。

如果两者不一致，就会出现：

- 未登录页面看到一套口径
- 登录后又变成另一套口径

这种问题不影响功能，但会明显影响产品感。

### 3.2 空状态和动作文案缺少统一来源

内容池、编审台、发布榜都有空状态。
这些内容在业务上其实属于同一个前台场景，但之前是：

- 一部分写在 `index.html`
- 一部分写在 `app-render.js`
- 一部分写在 `app-utils.js`

这会让后续修改时很难确认“哪一句才是当前标准口径”。

## 4. 候选方案比较

### 方案 A：只改静态 HTML，不改 JS

核心思路：

- 把 `index.html` 里的文字改成和当前运行态更接近。

优点：

- 改动最小。
- 很快能看到页面初始效果变好。

缺点：

- 运行时渲染文案仍然分散。
- 后续继续改文案时还是会出现双维护。

放弃原因：

- 只能治表，不治根。

### 方案 B：只改 JS，不管静态 HTML

核心思路：

- 保证登录后的最终页面口径统一。

优点：

- 对真正使用中的页面更有效。

缺点：

- 未登录页和脚本未完成初始化时仍会出现旧文案。
- 页面首次可见内容仍然不统一。

放弃原因：

- 这会让“初始态”和“最终态”继续分裂。

### 方案 C：抽出前台文案常量，并同步更新静态 HTML

核心思路：

- 在状态层集中定义前台文案常量。
- 运行时渲染和布局收口都优先读这套常量。
- `index.html` 同步改成与当前标准口径一致。

优点：

- 静态与运行时口径一起收住。
- 后续继续改前台提示时，有明确入口。
- 对现有技术路线影响很小。

缺点：

- 不能一次性彻底解决所有文字分散问题。
- 部分动态拼接文案仍然保留在工具层函数里。

最终选择：

- 采用方案 C。

## 5. 最终决策

本轮最终决策如下：

1. 在前台状态层新增统一的 `FRONTSTAGE_COPY` 文案常量。
2. 把最容易持续漂移的几类文案先收进去：
   - 区块说明
   - 空状态
   - 选择摘要
   - 核心动作按钮
   - 基础标签口径
3. 同步修正 `index.html` 里的静态文案和资源版本号。

这轮没有把所有动态文案都彻底抽成配置项，主要是为了保持最小改动和低风险。

## 7. 本轮优点

本轮带来的主要正向效果有：

- 页面初始态和登录后最终态的文案更加一致。
- 后续继续改前台体验时，不需要再在多个文件里反复找相似句子。
- 内容池、编审台、发布榜这三块的口径更统一，前台更像一个完整工作台，而不是各块拼起来的演示页。

改善明显的质量属性包括：

- 可维护性
- 一致性
- 前台产品感

## 8. 本轮缺点与代价

这轮仍然存在明确的妥协和限制：

- 文案常量还没有覆盖所有动态文本。
- `app-utils.js` 和 `app-render.js` 里仍保留了一部分直接拼接文案。
- 这轮主要收的是“面向用户的主口径”，没有去动后台隐藏区域文案。

也就是说，这轮是“先把最容易看见、最容易漂移的文案收住”，不是彻底完成前台 copy system。

## 10. 本轮结论

本轮最重要的结论是：

- 当前这套前台继续收口时，文案也必须像代码一样开始有“单一可信来源”，否则会很快重新变乱。

本轮证实了两个判断：

1. 只要前台已经进入持续迭代阶段，静态 HTML 和运行时渲染就不该继续维护两套口径。
2. 文案统一这件事虽然不改功能，但会直接提高后续迭代效率。

本轮修正了一个旧判断：

- 之前更关注结构和脚本边界，默认“文案后面再统一”问题不大；现在看，这件事已经值得前置处理。
