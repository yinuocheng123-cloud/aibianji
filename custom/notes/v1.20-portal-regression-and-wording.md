# 版本记录：v1.20-portal-regression-and-wording

## 1. 本轮目标

本轮目标是在上一轮“整木网资讯栏目模拟化”基础上，继续把 MVP 收口到更适合测试的程度，重点解决两件事：

1. 把 3 个模拟资讯栏目都纳入固定回归，不再只验证其中一个栏目
2. 把前台和文档里剩余的“发布”口径继续统一成“转发到整木网”

这轮值得优先处理，因为上一轮虽然已经把模拟栏目做通了，但验证只覆盖了单个栏目，仍然不够稳；同时前台还残留少量“发布”字样，和当前产品口径不完全一致。

本轮属于：

- 可维护性提升
- 测试收口
- 交互文案统一

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.19-simulated-portal-columns.md`
- `custom/notes/v1.18-front-workflow-and-backstage-hide.md`

上一轮已经完成的前提是：

- 默认栏目已切到行业资讯、企业资讯、市场动态
- 转发回写已按模拟栏目生成主站 ID 与 URL
- 3210 服务已切到新版本

但还存在两个问题：

- 只验证了企业资讯这一条模拟栏目路径
- 前台个别字段和说明还留着“发布”表述，容易和“转发到整木网”这条主链路混淆

## 3. 问题分析

表面上看，系统已经能用了，但从可测试性角度还不够完整：

1. 只验证一个栏目时，另外两个栏目路径仍然可能隐藏问题
2. 文案残留会让测试人员误以为系统已经在走真实发布链路，而不是当前的整木网模拟转发链路
3. 如果前端资源版本号不升级，浏览器仍可能继续执行旧缓存脚本，导致用户看到和代码不一致的结果

核心矛盾是：

- 代码功能已经逐步收口
- 但自动回归覆盖面和前台语义一致性还没完全补齐

## 4. 候选方案比较

### 方案 A：继续只保留单栏目 smoke 验证

核心思路：

- 保持现有 `smoke:e2e`
- 只用默认栏目继续联调

优点：

- 改动最小

缺点：

- 无法证明 3 个栏目都能走通
- 后续每次改动仍需要人工切换栏目验证

放弃原因：

- 当前阶段最需要的是建立更强的回归基线，而不是继续依赖人工点测

### 方案 B：把 3 个模拟栏目都纳入统一回归脚本，并同步清理前台口径

核心思路：

- 扩展现有 `smoke-e2e.js`
- 支持按指定栏目运行
- 也支持一次性遍历 3 个模拟栏目
- 同时继续清理前台剩余“发布”表述

优点：

- 不新增额外复杂接口
- 复用已有回归脚本
- 3 个栏目都能固定回归
- 前台口径和当前产品阶段更一致

缺点：

- 脚本逻辑会比上一轮复杂一点
- 需要再补一轮文档与缓存版本管理

最终选择原因：

- 这是最符合“继续修订，并在结束后统一提交推送”的增量方案

## 5. 最终决策

本轮最终采用方案 B：

1. 扩展 `smoke-e2e.js`，支持单栏目和全栏目两种模式
2. 在 `package.json` 中新增 `smoke:portal-columns`
3. 把前台字段和占位文案继续统一成“转发到整木网”
4. 升级前端资源版本号，避免浏览器缓存干扰验证

## 6. 具体实现

### 6.1 `custom/content-center-demo/scripts/smoke-e2e.js`

本轮把回归脚本从“单场景”升级成“可选多场景”。

新增内容：

- `TEST_ALL_PORTAL_CATEGORIES`
- `TARGET_CATEGORY_ID`
- `SIMULATED_PORTAL_CATEGORY_TARGETS`

结构调整：

- 把原来单条主流程拆成 `runScenario(targetCategoryId)`
- `runSmoke()` 负责决定执行一个栏目还是全部栏目

新增验证点：

- 保存草稿后 `recommendedCategoryId` 是否正确
- `portalArticleId` 是否按栏目 ID 生成
- `portalUrl` 是否按栏目路径生成
- 发布日志是否明确写出栏目名称

效果：

- 现在可以一次性验证行业资讯、企业资讯、市场动态三条转发路径

### 6.2 `custom/content-center-demo/package.json`

新增脚本：

- `smoke:portal-columns`

作用：

- 一条命令直接跑 3 个模拟栏目回归

### 6.3 `custom/content-center-demo/public/index.html`

继续收口前台文案：

- `发布时间` -> `原文发布时间`
- `发布标题` -> `转发标题`
- `发布备注` -> `转发备注`
- `通过并发布` -> `审核后转发到整木网资讯栏目`

同时把静态资源版本号升级为：

- `v23-portal-regression`

作用：

- 降低语义混淆
- 强制浏览器拉取最新脚本

### 6.4 `custom/content-center-demo/public/app.js`

补充前台细节收口：

- 详情区时间文案改为：
  - `原文发布时间 / 抓取时间`

这样可以避免用户误把这里理解成“已经发布到整木网”的时间。

### 6.5 `custom/content-center-demo/README.md`

补充说明：

- 当前系统支持整木网资讯栏目模拟转发
- 新增 `npm run smoke:portal-columns`
- 当前发布说明统一改成“转发”

## 7. 本轮优点

这轮相比上一轮的主要提升是：

- 3 个模拟栏目都纳入了固定回归，不再只靠人工验证
- 前台和文档中的“发布”语义进一步收口
- 浏览器缓存风险通过版本号升级再次压低
- 后续接真实整木网接口时，栏目维度的联调基线更完整

## 8. 本轮缺点与代价

本轮仍然有这些代价：

- 仍然是模拟栏目，不是真实整木网接口
- 回归脚本更强了，但依旧会向本地数据层写测试数据，虽然现在已有 cleanup 模式
- 状态枚举内部仍然保留“已发布 / 待发布 / 未发布”作为内部状态，不等同于真实主站发布状态

本轮完成的验证如下：

4. 关键结果
   - 行业资讯：通过
   - 企业资讯：通过
   - 市场动态：通过
   - 三个栏目都成功生成对应的：
     - `portalArticleId`
     - `portalUrl`
     - 发布日志

5. 3210 运行态检查
   - 确认服务可访问
   - 接下来会重启到本轮最新版本继续验证

## 10. 本轮结论

本轮最重要的结论是：

- 当前整木网模拟资讯栏目不仅“能用”，而且已经具备固定回归能力

被证实的判断：

- 三个模拟栏目路径都可以用同一套链路跑通
- 前台文案继续从“发布”收口到“转发”，会让用户对当前阶段边界更清楚
- 静态资源版本号管理仍然是本地 MVP 中必须保留的动作

需要默认继承的经验：

- 每次改前台登录或工作流相关逻辑，都应同步考虑静态资源版本号
- 每次改整木网模拟栏目相关逻辑，都应至少执行一次 `smoke:portal-columns`

## 11. 下一轮建议

下一轮最值得继续做的 3 件事：

1. 基于真实整木网接口资料，替换当前模拟发布适配层
2. 把 AI 配置和失败日志真正拆到独立后台页面
3. 继续清理历史 note 中的乱码文本显示问题

当前收益最高的是第 1 件事，但它依赖真实接口资料准备好之后再推进。
