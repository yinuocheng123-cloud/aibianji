# 版本记录：v1.16-login-fetch-and-favicon

## 1. 本轮目标

本轮目标是继续修复“点击登录进入中台后仍然留在登录页”的问题，并把浏览器控制台里无关的 `favicon.ico 404` 噪音去掉。

这个问题值得优先处理，因为上一轮虽然已经修正了服务端的 token 优先级，但用户仍然反馈真实浏览器里“点登录没进去”。这说明问题可能不再是认证结果本身，而是登录按钮触发后的浏览器交互方式。

本轮属于：

- Bug 修复
- 交互链路加固
- 调试噪音清理

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.15-login-token-priority-and-client-events.md`
- `custom/notes/v1.14-login-cookie-sync.md`
- `custom/notes/experiments/exp-20260404-login-runtime-check.md`

上一轮已经确认：

- `POST /api/client-events` 已不再 404
- 服务端在 cookie 与 header token 冲突时，已经会优先选择有效 token
- 即使故意带入错误旧 cookie，只要新 token 有效，`/api/bootstrap` 也能返回 `200`

但用户继续反馈“点登录还是进不去”，同时控制台只剩 `favicon.ico 404`。

这说明：

- 真正阻塞用户的已不太像是后端会话判断
- 更像是浏览器原生表单提交、302 跳转、页面再次初始化之间的链路仍然不够稳

## 3. 问题分析

表面现象：

1. 控制台里只看到 `favicon.ico 404`
2. 用户点击登录后依然觉得没有进入中台

核心矛盾：

- 当前登录按钮依赖浏览器原生表单提交
- 原生表单提交会触发：
  - `POST /api/auth/login`
  - 服务端 `302`
  - 浏览器导航到 `/?session_token=...`
  - 页面重新初始化

这条链路理论上可行，但在真实浏览器里会多出若干不稳定因素：

- 表单重定向本身是否被浏览器或扩展干扰
- 导航过程中旧页面状态是否残留
- 跳转后脚本重新加载的时机是否被缓存影响

而当前服务端已经支持更稳的一条路径：

- 直接用 JSON 调用 `/api/auth/login`
- 成功后立刻返回 `bootstrap payload`
- 前端无需刷新页面，就能直接进入中台

这意味着：

- 继续依赖浏览器原生表单跳转已经没有必要
- 更稳的做法是：前端原地登录、原地更新页面状态

## 4. 候选方案比较

### 方案 A：继续保留原生表单提交，只做更多后端修补

核心思路：

- 登录继续依赖表单 `POST + 302`
- 只从后端和 cookie 处理角度继续加固

优点：

- 保持现有交互不变
- 与传统无 JS 表单更兼容

缺点：

- 仍然依赖浏览器跳转链路
- 问题一旦出现在重定向或页面刷新阶段，定位和修复都更慢

未选原因：

- 现有后端已经足够支撑原地登录，没有必要继续把登录过程绑死在 302 跳转上

### 方案 B：前端接管登录动作，原地 fetch 登录并直接刷新界面状态

核心思路：

- 阻止表单默认提交
- 改为前端 `fetch('/api/auth/login')`
- 成功后：
  - 保存 `sessionToken`
  - 直接 `hideLogin()`
  - 直接 `refreshData(result)`

优点：

- 绕开浏览器原生重定向
- 登录结果立即体现在当前页
- 失败时也能在当前页清晰展示错误信息
- 更符合本地 MVP 演示系统的需求

缺点：

- 不再依赖“纯 HTML 表单即可登录”的模式
- 登录行为更依赖前端脚本正常执行

最终选择原因：

- 当前问题已经明显落在“浏览器交互链路”而非“接口本身”
- 原地登录是最直接、最稳的修复方向

## 5. 最终决策

本轮最终采用方案 B，并顺手清理控制台噪音：

1. 登录表单改为前端原地 fetch 登录
2. 登录成功后直接刷新中台状态，不再依赖跳转
3. 登录失败时在原页面显示错误
4. 在页面头部增加空 favicon，避免无意义的 `favicon.ico 404`

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

把 `handleLoginSubmit()` 改成异步登录流程：

- `event.preventDefault()`
- 读取用户名与密码
- 发起 JSON 登录请求
- 成功后：
  - `setSessionToken(result.sessionToken)`
  - `hideLogin()`
  - `refreshData(result)`
  - 记录 `login_success` 客户端事件
- 失败后：
  - 在登录面板显示错误信息
  - 记录 `login_error` 客户端事件

同时增加了一个小的交互优化：

- 登录请求过程中，按钮文字改成“登录中...”
- 请求结束后再恢复“登录进入中台”

### 6.2 `custom/content-center-demo/public/index.html`

做了两类修改：

1. 资源版本号提升到：
   - `v19-login-fetch`

2. 新增：
   - `<link rel="icon" href="data:," />`

作用分别是：

- 强制浏览器拉取最新脚本
- 去掉无意义的 favicon 控制台报错

## 7. 本轮优点

当前方案的优点：

- 登录链路不再依赖浏览器原生跳转
- 用户点击后会直接在当前页完成登录
- 若登录失败，错误也能原地显示，不再靠 URL 参数与跳转回写
- 控制台更干净，后续更容易判断真正异常

## 8. 本轮缺点与代价

这轮仍有这些代价：

- 登录越来越依赖前端脚本，和“纯表单可用”方向进一步拉开
- 当前只是让 MVP 登录更稳，不是正式生产登录方案
- 仓库里历史记录文件在终端中的编码显示问题仍未系统清理

## 10. 本轮结论

本轮最重要的结论是：

- 当前最值得修的不是后端认证结果，而是浏览器端“点击登录后的跳转链路”
- 把登录切换为前端原地 fetch，是比继续围着 302 跳转打补丁更稳的方案

应默认继承的经验：

- 对本地演示系统来说，能原地完成登录和状态刷新，比追求传统表单重定向更重要
- 控制台里的 favicon 报错应尽量清掉，避免干扰用户判断
