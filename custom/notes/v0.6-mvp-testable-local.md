# 版本记录：v0.6-mvp-testable-local

## 1. 本轮目标

本轮目标是把原来的“可演示 demo”升级为“可本地做 MVP 测试”的版本。

本轮目标类型：

- 功能补充
- 可维护性提升
- 实验探索

## 2. 背景与上下文

本轮承接 `v0.5-apple-minimal-dashboard`。

上一轮已经把页面视觉收敛到更稳定的极简风，但功能层仍主要停留在演示数据展示阶段。用户本轮明确要求“接着补代码，一次性做到可以实现 MVP 测试的程度”，因此重点必须从视觉转回业务链路。

本轮开始前，我按仓库要求尝试执行 `git pull --ff-only`，但当前 `D:\ceshi\aibianji` 仍然不是 Git 仓库，没有 `.git`，因此这项要求客观上无法落地。

## 3. 问题分析

现有版本距离“可测试 MVP”主要差以下几项：

1. 没有数据持久化。
   - 刷新页面或重启服务后，所有操作都会丢失。

2. 没有真正的登录与角色区分。
   - 无法测试管理员、编辑、审核员之间的操作边界。

3. 没有真正可执行的抓取任务入口。
   - 只能看预置内容，无法测试“任务 -> 去重 -> 入库”链路。

4. 内容池缺少筛选和更完整的状态流转。
   - 无法模拟真实编辑流程。

因此本轮最关键的不是继续补视觉，而是补“能操作、能保存、能验证”的本地业务闭环。

## 4. 候选方案比较

### 方案 A：直接接数据库和真实抓取

优点：

- 更接近正式系统

缺点：

- 当前空仓库和低依赖结构下，成本太高
- 会迅速把范围拉到真实工程化问题，不利于尽快进入测试

结论：

- 本轮不采用。

### 方案 B：继续只用内存数据，但把交互做全

优点：

- 实现快

缺点：

- 仍然不适合真正做 MVP 测试
- 每次刷新都丢数据，测试价值有限

结论：

- 本轮不采用。

### 方案 C：本地 JSON 持久化 + 登录角色 + 模拟抓取任务 + 完整状态流

优点：

- 成本可控
- 本地即可测试
- 后续仍可替换为正式数据库和真实接口

缺点：

- 抓取和 AI 仍是模拟能力
- 鉴权仍只是本地测试级别

结论：

- 本轮采用方案 C。

## 5. 最终决策

本轮最终决策如下：

1. 服务端加入本地 JSON 持久化，默认数据自动初始化到 `custom/content-center-demo/data/state.json`。
2. 加入基础登录体系，提供管理员、编辑、审核员、查看角色四种本地测试账号。
3. 加入抓取任务模块，可从抓取源和关键词生成模拟线索，并走 URL / 标题 / 正文相似度去重。
4. 前端升级为真正可操作的工作台：
   - 可登录
   - 可新增/编辑抓取源
   - 可新增/编辑关键词
   - 可执行模拟抓取
   - 可筛选内容池
   - 可完成编辑、提交审核、审核、发布、归档
5. 保留 AI、发布接口的可替换占位能力，不直接接真实外部系统。

## 7. 本轮优点

- 本地已经可以完成一整条 MVP 测试闭环
- 数据不会因刷新页面而全部丢失
- 可以切换不同角色测试权限边界
- 可以主动触发抓取任务并验证去重结果
- 当前结构仍然保持低依赖，方便后续继续演进

## 8. 本轮缺点与代价

- 仍未接真实抓取能力
- 仍未接真实 DeepSeek API
- 仍未接真实主站发布接口
- 登录鉴权是本地测试级别，不是正式生产方案
- 由于当前目录不是 Git 仓库，仍无法满足“每次先 pull 最新仓库”的流程要求

## 9. 验证与结果

本轮计划验证：

1. `node --check` 校验服务端与前端脚本语法
2. 启动本地服务
3. 用登录接口获取会话
4. 执行一次模拟抓取任务
5. 校验 bootstrap 返回的数据是否包含任务、文章、日志更新

本轮结果：

- 若以上步骤通过，则可证明本地 MVP 测试版已可运行

## 10. 本轮结论

本轮最重要的结论是：

- 对当前阶段而言，最有效的推进方式不是直接做重型正式系统，而是先把“本地可测的业务闭环”做完整

## 11. 下一轮建议

下一轮最值得做的事：

1. 接入真实 DeepSeek API
2. 把模拟抓取替换为白名单真实抓取最小链路
3. 把本地 JSON 持久化替换为正式数据库
