# 版本记录：v1.24-keyword-stats-and-toggle

## 1. 本轮目标

本轮要解决的问题，是把“信息源 -> 关键词”这块继续收口到更清爽、更直观的状态。

用户这轮给出的明确要求有两点：

1. 关键词不再展示解释性文案
2. 关键词只在下方展示抓取量，并支持直接勾选“是否启用”

本轮优先处理这个问题，是因为上一轮虽然已经把前台结构收成了“仪表盘、信息源、内容池、编审台、发布榜”，但关键词列表仍然带有说明文字，阅读负担偏重，也没有把“启用/停用”放到最顺手的位置。

本轮属于：

- 前台交互收口
- 信息结构简化
- 可测试性提升

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.23-keyword-driven-open-discovery-flow.md`
- `custom/notes/v1.22-clean-list-layout-and-source-delete.md`

修改前的状态是：

- 信息源已经被收成“关键词 + 抓取源”
- 采集模式已经切到“关键词驱动 + 重点源优先 + 开放发现补充”
- 关键词可以新增、编辑、删除
- 但关键词列表本身仍然更像“说明卡片”，而不是“操作列表”

这意味着上一轮已经解决了“采集模式口径不清”的问题，但这一轮还要继续解决“关键词列表不够轻、不够像配置清单”的问题。

## 3. 问题分析

表面现象是关键词列表里还有说明文案，但核心矛盾其实有三层：

1. 关键词列表承载了过多解释信息  
   当前业务口径已经明确，关键词的作用不需要在每一项里重复解释。继续保留说明文案，只会增加阅读噪音。

2. 启用状态离操作点太远  
   虽然表单里已经有“启用”字段，但列表里看不到也改不到，用户每次都要先点编辑再切换，操作成本偏高。

3. 缺少更有价值的反馈信息  
   与其展示“这是用来驱动采集的关键词”这种已知信息，不如直接展示抓取量，让用户知道哪些关键词真的产生了线索。

真正的瓶颈不是“功能没有”，而是“同样一块区域里，应该展示什么信息才最有用”。

## 4. 候选方案比较

### 方案 A：只删解释文案，不补任何新信息

核心思路：

- 去掉关键词备注
- 保持列表其余结构不变

优点：

- 改动最小
- 风险最低

缺点：

- 删完以后列表会显得空
- 无法回答“这个关键词有没有抓到内容”
- 启用状态仍然要进编辑表单才能改

放弃原因：

- 只能做到“更少信息”，做不到“更有效信息”

### 方案 B：去掉解释文案，改成“抓取量 + 启用勾选”

核心思路：

- 删除关键词项里的说明文案
- 在列表项下方只保留“总抓取量 / 本月抓取量”
- 把启用勾选直接放到列表项头部
- 不新增后端切换接口，直接复用已有 `/api/keywords/save`

优点：

- 和用户描述最一致
- 不需要改数据库结构
- 不需要新增后端接口
- 列表更像真正的配置清单

缺点：

- 抓取量是前端根据当前内容池数据实时统计的，不是预聚合字段
- 如果后续文章量很大，前端统计仍有一定遍历成本

最终选择原因：

- 这是当前收益最高、改动最小、也最符合页面目标的一种做法

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 关键词项不再显示解释说明  
   因为业务作用已经由页面整体结构表达清楚，不需要在每条关键词上重复解释。

2. 抓取量按“总抓取量 / 本月抓取量”展示  
   这样用户能快速判断关键词的活跃度，而不用看备注。

3. 启用切换放在列表行内  
   用户可以直接在关键词列表里勾选启用或停用，不需要先进入编辑态。

4. 启用切换继续复用 `/api/keywords/save`  
   不为了一个布尔切换再新增接口，保持现有后端边界不变。

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

本轮在前端补了三组逻辑：

1. 日期辅助函数  
   - `normalizeDateValue`
   - `isCurrentMonthValue`

   作用：把 `crawlTime / createdAt / publishTime` 统一转成可比较的日期，用来判断是否属于本月。

2. 关键词抓取量统计函数  
   - `getKeywordCaptureStats`

   作用：遍历当前内容池文章，统计某个关键词的：
   - 总抓取量
   - 本月抓取量

3. 关键词列表渲染与启用切换  
   - 新的 `renderKeywords`
   - `toggleKeywordEnabled`

   作用：
   - 删掉每条关键词的解释文案
   - 在列表项中展示抓取量
   - 给每个关键词加启用勾选
   - 勾选后复用现有保存接口更新状态

### 6.2 `custom/content-center-demo/public/styles.css`

补了两个轻量样式：

- `keyword-actions`
- `keyword-toggle`

作用：

- 让关键词项里的“启用勾选 + 编辑/删除”保持在一行里更整洁
- 保持当前苹果极简风格，不重新发散出新视觉体系

### 6.3 `custom/content-center-demo/public/index.html`

只做了一项必要调整：

- 前端静态资源版本从 `v26-keyword-discovery` 升到 `v27-keyword-toggle`

作用：

- 避免浏览器继续命中旧缓存
- 保证这轮 `app.js` 和 `styles.css` 能被最新加载

## 7. 本轮优点

相比上一轮，这轮的正向效果主要有：

- 关键词列表更轻，阅读负担更低
- 用户能直接看到关键词的抓取活跃度
- 启用/停用更顺手，不需要多一步进入编辑表单
- 没有新增后端接口，改动范围可控
- 继续保持了前台“清清爽爽”的方向

## 8. 本轮缺点与代价

这轮也有明确代价：

1. 抓取量统计仍是前端实时算出来的  
   当前适合 MVP，但不是高数据量场景下的最终形态。

2. 旧的 `renderKeywords` 逻辑块仍然保留在文件里  
   这轮为了最小改动与避开历史乱码块，采用了后置覆盖写法。功能上没问题，但后续最好顺手清理旧块，避免文件继续变长。

3. 抓取量口径目前基于内容池文章  
   它反映的是“系统当前持有文章”的数量，不是独立任务维度的全历史命中统计。

## 10. 本轮结论

本轮最重要的结论是：

关键词列表不需要继续承担“说明卡片”角色，更合适的形态是“轻量操作列表”。

被证实的假设：

- 把解释文案去掉后，列表反而更清楚
- 用“总抓取量 / 本月抓取量”替代备注，更符合用户决策需求
- 直接复用现有保存接口，足够支撑启用勾选切换

被修正的旧判断：

- 之前默认列表里保留备注更完整，但实际会让关键词区显得拖沓

以后应当默认继承的经验：

- 前台信息源区优先展示“可操作信息”和“可判断信息”
- 不要在每条配置项里重复业务背景说明

## 11. 下一轮建议

   收益最高，能减少后续继续迭代时的阅读负担。

2. 把抓取源列表文案也进一步压轻  
   和这轮关键词列表风格继续统一。
