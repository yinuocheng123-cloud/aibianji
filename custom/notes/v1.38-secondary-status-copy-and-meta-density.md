# 版本记录：v1.38-secondary-status-copy-and-meta-density

## 1. 本轮目标
本轮继续承接前台基础收口，不扩新功能，重点完成两件事：

1. 继续把前台剩余更细碎的次级状态文案、提示文案并入统一 `copy/helper`
2. 再压一轮内容池和发布榜的行内元信息密度，让它们更像文章清单

这一步之所以值得继续做，是因为 `v1.37` 已经把编审台右侧工作区收得更轻，当前最明显的残留问题已经转向：

- 内容池、任务列表、失败空态、AI 设置状态里还残留少量散写短文案
- 内容池和发布榜的元信息虽然已经比早期轻，但还可以更短、更聚焦

本轮属于：

- 可维护性提升
- 前台体验收口

本轮承接：

- [v1.37-secondary-copy-deepening-and-readonly-lightening](D:\ceshi\aibianji\custom\notes\v1.37-secondary-copy-deepening-and-readonly-lightening.md)

- 编审台右侧的只读块、AI 历史块和发布结果块已经压轻了一轮
- 但 `renderTasks()`、`renderAiSettings()`、`renderFailures()`、`renderSources()` 里仍有少量次级文案直接散写
- 内容池文章行和发布榜日期行的元信息仍然偏“略松”，还不够像干净的文章清单

- 不新增前台模块

## 3. 问题分析
本轮的表面问题比较轻，但核心矛盾仍然清晰：

1. `copy/helper` 已经建立，如果剩余状态文案继续散写，后续迭代仍会重新回到“到处找字符串”  
2. 内容池和发布榜如果元信息不继续压短，视觉上就还会保留“轻卡片”而不是“清单”感

主要原因：

1. 一些状态文本仍直接写在渲染层，例如：
   - 任务行的“疑似重复保留 / 跳过”
   - “新增 / 重复 / 失败”
   - 抓取源启停标签
   - AI 设置状态
   - 抓取失败空态说明
2. 内容池和发布榜虽然已转成行级列表，但字号、padding、gap 仍略大
3. 发布榜元信息仍然保留较明显的行高和块距，没有完全贴近文章清单视觉

## 4. 候选方案比较
### 方案 A：只继续补 copy，不动样式
核心思路：

- 把剩余次级文案全收进统一入口
- 暂时不继续动内容池和发布榜的样式密度

优点：

- 维护性继续提升
- 改动风险小

缺点：

- 用户侧能感知到的“更清爽”变化有限

### 方案 B：只继续压样式，不动剩余散写文案
核心思路：

- 只继续压内容池和发布榜的间距、字号和 meta 表达
- 暂时保留剩余散写短文案

优点：

- 页面观感会继续变轻

缺点：

- 前台文案入口仍然不干净
- 后续继续迭代时仍有维护噪音

### 方案 C：继续收状态 copy，同时再压清单密度
核心思路：

- 一边继续收剩余状态文案和提示文案
- 一边对内容池和发布榜做一轮低风险密度压缩

优点：

- 同时提升可维护性和视觉一致性
- 不触碰主业务逻辑
- 与 `v1.37` 的方向完全一致

缺点：

- 需要同时改 `app-state / app-utils / app-render / styles / index`

最终选择：

- 采用方案 C

原因：

- 这是当前最稳、最自然的继续收口路径

## 5. 最终决策
本轮最终采用以下策略：

1. 继续扩展 `FRONTSTAGE_COPY`
2. 在 `app-utils.js` 中补更细的状态和元信息 helper
3. 在 `app-render.js` 中收掉任务行、失败空态、抓取源启停、AI 设置状态等残留散写文案
4. 在 `styles.css` 中继续压内容池和发布榜的行内元信息密度
5. 同步升级前端资源版本，确保浏览器拿到新资源

## 10. 本轮结论
本轮最重要的结论是：

- 前台 copy 统一如果要真正收稳，就必须继续往状态文案和统计文案层推进
- 内容池和发布榜的“清单感”主要来自元信息的短化和密度控制，而不是继续删功能
- 当前这个阶段继续做前台表达层收口，仍然是收益高、风险低的正确顺序

本轮继承了哪些旧思路：

- 延续 `v1.37` 的方向，继续收次级 copy 与轻量工作区
- 延续 `v1.36` 的方向，不碰主链路，只收基础问题

本轮修正了哪些旧判断：

- 不再把“主要文案已经统一”视为 copy 收口已基本完成
- 不再把“内容池和发布榜已经够轻”视为文章清单表达已收完
