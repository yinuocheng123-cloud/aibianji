# 版本记录：v1.15-login-token-priority-and-client-events

## 1. 本轮目标

本轮目标是继续修复“登录后仍回到未登录状态”的问题，并补上前端诊断上报接口，避免浏览器控制台继续出现 `POST /api/client-events 404`。

这个问题仍值得优先处理，因为用户已经给出了更明确的浏览器线索：前端确实进入了启动失败分支，而诊断接口本身又没有接住，导致我们虽然知道“出错了”，却看不到页面实际报了什么。

本轮属于：

- Bug 修复
- 可测试性提升
- 诊断能力补强

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.14-login-cookie-sync.md`
- `custom/notes/v1.13-copy-audit-and-local-commit.md`
- `custom/notes/experiments/exp-20260404-login-runtime-check.md`

上一轮已经做了两件事：

1. 前端拿到新 `session_token` 后，主动同步浏览器 cookie
2. 静态资源版本提升到 `v18-session-cookie-sync`

但用户继续反馈：

- 页面能打开
- 登录仍然失败
- 按 `Fn + F12` 能看到 `:3210/api/client-events 404`

这说明单纯前端 cookie 同步还不够，本轮需要把“日志链路”和“后端会话优先级”一起收口。

## 3. 问题分析

表面现象有两个：

1. 登录还是进不去
2. 控制台出现 `POST /api/client-events 404`

这两个现象背后的关键矛盾是：

- 前端已经在尝试把启动异常回传给服务端
- 但服务端没有对应接口，导致异常信息丢失
- 同时，后端在解析会话时仍优先吃 cookie，再看 header token

如果浏览器里残留旧 cookie，就会出现下面的冲突：

1. 表单登录后，页面拿到新的 `session_token`
2. 前端发 `/api/bootstrap` 时也可能带上新的 `X-Session-Token`
3. 但服务端先读取旧 cookie
4. 结果 bootstrap 被判断成未登录，页面重新显示登录层

上一轮用前端同步 cookie 的方法已经缓解了问题，但这轮进一步验证后确认：

- 真正更稳的修复点，仍然是服务端优先采用有效 token，而不是盲目先吃 cookie

## 4. 候选方案比较

### 方案 A：只补 `/api/client-events` 路由

核心思路：

- 保持认证逻辑不变
- 先把前端诊断日志接住

优点：

- 能立刻消除 `404`
- 便于继续观察前端错误

缺点：

- 只能提高可观测性
- 不能直接修掉登录态冲突

未选原因：

- 用户当前卡住的是“无法登录”，不能只停在观测层

### 方案 B：同时补诊断路由，并把后端会话解析改成优先有效 token

核心思路：

- 新增 `/api/client-events`
- 同时调整 `getCurrentUser(request)`：
  - 不再直接 `cookie || header`
  - 改成在 `headerToken` 与 `cookieToken` 中优先选取“存在且在 sessions 中有效”的 token

优点：

- 同时解决“看不见前端异常”和“旧 cookie 抢新 token”两个问题
- 对现有登录流程改动小，但对稳定性提升明显

缺点：

- 需要继续修改 `api-router.js`
- 该文件在终端里仍存在编码显示不友好的问题，补丁要更谨慎

最终选择原因：

- 这是当前最直接、最贴近根因的修复路径

## 5. 最终决策

本轮最终采用方案 B：

1. 在服务端新增 `/api/client-events`
2. 把 `getCurrentUser` 改成优先选取有效 token
3. 重启本地 `3210` 服务
4. 验证：
   - `client-events` 不再 404
   - “旧 cookie + 新 header token” 场景下 `/api/bootstrap` 返回 `200`

## 6. 具体实现

本轮改动文件如下：

### 6.1 `custom/content-center-demo/src/routes/api-router.js`

改动一：修正会话解析

- 原逻辑：
  - `const token = cookies[SESSION_COOKIE] || headerToken`
- 新逻辑：
  - 先取 `cookieToken`
  - 再取 `headerToken`
  - 在两者中按顺序找出“存在且在 `sessions` 中有效”的 token

这意味着：

- 只要 header token 是有效的，就不会再被残留旧 cookie 抢走优先级

改动二：新增 `/api/client-events`

- 接收前端上报的 `type / message / url / sessionUser / articleCount`
- 写入现有日志体系
- 返回 `200`

这意味着：

- 浏览器控制台里的 `404 /api/client-events` 被消除
- 后续若前端初始化仍失败，至少服务端能收到诊断信息

### 6.2 `custom/content-center-demo/public/app.js`

本轮没有新增新逻辑，但继续沿用上一轮的 `syncSessionCookie(token)` 修复。

### 6.3 运行态

- 重新拉起了 `3210` 本地服务
- 确认当前首页与最新脚本都能访问

## 7. 本轮优点

这轮相较上一轮的改进点：

- 不再只依赖前端去“尽量同步 cookie”
- 后端也开始真正按“有效 token”做选择
- 控制台 404 被补掉，后续排错链路更完整
- 验证方式更贴近真实用户浏览器冲突场景

## 8. 本轮缺点与代价

本轮仍有这些代价：

- `api-router.js` 仍然是大文件，继续迭代会越来越难维护
- 终端里仍存在中文显示编码不友好的问题，影响局部补丁效率
- 本轮没有直接做浏览器自动化点击验证，主要依赖定向接口与冲突场景验证

本轮完成了这些验证：

1. `git pull --ff-only`

2. 语法检查
   - `node --check src/routes/api-router.js`

3. 服务重启后首页检查

4. 诊断接口检查

   - 结果：`200`

5. 冲突场景验证
   - 先登录拿到最新 `session_token`
   - 再故意请求 `/api/bootstrap` 时附带：
     - `Cookie: content_center_session=stale-token`
     - `X-Session-Token: <最新 token>`

这个验证很关键，因为它直接说明：

- 即使浏览器残留错误 cookie
- 只要前端有有效 token
- 当前后端也会优先认最新登录态

## 10. 本轮结论

本轮最重要的结论是：

- 这次登录问题的真正根因，不只是“前端有没有把 token 存好”，而是“服务端如何在 cookie 与 header token 冲突时做选择”
- 现在这条优先级链已经修正
- 同时前端诊断接口也已经接上

后续应继承的默认经验：

- 如果本地演示系统再次出现“明明拿到 token 但页面仍未登录”，应优先检查服务端是否按“有效 token 优先”解析
- 浏览器控制台出现前端诊断请求时，不应该再允许它 404
