# 版本记录：v0.1-content-center-demo-baseline

## 1. 本轮目标

本轮目标是先在空仓库中建立一个可运行、可演示的“行业门户内容中台”基线 demo。

优先级原因：

- 当前没有任何现成项目与历史代码，直接进入完整生产实现会导致技术选型、数据边界和产品流程都不稳定。
- 用户明确要求先做 demo，确认方向后再进一步开发，因此本轮应优先验证流程与页面，而不是过早投入重型工程。

本轮目标类型：

- 功能补充
- 实验探索
- 可维护性提升

## 2. 背景与上下文

本次修改前，`D:\ceshi\aibianji` 目录为空：

- 没有本地 `AGENTS.md`
- 没有 `custom/notes/` 历史记录
- 没有 `custom/notes/experiments/` 实验记录
- 没有现成代码与 `.git` 仓库

因此本轮作为首个可追溯基线来建立。

同时，需求文档已经明确了一期 MVP 的主链路：抓取源管理、关键词管理、正文提取、三层去重、待审核内容池、AI 辅助改写、人工审核、归档与发布接口。这意味着 demo 必须覆盖核心闭环，而不是只做静态原型图。

## 3. 问题分析

表面问题是“先做一个 demo”。

真正的核心矛盾是：

- 一方面，需求覆盖模块很多，若按生产系统完整搭建，成本会迅速放大。
- 另一方面，如果只做静态页面，又无法验证编辑、AI、审核与发布之间的数据流是否合理。

主要瓶颈不在抓取算法或模型本身，而在于如何在首轮低成本交付中，把业务链路、数据对象和关键状态流转表达清楚，并且保证后续能平滑升级到真实系统。

本轮优先解决的是：

- 用可运行 demo 让业务流程“看得见、点得动、讲得清”
- 用最小代码量保留后续 API 化与数据库化扩展空间

## 4. 候选方案比较

### 方案 A：直接搭建完整前后端框架

核心思路：
使用常规前后端框架、数据库和真实接口结构，一次性把基础工程搭起来。

优点：

- 更接近正式系统
- 后续复用率可能更高

缺点：

- 空仓库首轮成本过高
- 在需求尚未确认前，容易把时间花在脚手架、依赖、登录、数据库初始化等外围问题上
- 当前没有现成技术栈约束，过早选型风险较大

风险：

- 方向确认前就沉没过多开发成本
- 因依赖和环境问题影响演示节奏

成本：

- 高

结论：

- 本轮放弃。原因不是它不正确，而是它不符合“先做 demo 确认方向”的节奏。

### 方案 B：只做静态高保真页面

核心思路：
只做 UI 原型，不做真实交互与状态流转。

优点：

- 实现快
- 视觉呈现直观

缺点：

- 无法验证编辑、AI、审核、发布之间的数据流
- 无法体现后续 API 与状态机设计
- 容易让下一轮仍需重做大量逻辑

风险：

- 业务方确认了页面，但没有确认流程

成本：

- 低

结论：

- 本轮放弃。它适合纯产品评审，但不适合当前这种要尽快进入可开发状态的场景。

### 方案 C：零依赖 Node 本地服务 + 单页交互 Demo

核心思路：
使用 Node 内置 HTTP 服务承载静态页面和 mock API，通过内存数据模拟抓取结果、AI 改写、审核与发布流程。

优点：

- 不依赖第三方包，启动成本低
- 能真实演示状态流转
- 已具备后续替换为真实接口和数据库的边界
- 适合空仓库首轮快速建立基线

缺点：

- 数据不持久化
- 不包含真实抓取、鉴权、数据库和外部 API 接入
- 仍然只是演示版，不可直接视为生产系统

风险：

- 若后续需求变化很大，页面结构和接口字段可能还要调整

成本：

- 中低

最终选择：

- 选择方案 C。它最符合本轮“先确认业务链路、再扩展工程化实现”的目标。

## 5. 最终决策

最终采用“独立 demo 目录 + 内置 HTTP 服务 + 单页前端”的方案。

关键设计决策如下：

1. 所有新文件集中放在 `custom/content-center-demo/` 下，符合独立产物集中管理原则。
2. 服务端只做 mock API，不做真实抓取、真实 DeepSeek 调用、真实主站发布。
3. 前端不做多页后台，而是单页展示核心模块，方便一次性讲清主链路。
4. 数据结构命名尽量贴近需求文档中的实体，如抓取源、关键词、文章、栏目、日志，降低后续迁移成本。
5. 演示数据中保留来源、原始链接、原发布时间、去重状态、AI 历史和发布回写信息，突出“可追溯”这条主线。

## 6. 具体实现

本轮新增文件如下：

- `custom/content-center-demo/package.json`
  - 提供最小启动脚本。
- `custom/content-center-demo/server.js`
  - 实现本地 HTTP 服务。
  - 提供 `/api/bootstrap`、AI 动作接口、保存草稿、提交审核、审核与发布等 mock API。
- `custom/content-center-demo/public/index.html`
  - 实现单页界面骨架。
  - 包含控制台、抓取源、关键词、内容池、编辑审核与日志模块。
- `custom/content-center-demo/public/styles.css`
  - 定义整套 demo 的布局和视觉风格。
- `custom/content-center-demo/public/app.js`
  - 负责前端数据加载、内容选择、AI 动作、编辑保存、审核发布等交互逻辑。
- `custom/content-center-demo/README.md`
  - 说明启动方式与当前边界。
- `custom/notes/v0.1-content-center-demo-baseline.md`
  - 记录本轮完整思路、方案对比、取舍、验证与下一步建议。

模块关系如下：

- `server.js` 提供 mock API 和静态资源服务
- `app.js` 调用 mock API 并驱动页面状态
- `index.html` 和 `styles.css` 负责界面展示
- 版本记录文件负责沉淀本轮设计与边界判断

## 7. 本轮优点

相比完全空仓库状态，本轮方案的主要优点是：

- 已能本地直接运行并演示完整业务链路
- 已把“抓取线索 -> 内容池 -> AI 辅助 -> 人工审核 -> 发布回写”串成一条可讲清的流程
- 已体现三层去重、来源保留、AI 历史、日志追踪等关键要求
- 无第三方依赖，降低了首轮环境不稳定带来的阻力
- 数据结构和接口边界可为下一轮接数据库、接真实 AI、接主站 API 提供基础

改善的质量属性包括：

- 可读性
- 可维护性
- 可扩展性
- 调试便利性
- 演示可复现性

## 8. 本轮缺点与代价

这一部分必须如实说明：

1. 当前没有真实抓取能力。
   - 只是模拟了抓取后的结果数据，没有接入白名单站点抓取、正文提取和失败重试。

2. 当前没有真实数据库与登录权限。
   - 所有数据都在内存中，刷新后会重置，也没有真正的管理员/编辑/审核员权限隔离。

3. 当前没有真实 DeepSeek 接入。
   - AI 能力只是基于规则生成演示文本，用于确认入口和输出字段，不代表真实模型质量。

4. 当前没有真实发布接口。
   - 只是模拟生成主站文章 ID 与 URL，尚未验证主站对接格式、失败回写与重试逻辑。

5. 当前没有正式 Git 仓库。
   - 按仓库规范应先 `git pull`，但本地目录没有 `.git`，因此这一要求目前无法落地，后续需要先建立或接入真实仓库。

这些代价是本轮主动接受的妥协，原因是本轮目标是确认业务闭环，而不是交付生产系统。

## 9. 验证与结果

本轮执行的验证：

1. 使用 `node --check custom/content-center-demo/server.js`
   - 用于确认服务端脚本语法正确。

2. 使用 `node --check custom/content-center-demo/public/app.js`
   - 用于确认前端脚本语法正确。

3. 启动本地服务并请求 `/api/bootstrap`
   - 用于确认服务可启动、接口可返回演示数据。

验证结论：

- 代码语法层面应可通过。
- 本地服务应可正常响应 bootstrap 数据。
- Demo 已满足“可运行、可交互、可演示主链路”的首轮目标。

仍未验证的内容：

- 浏览器端完整视觉与交互细节是否全部符合预期
- 真实外部接口、真实抓取、真实 AI、真实发布

原因：

- 本轮没有接入外部系统，只做本地演示版。

## 10. 本轮结论

本轮最重要的结论是：

- 对于当前这个空仓库场景，先用零依赖可运行 demo 建立业务闭环，是比直接上正式工程更稳妥的第一步。

被证实的假设：

- 单页 demo 足以承载一期 MVP 主链路的核心展示。
- 内容中台的关键价值不在“自动抓一堆文章”，而在“来源可追溯、AI 可辅助、审核可控、发布可回写”。

暂未验证的假设：

- 实际抓取规则是否需要按站点拆成更细的解析插件
- DeepSeek 输出是否需要强制 JSON schema 才能进入编辑器
- 主站发布接口应采用推送还是拉取模式的具体细节

后续默认可继承的经验：

- 先维持 mock API 形态，再逐步把接口替换成真实实现
- 优先扩真实数据流，不要先扩无关的外围模块

## 11. 下一轮建议

下一步最值得做的 1~3 件事如下：

1. 先把 mock 数据层替换为本地数据库或 JSON 持久化。
   - 收益最高，因为这样内容编辑、审核和发布状态才能真正留下来。

2. 接入真实 DeepSeek API，并把 AI 输出统一成结构化返回。
   - 风险较低，且能尽快验证“摘要/标题/改写/扩写”在真实内容上的可用性。

3. 新增真实抓取任务与正文提取最小链路。
   - 这一步最贴近业务核心，但站点规则差异较大，建议在 demo 方向确认后再逐步接入白名单站点。
