# 版本记录：v1.28-bootstrap-login-and-legacy-cleanup

## 1. 本轮目标

本轮要解决的问题，是在 `v1.27` 已经把前台列表压轻之后，继续把影响体验判断的基础问题收口：

- 未登录访问首页时，不应显示“初始化失败：请先登录”
- 首页静态提示文案要和当前真实交互口径一致
- `app.js` 里最明显的历史遗留函数名要先清出来，避免后续继续误判

之所以优先处理这一轮，是因为这些问题虽然不属于新功能，但会直接影响用户对系统是否“稳定、清楚、可继续测试”的判断。

本轮属于：

- Bug 修复
- 可维护性提升
- 前台基础体验收口

## 2. 背景与上下文

本轮承接以下记录：

- `custom/notes/v1.27-frontstage-list-cleanup-and-sorting.md`
- `custom/notes/v1.26-content-pool-split-by-priority-and-open.md`

修改前的状态是：

- `v1.27` 已经把信息源、内容池、编审台的列表表达压轻
- 但前端初始化仍然有一个体验问题：未登录时首页会先走 bootstrap，再被当作初始化失败处理
- 同时，`app.js` 中还保留了若干前几轮演化留下的旧函数名，当前靠文件底部同名覆盖实现维持行为

已知约束有三点：

1. 这一轮不扩功能，不改整木网接口适配层
2. 尽量保持最小改动，不做大拆分
3. 必须留下正式记录，便于后续继续清理前端单文件遗留逻辑

## 3. 问题分析

表面现象有三类：

1. 首次进入登录页时，前端可能显示“初始化失败：请先登录”
2. 首页 `discover-status` 静态提示和当前真实行为口径不完全一致
3. `app.js` 里存在多处“旧实现 + 后置覆盖实现”并存的情况

核心矛盾不是“功能没做通”，而是：

- 未登录本应是正常状态分支，却被归类到了异常分支
- 页面静态文案和运行后动态文案不一致，会放大用户对系统不稳定的感知
- 代码层虽然功能可用，但遗留的重复函数会持续增加后续迭代判断成本

主要原因是：

- 早期 bootstrap 请求统一走 `request()`，401 时会抛错；如果初始化层不单独识别未登录，就会走到 `showBootError`
- 之前为了快速收口前台，更多采用“后置覆盖”而不是直接清掉明显失效的旧函数

## 4. 候选方案比较

### 方案 A：继续维持当前覆盖式实现，只补一层错误文案兜底

核心思路：

- 不改 `request()`
- 只在 `initialize()` 的 catch 中继续补更多判断

优点：

- 改动最小
- 风险最低

缺点：

- `request()` 的 `allowUnauthorized` 仍然没有实际意义
- 代码会继续依赖“后面再补一层兜底”
- 不能顺手降低后续阅读成本

放弃原因：

- 只能压现象，不能顺手收一层基础实现

### 方案 B：把未登录识别收回到请求层和初始化层共同处理，再清一批最明显的历史遗留函数名

核心思路：

- 让 `request()` 在 `allowUnauthorized` 场景下不主动弹登录，只抛稳定错误
- 初始化层把“请先登录”视为正常登录态分支，而不是 boot error
- 顺手把已经失效的旧函数改成 `Legacy` 命名，避免后面继续误判

优点：

- 体验问题和可维护性问题一起收口
- 不需要大拆文件
- 行为边界更清晰

缺点：

- 仍然保留单文件前端结构
- 还没有彻底删除全部历史实现

最终选择原因：

- 这是当前“收益最高 / 改动边界最小”的方案

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 未登录是正常入口，不算初始化失败
2. `request()` 的 `allowUnauthorized` 要开始真正生效
3. 对已经明确失效的旧实现，不再继续保留同名，而是先改成 `Legacy`
4. 首页静态提示先和当前实际采集口径对齐

## 6. 具体实现

### 6.1 `custom/content-center-demo/public/app.js`

本轮主要修改了三类内容：

1. 调整 `request()` 的 401 处理
   - 当 `allowUnauthorized = true` 时，不主动 `showLogin()`
   - 仍然清理本地 session，并抛出稳定的“请先登录”

2. 清理最明显的历史遗留函数名
   - `renderCollectionStatus` -> `renderCollectionStatusLegacy`
   - `renderSources` -> `renderSourcesLegacy`
   - `renderArticles` -> `renderArticlesLegacy`
   - `initialize`（旧版） -> `initializeLegacy`
   - 同时删除了已不再使用的开放发现来源聚合辅助函数

3. 保留文件底部的最终生效实现
   - 让当前活跃实现继续负责前台列表和初始化逻辑
   - 避免这一轮扩大为大规模重构

### 6.2 `custom/content-center-demo/public/index.html`

本轮做了两项必要调整：

1. 把 `discover-status` 的静态文案改成当前口径：
   - “已启用关键词和抓取对象后，系统会先抓重点源，再补开放发现。”

2. 更新静态资源版本号：
   - `v31-bootstrap-cleanup`

### 6.3 `custom/notes/v1.28-bootstrap-login-and-legacy-cleanup.md`

新增本轮正式记录，用于说明：

- 这轮继承了什么
- 修正了什么旧判断
- 为什么选择“最小收口”而不是继续拆前端文件

## 7. 本轮优点

相比 `v1.27`，本轮的正向效果主要有：

- 登录页不再把“未登录”误判成“初始化失败”
- 首页未登录时的静态提示不再和当前采集口径错位
- `app.js` 中最容易误导后续判断的一批重复函数名先被清出来
- 后续如果继续拆前端单文件，会更容易识别哪部分是当前生效逻辑

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. 仍然没有把 `app.js` 拆分成多文件
2. 旧逻辑没有一次性全部删除，只是先做了最小清理
3. 当前仍采用“前期旧实现保留 + 当前实现生效”的演化式结构，不是最终理想状态

## 10. 本轮结论

本轮最重要的结论是：

- 未登录 bootstrap 不能再被当作 boot error 处理，这属于正常登录态分支

被证实的假设：

- 用户对“系统是否稳定”的判断，很大程度来自登录页的第一屏反馈，而不是后续功能是否丰富
- 即使不做大重构，只清理少量误导性的重复函数名，也能显著降低继续迭代时的判断成本

被修正的旧判断：

- 之前默认“只要功能通了，初始化里多一层 boot error 也问题不大”，这一点已被证伪

以后应默认继承的经验：

- 对登录页、bootstrap、会话恢复这类入口逻辑，必须把“正常未登录”和“真实异常”分开处理

## 11. 下一轮建议

下一步最值得继续做的 3 件事：

1. 继续清理 `app.js` 里剩余的历史辅助逻辑
   - 收益最高

2. 把编审台右侧表单再压轻一轮
   - 风险最低

3. 等前台结构彻底稳定后，再接真实整木网发布接口
   - 需要更多对接资料后再做
