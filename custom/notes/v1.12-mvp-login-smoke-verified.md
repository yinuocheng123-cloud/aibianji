# 版本记录：v1.12-mvp-login-smoke-verified

## 1. 本轮目标

本轮目标是把最近几轮登录修复真正验证到“MVP 可测试”的程度，而不是停留在静态检查或单点接口检查。

这个问题值得继续优先处理，因为：

- 用户连续反馈“点击登录后进不去”，说明之前的修复虽然方向正确，但还没有形成足够可信的闭环验证。
- 只有把登录、进入中台、核心工作流三件事一起跑通，才能说明当前版本达到本地 MVP 测试标准。

本轮属于：

- Bug 修复收口
- MVP 验证补强
- 可测试性确认

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.11-session-token-fallback.md`
- `custom/notes/v1.10-login-form-field-names.md`

上一轮已经做了这些关键改动：

- 表单字段补齐 `name`
- 表单登录改为原生 `POST`
- 前端增加 `session_token` 保存与 `X-Session-Token` 请求头
- 后端鉴权同时支持 cookie 与 header token

上一轮没有收完的点是：

- 当前工具环境里，后台子进程无法稳定常驻，导致“服务已改好”与“实际可验证”之间存在落差。

## 3. 问题分析

本轮面对的不是新的业务 bug，而是验证层面的核心矛盾：

- 代码逻辑看起来已经合理
- 但如果验证方式本身不稳定，就无法确认用户侧是否真的能用

这轮重新厘清后的关键判断是：

- 问题并不在登录逻辑持续失败，而在于之前的后台拉起方式让服务在验证中途退出，干扰了结论
- 因此应该把“启动服务 + 执行验证 + 停止服务”放到同一个命令内完成

## 4. 候选方案比较

### 方案 A：继续尝试常驻后台服务再手工点测

核心思路：

- 继续用后台方式拉起 `npm run dev`，再单独做页面验证。

优点：

- 更接近用户最终使用方式。

缺点：

- 在当前工具环境中，子进程会在命令结束后被清理，验证结果不稳定。
- 很容易把“工具环境问题”误判成“业务逻辑问题”。

放弃原因：

- 这条路已经多次影响判断，不适合作为本轮验证基线。

### 方案 B：同命令内启动服务并完成闭环验证

核心思路：

- 在同一个命令中临时拉起 `node server.js`
- 立刻执行登录验证或 smoke 回归
- 验证结束后再停止进程

优点：

- 验证过程稳定，可重复。
- 能排除后台进程被外部环境清理造成的误判。

缺点：

- 不等于长期常驻运行，只适合作为开发验证手段。

最终选择原因：

- 这是当前环境下最可靠、最能给出真实结论的验证方式。

## 5. 最终决策

本轮最终采用方案 B，并形成了两条有效验证：

1. 登录闭环验证
2. `smoke:e2e` 全流程回归验证

同时保留上一轮的 `session_token + X-Session-Token` 兜底方案，不再改动其核心实现。

## 6. 具体实现

### 6.1 代码层

本轮没有继续扩展新的业务能力，核心代码仍沿用上一轮改动：

- `custom/content-center-demo/public/app.js`
- `custom/content-center-demo/public/index.html`
- `custom/content-center-demo/src/routes/api-router.js`

### 6.2 验证层

本轮新增的是更可靠的验证方式：

- 临时拉起 `node server.js`
- 验证 `session_token -> X-Session-Token -> /api/bootstrap`
- 执行 `npm run smoke:e2e`

这一步虽然不新增业务文件，但对判断“当前版本是否达到 MVP 可测标准”非常关键。

## 7. 本轮优点

当前方案相比之前的最大优点是：

- 终于把“登录看起来修好了”推进到了“登录和主流程都实际跑通了”
- 把工具环境对子进程的干扰从业务结论里剥离掉了
- `smoke:e2e` 已经能证明当前版本至少具备本地 MVP 的核心闭环

## 8. 本轮缺点与代价

当前仍有明确代价：

- 在本对话工具环境中，后台服务不能作为长期常驻验证手段
- `session_token` 方案仍只适用于本地 MVP，不适合生产
- 前端残留乱码文案还没有系统清理

## 9. 验证与结果

本轮完成了这些关键验证：

### 9.1 登录兜底链路验证

在同一个命令内临时启动服务后，验证结果为：

- 登录成功后的最终地址为：
  - `http://127.0.0.1:3210/?session_token=...`
- 提取到 `session_token`
- 仅通过 `X-Session-Token` 请求 `/api/bootstrap`
  - 返回 `200`

这说明：

- 表单登录成功
- token 兜底链路有效
- 后端 header 鉴权已生效

### 9.2 MVP 全流程 smoke 验证

执行：

- `npm run smoke:e2e`

结果通过，关键输出包括：

- `retainedSuspected = true`
- `previewFallback = true`
- `duplicateStatus = 通过`
- `aiHistoryCount = 1`
- `savedStatus = 编辑中`
- `submittedStatus = 待复审`
- `publishedStatus = 已发布`
- `publishStatus = 已发布`
- `portalArticleId` 与 `portalUrl` 已回写

这说明当前 MVP 主链路已经能跑通：

- 登录
- 抓取
- 去重保留
- AI 处理
- 保存草稿
- 提交审核
- 审核发布
- 主站回写

## 10. 本轮结论

本轮最重要的结论是：

- 当前版本已经达到“本地 MVP 可测试”的程度
- 登录问题的业务层核心链路已打通
- 之前反复出现的“不确定是否修好”，主要是验证环境不稳定，而不是登录修复本身持续失败

以后默认应继承的经验：

- 在当前工具环境里，涉及本地服务验证时，应优先使用“同命令启动并完成验证”的方式
- 对本地演示系统，登录态最好保留 cookie 之外的兜底路径

## 11. 下一轮建议

下一轮最值得继续做的 3 件事：

1. 清理 `index.html` 和 `app.js` 里的乱码文案，提高可用性与演示质量
2. 把当前 smoke 回归结果进一步整理成可读报告或可视化提示
3. 如果进入下一阶段开发，再把本地 MVP 逐步收向更正式的部署与持久运行方式
