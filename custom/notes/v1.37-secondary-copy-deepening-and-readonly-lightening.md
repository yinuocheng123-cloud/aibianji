# 版本记录：v1.37-secondary-copy-deepening-and-readonly-lightening

## 1. 本轮目标
本轮继续承接前台基础收口，不扩新功能，重点完成两件事：

1. 继续把前台剩余更次级的动态文案并入统一 `copy/helper`
2. 再压一轮编审台右侧只读块、AI 历史块和发布结果块的密度

之所以优先处理这两件事，是因为 `v1.36` 已经把 smoke 回归基线收稳，当前最影响继续迭代效率的，已经不是功能缺口，而是前台局部文案漂移和右侧工作区仍偏重的问题。

本轮属于：

- 可维护性提升
- 前台体验收口
- 验证基线巩固

## 2. 背景与上下文
本轮承接：

- [v1.36-smoke-stability-and-workbench-tightening](D:\ceshi\aibianji\custom\notes\v1.36-smoke-stability-and-workbench-tightening.md)
- [v1.35-secondary-copy-and-ai-history-lite](D:\ceshi\aibianji\custom\notes\v1.35-secondary-copy-and-ai-history-lite.md)

修改前的状态：

- `v1.36` 已经把 `smoke:e2e` 的历史数据干扰问题压住
- 前台主路径的大部分文案已经开始汇总到 `FRONTSTAGE_COPY`
- 但内容池、编审台和发布榜里仍有一批次级短文案直接写在渲染层
- 编审台右侧虽然已经比之前轻，但“只读信息块 / AI 历史 / 发布结果”仍然有偏卡片化的厚重感

已知约束：

- 不改采集主链路
- 不改整木网模拟发布逻辑
- 不新增前台新模块
- 保持当前前台脚本拆层结构，不回退到单文件

## 3. 问题分析
表面现象是“页面已经能用，但局部仍有些散和重”，核心矛盾其实有两个：

1. 前台 copy 体系虽然已经建立，但还没覆盖到内容池、编审台和发布榜中的次级短文案  
2. 编审台右侧的说明层和只读块虽然已经压过一轮，但还没有真正形成“轻量工作区”的密度

主要原因：

1. 一些次级文本仍然直接写在 `render` 层，例如：
   - 内容池摘要 fallback
   - 内容池分组标题与计数
   - “转为重点源”
   - 修订列表摘要 fallback
   - 明细时间与相似度拼接
   - 发布榜未标注日期 fallback
2. 错误提示虽已开始统一，但交互层里仍缺少统一的 `showActionError` 类入口
3. 编审台右侧的 `detail-meta / read-box / history-item / publish-result` 仍沿用偏厚的间距和块状层级

## 4. 候选方案比较
### 方案 A：只继续改样式，不再动 copy/helper
核心思路：

- 只压 CSS
- 保持现有次级短文案继续写在渲染层

优点：

- 改动快
- 不容易碰到业务逻辑

缺点：

- 前台文案入口仍然分散
- 后续继续改文案时，仍要到多个渲染点找字符串

风险：

- 页面看起来会继续“局部顺手”，但维护成本仍高

### 方案 B：只继续收 copy/helper，不动样式密度
核心思路：

- 把次级短文案再收一轮
- 暂时不继续压编审台右侧视觉密度

优点：

- 可维护性继续提升

缺点：

- 右侧工作区仍会显得偏重
- 用户感知上的“更清爽”不会明显提升

风险：

- 页面代码更整齐了，但视觉收益不够大

### 方案 C：继续收 copy/helper + 同步压轻右侧只读块
核心思路：

- 继续把次级文案并入统一 copy/helper
- 同时做一轮低风险样式收口，把编审台右侧只读区再压轻

优点：

- 同时提升维护性和前台观感
- 改动仍然局限在前台脚本和样式层
- 不影响既有接口和 smoke 基线

缺点：

- 改动文件数比单点修补更多

风险：

- 需要同步静态 HTML、渲染层和样式层，否则容易又出现“静态一套、运行时一套”

最终选择：

- 采用方案 C

原因：

- 这是当前收益最高、风险最低的一种继续收口方式

## 5. 最终决策
本轮最终决定：

1. 继续扩展 `FRONTSTAGE_COPY`
2. 在 `app-utils.js` 中补最小 helper，避免继续在渲染层散写次级短文案
3. 在 `app-render.js` 中把内容池、编审台和发布榜相关短文案全部切到统一 helper
4. 在 `styles.css` 中继续压轻编审台右侧的只读块、AI 历史块和发布结果块
5. 同步更新 `index.html` 的静态初始文案与资源版本，保持与运行时一致

## 7. 本轮优点
本轮带来的主要正向效果：

- 前台 copy 统一继续推进到更次级的交互层
- 内容池、修订列表和发布榜的短文案不再继续散落在渲染层

## 8. 本轮缺点与代价
本轮仍然有这些明确边界：

- 仍然没有把所有动态文案都收进统一 copy/helper，只是继续向前推进了一层
- 右侧工作区虽然更轻了，但整体仍然是原生脚本拼装的结构，不是更彻底的组件化 UI
- 这轮主要收的是前台表达和工作区密度，不涉及真实整木网接口
- 当前仓库里仍保留前几轮未统一提交的正式记录文件，需要后续一起入库

## 10. 本轮结论
本轮最重要的结论是：

- 前台 copy 统一如果要真正形成可继承的体系，就必须继续往次级短文案和错误提示层推进，而不是停在主按钮层
- 编审台右侧要继续变轻，最有效的方式不是再删字段，而是把只读信息带、历史记录和结果提示一起做薄
- 在不改接口的前提下，继续收 frontstage 的表达层，仍然是当前最值当的低风险迭代方向

本轮继承了哪些旧思路：

- 延续 `v1.36` 的方向，继续先收基础问题，不扩新功能
- 延续 `v1.35` 的方向，继续让前台说明层和历史记录块向“轻量工作区”演化

本轮修正了哪些旧判断：

- 不再把“主按钮已经统一了”视为 copy 工作已基本完成
- 不再把“右侧已经压过一轮”视为编审台视觉收口已足够

## 11. 下一轮建议
下一步最值得继续做的 1~3 件事：

1. 继续把前台剩余更细碎的动态提示与状态文本收进统一 `copy/helper`  
   收益最高，能继续降低后续维护中的散写概率。

2. 再压一轮发布榜与内容池的行内元信息密度  
   风险低，且能继续提升整体“文章清单感”。

3. 当前 frontstage 基线完全稳定后，再开始真实整木网接口适配层  
   需要更多外部信息，适合在前台主链路确认稳定后再推进。
